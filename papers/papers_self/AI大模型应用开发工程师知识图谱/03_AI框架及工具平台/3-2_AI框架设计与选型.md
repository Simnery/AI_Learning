# 3.2 AI框架设计与选型

> 3.2 AI框架设计与选型 > 3. AI框架及工具平台

## 核心概念

- **是什么**：AI 框架是构建 LLM 应用的"脚手架"——它提供了一套标准化的组件、数据流和工具集成方式，让你不用每次从零开始写胶水代码。框架选型就是根据业务需求选择最合适的脚手架。
- **为什么重要**：选错框架轻则开发效率低下，重则架构推倒重来。理解框架设计的底层逻辑，不仅帮你选出合适的开源框架，更重要的是当开源方案不满足需求时，能自己设计一套。
- **前置知识**：Python 基础、了解 LangChain 基本用法（参考 [3-1 LangChain多任务应用开发](3-1_LangChain多任务应用开发.md)）、理解 Agent 概念（参考 [2-2-1 Function Calling与MCP](../02_AI应用技术/2-2_Agent理论知识+案例详解+实操/2-2-1_Function_Calling与MCP.md)）。

### 三大框架一句话定位

| 框架 | 核心定位 | 最适合 |
|------|---------|--------|
| **LangChain** | LLM 应用通用框架 | 需要灵活组合各种 LLM 模式的复杂应用 |
| **LlamaIndex** | 数据密集型 RAG 框架 | 以数据索引和检索为核心的应用 |
| **AutoGen** | 多智能体对话框架 | 多个 Agent 协作完成任务的场景 |

## 原理讲解

### 1. 框架设计的核心三要素

任何一个 AI 开发框架，本质上都在解决三个问题：

```
┌─────────────────────────────────────────┐
│               AI 开发框架                │
├─────────────────┬───────────────────────┤
│   核心组件抽象    │    数据流与控制流       │
│  (What)          │    (How)              │
├─────────────────┴───────────────────────┤
│         状态管理与性能考量                │
│         (When & How Fast)               │
└─────────────────────────────────────────┘
```

**核心组件抽象**：把 LLM、工具、记忆、提示词这些概念抽象成统一接口，方便替换和扩展。比如 LangChain 的 `BaseLLM` 抽象让切换 OpenAI → ChatGLM → Ollama 只需改一个参数。

**数据流与控制流**：定义数据在各组件间如何流转，以及谁来决定下一步执行什么。最简单的就是链式调用（Chain），复杂的有状态图（LangGraph）、多 Agent 协商（AutoGen）。

**状态管理与性能考量**：对话历史怎么存？并发请求怎么调度？缓存怎么设计？日志怎么追踪？这是"能用"到"好用"的关键。

### 2. LlamaIndex 深度解析

LlamaIndex 的核心是 **Data Ingestion + Index + Query** 三步：

```
原始数据 → 加载器(Loaders) → 解析器(Parsers) → 节点(Nodes)
                                                      ↓
用户提问 → 查询引擎(QueryEngine) ← 索引(Index) ← 向量化嵌入
              ↓
          生成回答
```

与 LangChain 的关键区别：
- LlamaIndex 默认是 **数据驱动**，一切围绕 Index 展开
- LangChain 默认是 **流程驱动**，一切围绕 Chain 展开
- LlamaIndex 的数据连接器（100+ 种）比 LangChain 更丰富

### 3. AutoGen 多智能体原理

AutoGen 的核心模式是 **Agent 对话即工作流**：

```
UserProxy ──提问──→ AssistantAgent
                      ↓
                  需要代码？
                      ↓
              UserProxy 执行代码
                      ↓
              结果返回 AssistantAgent
                      ↓
                  最终回答 → UserProxy
```

每个 Agent 有独立角色和权限，通过"对话"完成协作。不需要显式定义 Chain 或 Graph，Agent 自己商量着来。

### 4. 三框架对比决策树

```
需要做什么？
├── 构建数据密集型 RAG 应用（大量文档索引和检索）
│   → LlamaIndex（数据连接器多，索引结构丰富）
│
├── 多 Agent 协作，Agent 自主对话协商
│   → AutoGen（Agent 对话即工作流，自动协调）
│
├── 灵活组合 LLM 模式（RAG + Agent + Chain + Memory）
│   → LangChain（生态最全，组件最灵活）
│
└── 三者可组合使用
    → LlamaIndex 做索引 + LangChain 做编排 + AutoGen 做多 Agent
```

## 代码实战

> 依赖安装：`pip install llama-index langchain autogen-agentchat openai python-dotenv`

### 示例1：LlamaIndex 快速上手

```python
from llama_index.core import VectorStoreIndex, SimpleDirectoryReader
import openai

openai.api_key = "your-api-key"  # 或 export OPENAI_API_KEY

# 1. 加载数据（自动识别文件夹下的 txt/pdf/md 等）
documents = SimpleDirectoryReader("./docs").load_data()

# 2. 构建索引（自动向量化 + 存储）
index = VectorStoreIndex.from_documents(documents)

# 3. 创建查询引擎
query_engine = index.as_query_engine()

# 4. 提问
response = query_engine.query("这份文档的主要内容是什么？")
print(response)
# 预期输出: 基于文档内容的摘要
```

### 示例2：LlamaIndex 高级特性——自定义索引结构

```python
from llama_index.core import (
    VectorStoreIndex, SummaryIndex, TreeIndex,
    SimpleDirectoryReader, Settings
)
from llama_index.llms.openai import OpenAI
from llama_index.embeddings.openai import OpenAIEmbedding

# 全局配置
Settings.llm = OpenAI(model="gpt-3.5-turbo")
Settings.embed_model = OpenAIEmbedding(model="text-embedding-3-small")

documents = SimpleDirectoryReader("./docs").load_data()

# 三种索引结构对比
# 1. 向量索引——语义检索（最常用）
vector_index = VectorStoreIndex.from_documents(documents)

# 2. 摘要索引——先生成摘要再检索（适合需要"先看概览"的场景）
summary_index = SummaryIndex.from_documents(documents)

# 3. 树索引——分层级检索（适合有明确层级的知识）
# tree_index = TreeIndex.from_documents(documents)

# 组合查询：先用摘要索引看全局，再用向量索引查细节
summary_engine = summary_index.as_query_engine(response_mode="tree_summarize")
vector_engine = vector_index.as_query_engine(similarity_top_k=3)

print("=== 摘要视图 ===")
print(summary_engine.query("总结文档内容"))
print("\n=== 细节检索 ===")
print(vector_engine.query("文档中提到了哪些具体技术方案？"))
```

### 示例3：AutoGen 多 Agent 协作

```python
from autogen import AssistantAgent, UserProxyAgent
import os

os.environ["OPENAI_API_KEY"] = "your-api-key"

# 配置 LLM
llm_config = {
    "config_list": [{"model": "gpt-3.5-turbo"}],
    "temperature": 0
}

# 创建助手 Agent——负责规划和生成代码
assistant = AssistantAgent(
    name="assistant",
    llm_config=llm_config,
    system_message="你是一个Python数据分析专家。收到任务后，写出可运行的代码并解释结果。"
)

# 创建用户代理——负责执行代码和提供反馈
user_proxy = UserProxyAgent(
    name="user_proxy",
    human_input_mode="NEVER",  # 全自动模式
    max_consecutive_auto_reply=5,
    code_execution_config={
        "work_dir": "./coding",
        "use_docker": False  # 本地 Python 环境执行
    }
)

# 发起任务——两个 Agent 自动协作完成
user_proxy.initiate_chat(
    assistant,
    message="""
    我有一个销售数据列表:
    sales = [120, 340, 280, 450, 390, 510, 420, 380]
    
    请帮我:
    1. 计算总销售额、平均值、最高和最低
    2. 找出销售额超过400的天数
    3. 用 matplotlib 画出趋势图并保存为 sales_trend.png
    """
)
# 预期: assistant 生成代码 → user_proxy 执行 → 返回结果 → assistant 分析
```

### 示例4：三框架混合使用

```python
"""
实战场景：智能客服系统
- LlamaIndex 管理产品文档（知识库）
- LangChain 编排对话流程（含记忆）
- AutoGen 风格的 Agent 决策（工具调用）
"""

from langchain_openai import ChatOpenAI
from langchain.memory import ConversationSummaryMemory
from langchain.tools import tool
from langchain.agents import AgentExecutor, create_tool_calling_agent
from langchain_core.prompts import ChatPromptTemplate

# 模拟 LlamaIndex 知识库查询（实际使用时替换为真实索引）
@tool
def search_knowledge_base(query: str) -> str:
    """搜索产品知识库，返回相关文档"""
    kb = {
        "退货": "7天无理由退货，需保留原包装。联系客服获取退货地址。",
        "保修": "电子产品保修1年，人为损坏不在保修范围。",
        "发货": "下单后48小时内发货，顺丰包邮。",
    }
    for key, value in kb.items():
        if key in query:
            return value
    return "未找到相关信息，建议转人工客服。"

@tool
def calculate_refund(amount: float, days: int) -> str:
    """计算退款金额，7天内全额，7-15天扣10%"""
    if days <= 7:
        refund = amount
    elif days <= 15:
        refund = amount * 0.9
    else:
        refund = 0
    return f"应退款: ¥{refund:.2f}（购买{days}天，扣款¥{amount - refund:.2f}）"

tools = [search_knowledge_base, calculate_refund]

llm = ChatOpenAI(model="gpt-3.5-turbo")
memory = ConversationSummaryMemory(
    llm=llm, return_messages=True, max_token_limit=500
)

prompt = ChatPromptTemplate.from_messages([
    ("system", "你是电商智能客服。先用知识库查政策，涉及金额就计算退款。态度友善。"),
    ("user", "{input}"),
    ("placeholder", "{agent_scratchpad}")
])

agent = create_tool_calling_agent(llm, tools, prompt)
executor = AgentExecutor(agent=agent, tools=tools, verbose=True)

print(executor.invoke({"input": "我买了一台3000元的电脑，用了5天想退货，能退多少钱？"}))
# 预期: 先查退货政策 → 计算退款 → 返回3000元+退货流程
```

## 进阶方向

### 1. 当前局限性

- 框架间的 API 设计理念不同，混合使用有学习成本和潜在的兼容问题
- 自研框架工作量巨大——除非团队有专门的基础设施组，否则优先用开源方案
- AutoGen 的多 Agent 对话开销大，Token 消耗约为单 Agent 的 3-5 倍

### 2. 下一步学习

| 主题 | 说明 | 相关节点 |
|------|------|----------|
| 自研轻量框架 | 当开源框架不满足需求时的替代方案 | 本文进阶 |
| LangGraph | LangChain 生态的状态图引擎 | [3-1 LangChain](3-1_LangChain多任务应用开发.md) |
| HuggingFace 生态 | 模型管理与微调的基础设施 | [3-3 HuggingFace](3-3_HuggingFace生态实战从模型应用到高效微调.md) |
| MCP/A2A 协议 | Agent 间通信的开放标准 | [2-2-1 Function Calling与MCP](../02_AI应用技术/2-2_Agent理论知识+案例详解+实操/2-2-1_Function_Calling与MCP.md) |

### 3. 自研轻量框架的核心思路

当开源方案不满足时，不一定要从头造轮子。可以从 **最小可用核心** 开始：

```python
# 最小可用框架核心（< 50 行）
class Tool:
    def __init__(self, name, description, func):
        self.name = name
        self.description = description
        self.func = func
    
    def run(self, input_str: str) -> str:
        return self.func(input_str)

class Agent:
    def __init__(self, llm, tools: list[Tool], system_prompt: str):
        self.llm = llm
        self.tools = {t.name: t for t in tools}
        self.system_prompt = system_prompt
    
    def run(self, user_input: str, max_steps: int = 5) -> str:
        messages = [{"role": "system", "content": self.system_prompt}]
        # ... ReAct 循环实现（核心逻辑 ~30 行）
        pass
```

真正的工业级框架需要在此基础上加：异步支持、流式输出、错误重试、并发控制、可观测性。

## 常见问题

### 小白最常踩的坑

1. **框架焦虑**：初学者容易纠结"该学哪个框架"。实际上 LangChain 是起点——生态最大、教程最多、社区最活跃。学会一个后，其他框架的设计理念都是相通的
2. **过度抽象**：简单的一问一答不需要 Chain/Agent，直接调 API 就行。被框架绑架会徒增复杂度
3. **版本更新踩坑**：LangChain 的 API 变更频繁（0.1 → 0.2 → 0.3 部分不兼容），写代码时务必锁定版本号，看教程时注意发布日期

### 自检题

**Q1**：LangChain、LlamaIndex、AutoGen 各自最适合什么场景？

<details><summary>答案</summary>

- LangChain：通用 LLM 应用，需要灵活组合 Chain/Agent/RAG/Memory 的复杂场景
- LlamaIndex：数据密集型 RAG，大量文档索引和检索为核心的应用
- AutoGen：多个 Agent 需要自主对话协作的任务
</details>

**Q2**：设计一个 AI 框架需要考虑哪三要素？

<details><summary>答案</summary>

1. 核心组件抽象：统一 LLM/Tool/Memory 等接口，方便替换扩展
2. 数据流与控制流：编排组件间的调用顺序和执行逻辑
3. 状态管理与性能：对话历史、并发、缓存、日志等
</details>

**Q3**：什么情况下应该考虑自研框架而不是用开源方案？

<details><summary>答案</summary>

- 业务场景极其特殊，开源框架的抽象反而增加工作量
- 对性能有极致要求（如毫秒级延迟），需要去框架化
- 团队有基础设施组且需要长期维护
- 一般原则：先用开源，不够用时再自研；不要一开始就造轮子
</details>
