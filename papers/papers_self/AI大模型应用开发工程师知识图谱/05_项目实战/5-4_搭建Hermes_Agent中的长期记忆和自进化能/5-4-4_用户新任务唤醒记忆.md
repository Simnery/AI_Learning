# 5.4.4 用户新任务唤醒记忆

> 5.4.4 用户新任务唤醒记忆 > 5.4 Hermes长期记忆 > 5. 项目实战

## 核心概念

- **是什么**：当用户发起新任务时，Hermes 自动检索相关记忆并注入当前上下文——让 Agent 不是从零开始，而是带着"过去的经验"来处理新任务。这是记忆系统的最终价值：不是存着好看，而是在需要时准确唤醒。
- **为什么重要**：没有记忆唤醒，每次对话都是孤立的。用户得反复重申偏好。有了它，Agent 自动适配用户习惯，越用越聪明。

## 原理讲解

### 上下文动态构建

```
用户新任务
    ↓
提取任务关键词 → 生成查询向量
    ↓
LanceDB 混合检索（各层记忆）
    ↓
按层级+相关性排序
    ↓
裁剪到上下文窗口限制
    ↓
注入当前对话的 system prompt 或前缀
```

### 注入策略

```python
def build_context(user_task: str, retrieved: list[dict], max_tokens: int = 2000):
    """动态构建上下文——在 token 限制内装最多有用信息"""
    context_parts = ["## 相关历史记忆\n"]
    token_count = 0
    
    # 优先级: Tier4 > Tier3 > Tier2（最精炼的先装）
    retrieved.sort(key=lambda x: (x["tier"], -x["score"]))
    
    for mem in retrieved:
        tok_est = len(mem["content"]) // 2  # 粗略估算 token 数
        if token_count + tok_est > max_tokens:
            break
        context_parts.append(f"- [{mem['type']}] {mem['content']}")
        token_count += tok_est
    
    return "\n".join(context_parts)
```


### 3. 新任务唤醒记忆流程

```
用户新消息 → 意图分类（是否需历史）
     ↓ 是
query_rewrite(当前句 + 上轮实体)
     ↓
LanceDB 检索 Top-5（5-4-3）→ 按 layer 排序拼接
     ↓
注入 System Prompt「相关记忆：…」→ 主 Agent 执行
```

**注意**：唤醒记忆总长建议 ≤1500 token，避免挤占工具结果上下文。

### 4. 唤醒 vs 心跳

| | 心跳 5-4-1 | 唤醒 5-4-4 |
|--|-----------|------------|
| 触发 | 定时 | 用户新任务 |
| 方向 | 写入记忆 | 读取记忆 |

## 代码实战

```python
# pip install numpy lancedb

class MemoryWakeup:
    """新任务唤醒记忆——完整流程"""
    
    def __init__(self, store: LanceDBMemoryStore, embedder, llm):
        self.store = store
        self.embedder = embedder
        self.llm = llm
    
    def wake(self, user_task: str, max_memories: int = 5) -> str:
        """唤醒相关记忆并构建增强上下文"""
        # 1. 生成查询向量
        query_embedding = self.embedder.embed(user_task)
        
        # 2. 混合检索（优先高层记忆）
        t4 = self.store.hybrid_search(user_task, query_embedding, tier=4, top_k=2)
        t3 = self.store.hybrid_search(user_task, query_embedding, tier=3, top_k=2)
        t2 = self.store.hybrid_search(user_task, query_embedding, tier=2, top_k=2)
        
        all_memories = t4 + t3 + t2  # T4 优先（最精炼）
        
        # 3. 去重 + 排序
        seen = set()
        unique = []
        for m in all_memories:
            key = m["content"][:50]
            if key not in seen:
                seen.add(key)
                unique.append(m)
        
        unique = unique[:max_memories]
        
        # 4. 构建增强上下文
        memory_context = "\n".join(
            f"[{m['type']}][T{m['tier']}] {m['content']}" 
            for m in unique
        )
        
        print(f"唤醒了 {len(unique)} 条相关记忆")
        return memory_context
    
    def respond_with_memory(self, user_task: str) -> str:
        """带记忆增强的回复"""
        memory_context = self.wake(user_task)
        
        prompt = f"""{memory_context}

---
当前任务: {user_task}

请结合上述历史记忆回答。如果记忆中有用户偏好，请遵循。如果没有相关信息，正常回答。"""
        
        return self.llm.complete(prompt)


# 完整示例
"""
# 初始化
store = LanceDBMemoryStore()
embedder = OpenAIEmbeddings()
llm = ChatOpenAI()
wakeup = MemoryWakeup(store, embedder, llm)

# 第1次交互：用户表达偏好
# → 心跳收集 → 四层提炼 → LanceDB 存储

# 第2次交互：新任务自动唤醒
response = wakeup.respond_with_memory("帮我写一个数据处理脚本")
# → 自动从记忆中检索到用户偏好 numpy，使用简洁代码风格
"""
```

## 进阶方向

### 当前方案的局限性

- **关键词提取粗糙**：当前用关键词匹配提取任务特征，复杂任务的关键词可能无法准确概括意图
- **记忆相关性判断不准**：向量检索可能返回"相似但不相关"的记忆——比如任务涉及"Python 数据清洗"却唤醒了"Python 游戏开发"的记忆
- **上下文窗口浪费**：某条记忆塞进上下文后被 LLM 完全忽略，说明它不相关但占用了宝贵的 token 预算
- **无反馈闭环**：唤醒了记忆但不知道它是否真的帮助了当前任务，无法优化唤醒策略

- **记忆消融测试**：定期做 A/B 测试——关闭记忆唤醒的对照组 vs 开启记忆唤醒的实验组，量化记忆的实际贡献
- **唤醒日志分析**：记录每次任务唤醒了哪些记忆、LLM 最终使用了哪些，分析记忆的有效利用率
- **记忆过期策略**：对超过 N 天未被唤醒的记忆降低检索权重，超过 M 天的自动归档

## 常见问题

### 小白最常踩的 3 个坑

**坑 1：不管任务类型，统一唤醒所有层级的记忆**

简单任务（如"格式化这段代码"）不需要深度记忆，只要 Tier4 的一句话偏好。复杂任务（如"重构整个模块"）需要 Tier2+Tier3 的详细记忆。应该根据任务复杂度动态调整检索层。

**坑 2：记忆塞满上下文窗口，挤压了实际问题空间**

Token 有上限。如果唤醒的记忆占了 50% token，留给用户问题和 LLM 回答的空间只有 50%，回答质量会下降。建议记忆注入量不超过总 token 的 20%。

**坑 3：没有处理"记忆冲突"**

昨天记的"用户喜欢面向对象风格"和今天用户说的"用函数式风格改写"冲突了。应该优先信任最近的用户行为，同时在上下文中标注"根据最近行为调整..."。

### 自检题

**Q1**：上下文动态构建为什么优先装 Tier4？<details><summary>答案</summary>Tier4 是最精炼的一句话记忆，信息密度最高。在 token 预算有限时，先装高密度信息，有余量再装详细记忆。按 T4→T3→T2 优先级排序。</details>

**Q2**：四种记忆类型的唤醒优先级是什么？<details><summary>答案</summary>preference（用户偏好）> decision（重要决策）> lesson（经验教训）。用户偏好直接影响当前任务的执行方式，优先级最高。</details>

**Q3**：如果没有检索到相关记忆怎么办？<details><summary>答案</summary>就当正常对话处理，不做任何记忆增强。记忆系统的原则是"有帮助就用，没有不强塞"。虚假的相关记忆比没有记忆更糟糕。</details>
