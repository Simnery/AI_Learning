# 5.4.2 整理成4层颗粒度记忆文字

> 5.4.2 4层颗粒度记忆 > 5.4 Hermes长期记忆 > 5. 项目实战

## 核心概念

- **是什么**：四层颗粒度是 Hermes 记忆系统的核心结构设计——从粗糙的原始日志到精炼的知识洞察，层层提炼。不同查询场景匹配不同层级：快速回复用高层摘要，深度分析用底层原文。
- **为什么重要**：如果只存一层（全文），检索要么太泛（找不到精确信息）要么太杂（返回过多噪声）。四层结构像金字塔，在"精准"和"全面"之间取得平衡。

## 原理讲解

### 四层记忆结构

```
Tier 4 (知识层): "用户喜欢简洁的代码风格"    ← 最短，最精炼
Tier 3 (洞察层): "用户偏好：代码注释少、变量名见名知意、单函数不超过30行" ← 中等
Tier 2 (摘要层): "在这次Python重构任务中，用户多次指出..." ← 较长
Tier 1 (原文层): 原始对话记录的完整文本         ← 最长，最完整
```

### Tier2 → Tier3 的自动扩展

```
Tier2: "用户要求代码性能优化，使用numpy替代循环"
  ↓ LLM 自动提炼
Tier3: "用户偏好: 1)性能优先于可读性 2)数据分析场景用numpy 3)循环>100次考虑向量化"
```

提炼 prompt: "从以下摘要中提取可泛化的规律、用户偏好或方法论。"

### 检索策略

| 查询类型 | 检索层 | 原因 |
|---------|--------|------|
| 快速回忆 | Tier 4 | 一句话回答，精准 |
| 相关信息 | Tier 3 + Tier 4 | 有细节但不过载 |
| 完整上下文 | Tier 2 + Tier 3 | 需要理解前因后果 |
| 精确引用 | Tier 1 | 需要原始对话 |


### 3. 四层记忆颗粒度（Hermes 约定）

| 层级 | 名称 | 典型内容 | 检索方式 |
|:---:|------|----------|----------|
| L0 | 原始片段 | 对话原文、工具返回 | 时间序 |
| L1 | 摘要 | 单轮/单任务摘要 | 关键词 |
| L2 | 主题记忆 | 用户偏好、项目设定 | 向量 |
| L3 | 原则 | 长期规则（「总是用中文」） | 高权重置顶 |

心跳队列中的条目经 LLM **合并、去重、升级层级** 后写入 L1–L3（见代码 `consolidate()`）。

### 4. 与 5-4-3 的衔接

L2/L3 文本 → Embedding → LanceDB 表 `hermes_memory`；L0 可只存对象存储路径。

## 代码实战

```python
# pip install — 本示例仅用 Python 3.10+ 标准库

class FourTierMemory:
    def __init__(self, llm):
        self.llm = llm
        self.memories = {tier: [] for tier in range(1, 5)}
    
    def ingest(self, raw_text: str):
        """摄入原始文本，自动生成四层记忆"""
        # Tier 1: 原文（截断到合理长度）
        t1 = raw_text[:2000]
        self.memories[1].append({"content": t1, "ts": datetime.now()})
        
        # Tier 2: LLM 生成摘要
        t2 = self.llm.complete(f"用2-3句话总结以下内容:\n{t1}")
        self.memories[2].append({"content": t2, "ts": datetime.now(), "source_t1_id": len(self.memories[1])-1})
        
        # Tier 3: 提取可泛化的规律/偏好（自动扩展）
        t3 = self.llm.complete(f"从以下摘要提取可泛化的规律、用户偏好或方法论:\n{t2}")
        self.memories[3].append({"content": t3, "ts": datetime.now()})
        
        # Tier 4: 一句话精华
        t4 = self.llm.complete(f"用一句话概括以下内容的核心要点:\n{t3}")
        self.memories[4].append({"content": t4, "ts": datetime.now()})
    
    def retrieve(self, query: str, tier: int = 3, top_k: int = 3) -> list[str]:
        """从指定层检索相关记忆"""
        candidates = self.memories.get(tier, [])
        if not candidates:
            return []
        
        # 简化的关键词匹配（生产环境用向量检索）
        scored = []
        for m in candidates:
            score = sum(1 for w in query if w in m["content"])
            scored.append((score, m["content"]))
        
        scored.sort(key=lambda x: x[0], reverse=True)
        return [s[1] for s in scored[:top_k]]
```

运行上述代码后，终端应看到类似以下结果（具体数值因环境与输入而异）：


## 进阶方向

### 当前方案的局限性

- **四层之间无双向链接**：Tier4 应该能追溯到 Tier1 的原始对话，当前只有单向引用
- **提炼质量依赖 LLM**：Tier2→Tier3 的自动提炼可能丢失重要细节或产生幻觉
- **层级固定不灵活**：某些场景可能需要 3 层或 5 层，固定的 4 层结构不够灵活
- **检索策略简单**：当前用关键词匹配，生产环境需要用向量检索 + 混合排序

### 工业界最佳实践

- **渐进式提炼**：不是一次性生成四层，而是 Tier1→Tier2（实时），Tier2→Tier3（空闲时批量），Tier3→Tier4（定期审查），平衡实时性和质量
- **人工审核层**：对 Tier4 知识层设置人工审核机制，避免 LLM 提炼出错误的"用户偏好"
- **记忆冲突检测**：新记忆与已有 Tier3/Tier4 内容矛盾时（如用户偏好变了），触发更新而非追加

## 常见问题

### 小白最常踩的 3 个坑

**坑 1：四层记忆内容高度重复**

如果 Tier3 和 Tier4 的内容几乎一样，说明提炼不够。Tier3 应该是"可操作的规律"（怎么做），Tier4 应该是"一句话原则"（为什么）。例如：Tier3 = "用户要求代码中循环 > 100 次时用 numpy 向量化"，Tier4 = "用户是性能优先型开发者"。

**坑 2：只记用户的显性偏好，不记隐性模式**

用户说"我喜欢简洁代码"是显性偏好。用户连续 5 次重构都删除注释、缩短变量名，是隐性模式——比显性偏好更真实。Tier3 的提炼 prompt 要包含"从行为模式中推断偏好"。

**坑 3：Tier1 原文不限长度，塞爆存储**

原始对话 2000 字、代码 diff 3000 行、日志 5000 行...如果不限制 Tier1 长度，存储很快爆炸。Tier1 应该只存关键片段（前后 N 轮对话），完整原文放外部存储（日志系统），Tier1 只存引用。

### 自检题

**Q1**：四层记忆分别是什么？各自适合什么检索场景？<details><summary>答案</summary>Tier4(知识层)：一句话精华，快速回忆。Tier3(洞察层)：可泛化规律，深度分析。Tier2(摘要层)：中等粒度，需要前因后果。Tier1(原文层)：原始对话，精确引用。</details>

**Q2**：Tier2→Tier3 的自动扩展做了什么？<details><summary>答案</summary>把具体的摘要提炼为可泛化的规律/偏好/方法论。例如从"用户要求用numpy优化循环"提炼出"用户偏好numpy数据处理，循环>100考虑向量化"。</details>

**Q3**：为什么需要分层而不是只存一层全文？<details><summary>答案</summary>如果只存全文，检索时要么返回过长的原文（包含大量噪声），要么返回太短的关键词匹配（丢失上下文）。分层检索可以根据查询需求选择合适粒度：快速回答用 Tier4，深度分析用 Tier2+Tier3。</details>
