# 5.4.1 心跳唤醒时，自主收集需要整理的记忆

> 5.4.1 心跳唤醒 > 5.4 Hermes长期记忆 > 5. 项目实战

## 核心概念

- **是什么**：心跳机制是 Hermes Agent 的"潜意识"——每隔一段时间（心跳间隔）自动唤醒，检查最近的对话和操作，识别有价值的信息并标记为"待整理记忆"。心跳频率自适应调整：活跃时高频，空闲时低频。
- **为什么重要**：没有心跳机制，Agent 只会在用户主动触发时才"回忆"，大量隐性知识白白流失。心跳让 Agent 像人一样在"空闲时"自动整理经验。

## 原理讲解

### 1. 自适应心跳频率

```
活跃期: 每 5 分钟一次心跳（用户正在高频交互）
半活跃期: 每 30 分钟一次（最近 2 小时有交互）
休眠期: 每 4 小时一次（长时间无交互）
```

自适应公式: `interval = base_interval * (1 + inactivity_ratio * 2)`，不活跃时间越长间隔越大。

### 2. 记忆价值评估算法

每条候选记忆根据三个维度打分：
- **重复度**（0-1）：类似记忆出现频率，越高越值得记
- **重要性**（0-1）：是否涉及关键决策、用户偏好、经验教训
- **衰减度**（0-1）：随时间衰减，新的记忆权重高

`价值分 = 0.4×重复度 + 0.4×重要性 + 0.2×(1-衰减度)`

分数 > 0.6 → 进入整理队列。


### 3. Hermes 心跳在流水线中的位置

```
用户对话/工具调用 → recent_interactions 环形缓冲
        ↓ 定时心跳（自适应间隔）
   collect_candidates() → LLM 提取 preference/decision/lesson
        ↓ evaluate_memory() 分值>0.6
   memory_queue（待整理，应持久化）→ 5-4-2 四层整理
```

**工程要点**：队列上限 200；入队即写磁盘；活跃期先用关键词预筛选再调 LLM。

### 4. 心跳调度伪代码

| 阶段 | 间隔 | 触发条件 |
|------|------|----------|
| 活跃 | 5 min | 最近 5 min 有交互 |
| 半活跃 | 30 min | 2h 内有交互 |
| 休眠 | 4 h | 长时间无交互 |

## 代码实战

```python
# pip install — 本示例仅用 Python 3.10+ 标准库

import time
from datetime import datetime, timedelta
from collections import deque

class HeartbeatMemory:
    def __init__(self, llm):
        self.llm = llm
        self.last_active = datetime.now()
        self.recent_interactions = deque(maxlen=100)
        self.memory_queue = []
    
    def adaptive_interval(self) -> int:
        """自适应心跳间隔（秒）"""
        idle_minutes = (datetime.now() - self.last_active).total_seconds() / 60
        if idle_minutes < 5:
            return 300       # 5 分钟
        elif idle_minutes < 120:
            return 1800      # 30 分钟
        else:
            return 14400     # 4 小时
    
    def collect_candidates(self) -> list[dict]:
        """收集待评估的记忆候选"""
        if len(self.recent_interactions) < 5:
            return []
        
        recent = list(self.recent_interactions)[-20:]
        text = "\n".join(str(r) for r in recent)
        
        prompt = f"""从以下交互中提取有价值的信息点（用户偏好、重要决策、经验教训）:
{text}
输出 JSON: [{{"content": "信息点", "type": "preference|decision|lesson"}}]"""
        import json
        return json.loads(self.llm.complete(prompt))
    
    def evaluate_memory(self, candidate: dict) -> float:
        """评估记忆价值"""
        # 重复度：在近期交互中出现的次数
        repeat_count = sum(1 for r in self.recent_interactions 
                          if candidate["content"][:20] in str(r))
        repeat_score = min(repeat_count / 5, 1.0)
        
        # 重要性分类
        importance_map = {"preference": 0.8, "decision": 0.9, "lesson": 1.0}
        importance = importance_map.get(candidate["type"], 0.5)
        
        # 衰减（新记忆分数高）
        recency = 1.0  # 刚从最近交互中提取
        
        score = 0.4 * repeat_score + 0.4 * importance + 0.2 * recency
        return round(score, 2)
    
    def heartbeat(self):
        """执行一次心跳"""
        print(f"[心跳] {datetime.now().strftime('%H:%M:%S')}")
        candidates = self.collect_candidates()
        
        for c in candidates:
            score = self.evaluate_memory(c)
            if score > 0.6:
                c["score"] = score
                self.memory_queue.append(c)
                print(f"  收集记忆: {c['content'][:50]}... (分数: {score})")
        
        return len(self.memory_queue)


class MockLLM:
    def complete(self, prompt: str) -> str:
        import json
        return json.dumps([
            {"content": "用户偏好简洁回复", "type": "preference"},
            {"content": "数据库查询需加超时", "type": "lesson"},
        ])


if __name__ == "__main__":
    hb = HeartbeatMemory(MockLLM())
    for _ in range(8):
        hb.recent_interactions.append("用户: 查询销售数据")
    print("interval_sec =", hb.adaptive_interval())
    n = hb.heartbeat()
    print("queued =", n)
```

### 预期输出

```

[心跳] HH:MM:SS
  收集记忆: 用户偏好简洁回复... (分数: 0.72)
  收集记忆: 数据库查询需加超时... (分数: 0.76)
queued = 2
```

## 进阶方向

### 当前方案的局限性

- **评分函数单一**：当前用固定权重（0.4+0.4+0.2），不同用户和场景可能需要不同权重
- **冷启动问题**：新用户没有历史数据时，心跳收集的记忆质量低
- **仅文本记忆**：不支持代码片段、图片、表格等结构化记忆的收集
- **单 Agent 心跳**：Hermes 多 Agent 场景下，每个 Agent 独立心跳可能导致重复收集

### 工业界最佳实践

- **心跳间隔动态调参**：用近期交互频率的滑动平均计算间隔，避免极端值干扰
- **记忆去重**：新记忆入库前与已有记忆做相似度比对，相似度 > 0.9 的合并而非新增
- **分级告警**：如果心跳连续 3 次收集到 0 条记忆，可能表示评估阈值太高，触发降阈

## 常见问题

### 小白最常踩的 3 个坑

**坑 1：心跳频率设太密，LLM 调用量爆炸**

每次心跳都调一次 LLM 做记忆评估。如果心跳 5 分钟一次，一天就是 288 次调用。建议：活跃期用规则预筛选（关键词匹配、长度阈值），只有通过预筛选的才送 LLM 评估。

**坑 2：不限制 memory_queue 大小，内存无限增长**

心跳收集的记忆先放在队列里等待整理。如果长时间不整理（用户一直活跃），队列可能膨胀到几千条。必须设 max_queue_size（建议 200）和自动触发整理的阈值（队列 > 100 时强制整理）。

**坑 3：重启丢失未整理的记忆队列**

如果 memory_queue 只存在内存里，进程重启就全丢了。心跳收集的记忆在入队后应该立即持久化（写到磁盘/数据库），等整理完成后再标记为已处理。

### 自检题

**Q1**：为什么心跳频率要自适应调整？<details><summary>答案</summary>活跃期高频心跳保证不遗漏记忆，空闲期降低频率节省资源。固定频率要么太密（浪费）要么太稀（遗漏）。</details>

**Q2**：记忆价值评估的三个维度是什么？<details><summary>答案</summary>重复度（类似记忆出现频率）、重要性（是否涉及关键决策/偏好/教训）、衰减度（新记忆权重高）。</details>

**Q3**：心跳机制最大的工程挑战是什么？<details><summary>答案</summary>在"不遗漏记忆"和"不浪费 LLM 调用"之间找平衡。纯靠 LLM 评估太贵，纯靠规则太粗糙。最佳方案是规则预筛选 + LLM 精评估 + 自适应频率。</details>
