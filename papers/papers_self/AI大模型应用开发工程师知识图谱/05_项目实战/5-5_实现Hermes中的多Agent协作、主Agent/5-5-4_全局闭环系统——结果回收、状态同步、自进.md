# 5.5.4 全局闭环系统——结果回收、状态同步、自进化基础底座

> 5.5.4 全局闭环系统 > 5.5 Hermes多Agent > 5. 项目实战

## 核心概念

- **是什么**：全局闭环是多Agent架构的"事后引擎"——每次任务执行完毕后自动复盘执行轨迹，提炼可复用经验和流程，生成个性化技能。这是Agent从"每次从零开始"到"越用越聪明"的关键。
- **为什么重要**：没有闭环，多Agent系统就是一次性执行器。有了闭环，失败教训不再重复，成功模式自动沉淀。

## 原理讲解

### 自进化流程

```
任务执行轨迹 → 事后复盘 → 自动提炼流程 → 生成个性化技能
                                                     ↓
                                             下次任务自动加载
```

| 阶段 | 做什么 | 产出 |
|------|--------|------|
| 记录轨迹 | 谁做了什么、耗时、成败、决策点 | 原始日志 |
| 事后复盘 | 哪些可优化？哪些失败可预防？ | 洞察报告 |
| 提炼流程 | 可复用的任务模板 | Spec模板 |
| 生成技能 | 针对用户/场景的特殊处理 | 自动触发规则 |


### 3. 全局闭环：回收与自进化

```
所有子任务 terminal → aggregate_results() → 用户可见回复
        ↓
失败率>阈值 → 写入「教训」记忆（5-4-2 L3）
成功模式 → 更新「技能模板」供下次 decompose 复用
```

### 4. 自进化边界

| 允许 | 禁止 |
|------|------|
| 调整 prompt 片段、拆解模板 | 自动改生产代码/删数据 |
| 记忆层沉淀 | 无人工审核上线新工具 |

## 代码实战

```python
# pip install — 本示例仅用 Python 3.10+ 标准库

"""全局闭环——自进化引擎"""
import json
from datetime import datetime

class EvolutionEngine:
    def __init__(self, llm, memory_store):
        self.llm = llm
        self.memory = memory_store
        self.traces = []
    
    def record_trace(self, task_id: str, trace: dict):
        self.traces.append({"task_id": task_id, 
                           "timestamp": datetime.now().isoformat(), "trace": trace})
    
    def retrospective(self, task_id: str) -> dict:
        traces = [t for t in self.traces if t["task_id"] == task_id]
        prompt = f"""复盘多Agent任务执行轨迹，输出 JSON:
{{"what_worked":[], "what_failed":[], "optimizations":[],
  "reusable_template":"", "new_skill":{{"name":"", "trigger":"", "action":""}}}}
轨迹: {json.dumps(traces, ensure_ascii=False)}"""
        return json.loads(self.llm.complete(prompt))
    
    def auto_evolve(self):
        for trace in self.traces[-10:]:
            insights = self.retrospective(trace["task_id"])
            if insights.get("reusable_template"):
                self.memory.add_memory(tier=3, content=insights["reusable_template"])
            if insights.get("new_skill"):
                self.memory.add_memory(tier=4, content=str(insights["new_skill"]))


class StateSynchronizer:
    """全局状态同步"""
    def __init__(self):
        self.global_state = {}
        self.agent_health = {}
    
    async def sync_task(self, agent: str, task_id: str, status: str):
        self.global_state[task_id] = {"agent": agent, "status": status}
    
    async def heartbeat(self, agent: str) -> bool:
        last = self.agent_health.get(agent, {}).get("last_beat")
        alive = last and (datetime.now() - last).seconds < 60
        self.agent_health[agent] = {"last_beat": datetime.now(), "alive": alive}
        return alive
```

## 进阶方向

### 当前方案的局限性

- **复盘依赖 LLM**：LLM 对执行轨迹的理解可能偏差，错误地把偶然失败标记为"需要改进的模式"
- **技能泛化难**：从一次任务中提炼的"skill"可能在下次相似任务中不适用，缺少泛化验证
- **进化方向失控**：自动进化的技能越来越多，可能相互冲突或降低系统可预测性
- **冷启动慢**：新用户/新场景没有历史轨迹，自进化底座闲置

### 下一步学什么

1. **[5-4 Hermes长期记忆](../5-4_搭建Hermes_Agent中的长期记忆和自进化能/)** — 自进化技能如何持久化存储和检索
2. **[2-2-3 Agent评估](../../02_AI应用技术/2-2_Agent理论知识+案例详解+实操/2-2-3_Agent的能力优化与效果评估.md)** — 如何验证自进化后的效果是否真的变好
3. **[Reinforcement Learning from Feedback](https://arxiv.org/abs/2203.02155)** — RLHF 的思路可用于 Agent 进化方向的有监督控制

### 工业界最佳实践

- **进化审批机制**：自动提炼的技能先进入"候选区"，经过人工确认或 A/B 测试验证后再正式上线
- **技能衰减**：超过 N 天未被触发的技能自动降级，超过 M 天删除，防止技能膨胀
- **回滚能力**：每次自进化保留上一版本快照，新技能导致系统退化时可一键回滚

## 常见问题

### 小白最常踩的 3 个坑

**坑 1：进化过度——系统越用越"固执"**

连续 3 次相似任务成功后自动固化了执行模式，但第 4 次场景变了，系统仍按旧模式执行。应该在每次任务前做"场景匹配度"检查，匹配度低于阈值就放弃历史经验。

**坑 2：只记录成功轨迹，不记录失败轨迹**

失败才是最有价值的学习材料。"为什么失败"比"为什么成功"更能防止未来踩坑。复盘时必须同等对待成功和失败两个方向的轨迹。

**坑 3：状态同步不同步——Agent 活在平行宇宙**

Agent A 修改了知识库，Agent B 不知道，继续用旧知识回答用户。必须建立变更通知机制——知识变更后推送给所有订阅者。

### 自检题

**Q1**：自进化底座的四步流程是什么？<details><summary>答案</summary>记录执行轨迹→事后复盘分析→自动提炼可复用流程→生成个性化技能。每次执行都为下次积累经验。</details>

**Q2**：状态同步包含哪些维度？<details><summary>答案</summary>任务状态（实时）、Agent健康（心跳30s）、知识/规则变更（变更时推送）、用户偏好更新（心跳触发）。</details>

**Q3**：自进化最大的风险是什么？<details><summary>答案</summary>失去控制——系统自动进化出的行为可能不适合新场景或与用户期望冲突。必须引入进化审批、效果验证和回滚机制，确保进化方向可控。</details>
