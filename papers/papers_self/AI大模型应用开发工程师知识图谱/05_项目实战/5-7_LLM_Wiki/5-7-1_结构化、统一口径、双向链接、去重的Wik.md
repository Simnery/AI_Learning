# 5.7.1 结构化、统一口径、双向链接、去重的Wiki文档

> 5.7.1 Wiki文档 > 5.7 LLM Wiki > 5. 项目实战

## 核心概念

- **是什么**：LLM Wiki 是用 AI 生成和维护的结构化知识库——不是简单地把文档扔进向量库，而是让 LLM 理解、整理、去重、关联知识，产出类似维基百科质量的文档。
- **为什么重要**：传统 RAG 知识库是"一堆碎片"，检索到什么都拼给 LLM。LLM Wiki 是"编译后的知识"——统一口径、消除矛盾、建立链接——让 AI 检索到的是高质量语义单元。

## 原理讲解

### 1. 双向链接与知识图谱

```
文档A ──引用──→ 文档B
  ↑              │
  └──被引用──────┘

每个文档自动生成:
- [[链接到其他文档]] (正向链接)
- "被以下文档引用" (反向链接)
- 知识图谱可视化
```

### 2. 智能去重与冲突解决

当两篇文档描述同一概念但内容矛盾时：

```
冲突检测 → LLM评估 → 方案选择:
  - 合并：两篇内容互补 → 合并为新文档
  - 选择：一篇明显更准确 → 淘汰另一篇
  - 标记：无法判断 → 标记冲突，人工裁决
```


### 3. LLM Wiki 文档规范

| 规则 | 说明 |
|------|------|
| 统一口径 | 术语表锁定「客户=付费用户」等 |
| 双向链接 | `[[订单流程]]` 自动图谱 |
| 去重 | 相似段落 >0.85 合并 |
| 结构化 frontmatter | title, tags, owner, updated |

### 4. 与经典 RAG 差异

Wiki 强调 **人可读 + 可编辑**；向量库是衍生索引（5-7-4），不是唯一真相源。

## 代码实战

```python
# pip install — 本示例仅用 Python 3.10+ 标准库

"""LLM Wiki 文档管理"""
class WikiManager:
    def __init__(self, llm):
        self.llm = llm
        self.documents = {}  # id → {content, links, backlinks}
    
    def add_document(self, title: str, content: str) -> str:
        """添加文档，自动提取双向链接"""
        doc_id = self._generate_id(title)
        
        # 提取 [[wiki链接]]
        import re
        links = re.findall(r'\[\[(.+?)\]\]', content)
        
        self.documents[doc_id] = {
            "title": title, "content": content,
            "links": links, "backlinks": []
        }
        
        # 更新被引用文档的反向链接
        for linked_title in links:
            linked_id = self._find_by_title(linked_title)
            if linked_id:
                self.documents[linked_id]["backlinks"].append(doc_id)
        
        return doc_id
    
    def check_conflict(self, doc_a: str, doc_b: str) -> dict:
        """检测两篇文档是否冲突"""
        prompt = f"""判断以下两篇文档是否存在冲突，输出 JSON:
{{"has_conflict": true/false,
 "conflict_point": "冲突点描述",
 "resolution": "建议解决方案（merge/select_a/select_b/manual）"}}

文档A: {doc_a}
文档B: {doc_b}"""
        import json
        return json.loads(self.llm.complete(prompt))
    
    def merge_documents(self, doc_ids: list[str]) -> str:
        """合并多篇文档为统一版本"""
        contents = "\n---\n".join(
            self.documents[did]["content"] for did in doc_ids)
        prompt = f"合并以下文档，消除矛盾，统一口径:\n{contents}"
        return self.llm.complete(prompt)
```

## 进阶方向

### 当前方案的局限性

- **链接提取粗糙**：用正则 `[[...]]` 提取链接，但真实文本中很少用 Wiki 语法，需要 LLM 自动识别实体并建立链接
- **冲突检测一步到位**：两篇文档直接比较，但大规模知识库中 O(n²) 比较不可行，需要聚类预筛选
- **纯文本链接**：双向链接只有文本引用，缺少链接强度（强相关/弱相关）、链接类型（继承/依赖/对比）
- **去重仅靠 LLM**：LLM 评估冲突的成本随文档数线性增长，百万级文档不现实

### 工业界最佳实践

- **实体链接自动化**：用 NER + 实体消歧模型自动识别文档中的实体并建立双向链接
- **链接权重衰减**：时间越久的链接权重越低，反映知识的时效性
- **冲突优先级**：核心概念冲突优先解决（影响面大），边缘概念冲突可暂存"争议区"

## 常见问题

### 小白最常踩的 3 个坑

**坑 1：把所有文档都加双向链接，导致"链接爆炸"**

每个文档链接 20 个其他文档，1000 个文档就是 20000 个链接。应该只链接直接相关的文档（一个文档通常 3-8 个链接足够），通过图的传递性自然关联。

**坑 2：去重删得太激进，丢失有价值的视角差异**

两篇文档讲同一个概念但角度不同（一篇是技术视角，一篇是产品视角）不应去重。去重只针对"内容高度重复"的文档，不同视角的保留并建立互链。

**坑 3：冲突解决全自动，人的判断被排除在外**

LLM 可能错误地判定一篇不准确的文档为"更优"，淘汰了真正正确的文档。高置信度冲突可自动解决，低置信度冲突必须保留人工裁决入口。

### 自检题

**Q1**：双向链接和传统超链接有什么区别？<details><summary>答案</summary>传统超链接是单向的（A→B），双向链接能看到"A链接到哪些"和"谁链接到A"。反向链接让知识图谱可遍历，支持"这个文档被谁引用"的发现。</details>

**Q2**：冲突解决的四种策略是什么？<details><summary>答案</summary>合并（互补）、选择（择优）、标记冲突（不确定）、人工裁决。LLM先评估，能自动解决的自动处理，不确定的交给人工。</details>

**Q3**：去重和冲突解决有什么区别？<details><summary>答案</summary>去重针对内容高度重复的文档（如两篇文档 90% 相同），直接合并或选一。冲突解决针对内容矛盾的文档（如两篇文档对同一概念给出了相反的结论），需要评估哪篇更准确。</details>
