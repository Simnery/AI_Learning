# 5.7.3 增量更新、持久化存储

> 5.7.3 增量更新与持久化 > 5.7 LLM Wiki > 5. 项目实战

## 核心概念

- **是什么**：增量更新是指只重新编译发生变化的部分而非全量重建——当 Raw 层新增数据时，只更新受影响的 Wiki 页面。Git 提供天然的版本控制，每次编译结果都可追溯。
- **为什么重要**：全量编译成本太高（特别是大知识库），增量更新让 Wiki 在"实时性"和"成本"之间取得平衡。

## 原理讲解

### 1. 基于Git的版本控制

```
每次编译 → Git commit
  ↓
- 谁改了什么（git blame）
- 什么时候改的（git log）
- 为什么改（commit message）
- 回滚到任意历史版本
```

### 2. 从RAG到"编译器模式"

| 维度 | RAG模式 | 编译器模式 |
|------|---------|-----------|
| 处理方式 | 检索时临时处理 | 编译时预处理 |
| 质量保证 | 依赖检索质量（不稳定） | 编译后验证（确定） |
| 更新成本 | 低（加文档即可） | 中（需重新编译相关页） |
| 一致性 | 可能检索到矛盾信息 | 编译时消除矛盾 |

### 3. 增量更新触发策略

```
触发条件:
- Raw层新增 > 10 条相关数据 → 重新编译Wiki页
- 手动触发 → 立即编译
- 定时编译（每6小时） → 批量更新
```


### 3. 增量更新与持久化

```
watch 文件/git hook → diff 片段
   ↓
仅对变更 chunk re-embed（5-7-4）
   ↓
更新 manifest.json（版本、checksum）
```

**回滚**：manifest 指向前一版本向量快照；Wiki Git revert 即可。

### 4. 存储选型

| 层 | 存储 |
|----|------|
| Wiki | Git + Markdown |
| 向量 | LanceDB / FAISS 文件 |
| Raw | 对象存储 |

## 代码实战

```python
# pip install — 本示例仅用 Python 3.10+ 标准库

"""增量更新 + Git版本控制"""
import subprocess, hashlib

class IncrementalWiki:
    def __init__(self, wiki: ThreeTierWiki, repo_path: str = "./wiki"):
        self.wiki = wiki
        self.repo = repo_path
        self.raw_hashes = {}  # {raw_id: hash} 检测变化
    
    def detect_changes(self) -> list[str]:
        """检测Raw层变化，返回需要重新编译的主题"""
        changed_topics = set()
        
        for i, raw in enumerate(self.wiki.raw):
            new_hash = hashlib.md5(raw["text"].encode()).hexdigest()
            if self.raw_hashes.get(i) != new_hash:
                # 提取主题关键词
                topics = self._extract_topics(raw["text"])
                changed_topics.update(topics)
                self.raw_hashes[i] = new_hash
        
        return list(changed_topics)
    
    def _extract_topics(self, text: str) -> list[str]:
        """从文本提取主题——简化版"""
        keywords = ["支付", "部署", "性能", "安全", "接口", "数据库"]
        return [kw for kw in keywords if kw in text]
    
    def incremental_compile(self):
        """增量编译——只更新变化的部分"""
        changed = self.detect_changes()
        print(f"检测到 {len(changed)} 个主题需要更新: {changed}")
        
        for topic in changed:
            doc = self.wiki.compile_wiki(topic)
            if doc:
                self._git_commit(topic, f"增量更新: {topic}")
    
    def _git_commit(self, topic: str, message: str):
        """Git 版本控制"""
        filepath = Path(self.repo) / f"{topic}.md"
        filepath.write_text(self.wiki.wiki.get(topic, {}).get("content", ""))
        
        subprocess.run(["git", "-C", self.repo, "add", str(filepath)])
        subprocess.run(["git", "-C", self.repo, "commit", "-m", message])
```

## 进阶方向

### 当前方案的局限性

- **MD5 检测碰撞**：用 MD5 检测内容变化，但 MD5 有碰撞风险（虽然极低），用 SHA256 更安全
- **主题提取过于简单**：用关键词表提取主题，新兴话题完全无法识别，需要 LLM 做主题提取
- **Git 频率过高**：每次增量更新就 commit，一天可能产生 100+ 个微小 commit，应该批量合并
- **无编译缓存**：同一主题连续两次增量更新，重复编译浪费 LLM 调用

### 工业界最佳实践

- **批量提交窗口**：每小时的增量更新合并为一个 commit，附汇总信息
- **增量编译 diff**：编译前后做 diff，只有 diff > 阈值（如 10% 内容变化）才更新 Wiki，避免微小变化频繁更新
- **Git LFS for Raw**：Raw 层大文件（如会议录音文本）存 Git LFS，不污染主仓库

## 常见问题

### 小白最常踩的 3 个坑

**坑 1：git commit 失败时静默跳过**

subprocess.run 执行 git commit，如果工作目录不是 git repo 或者有冲突，进程静默失败——Wiki 更新了但版本记录丢了。必须检查 returncode，失败时告警。

**坑 2：增量更新遗漏间接影响**

改了 Raw 层"支付接口超时问题"的解决方案，只触发"支付"主题的重新编译。但"性能优化"主题也引用了这个方案，它也应该被更新。需要利用反向链接追踪间接影响。

**坑 3：全量更新不设时间窗口**

全量更新可能耗时数小时，期间 Wiki 查询返回混合态——有的文档已更新，有的还没。应该用蓝绿部署：新版本编译完成后一次性切换。

### 自检题

**Q1**：增量更新 vs 全量更新，何时用哪种？<details><summary>答案</summary>增量更新用于日常追加（Raw层新增少量数据时只更新受影响的Wiki页），全量更新用于重大结构调整或定期全量重建。增量省成本，全量保一致性。</details>

**Q2**：Git在LLM Wiki中的作用是什么？<details><summary>答案</summary>版本追溯（谁改了、什么时候、为什么）、回滚能力、变更对比（git diff看LLM改了什么）、协作（多人修改不冲突）。</details>

**Q3**：增量更新的最大风险是什么？<details><summary>答案</summary>更新了直接相关的文档但遗漏了间接相关的文档，导致知识库出现"部分更新"的不一致状态。需要利用双向链接的"反向链接"追踪所有受影响的文档。</details>
