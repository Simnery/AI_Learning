# 5.7.4 知识库索引体系

> 5.7.4 知识库索引体系 > 5.7 LLM Wiki > 5. 项目实战

## 核心概念

- **是什么**：LLM Wiki 采用"轻量化索引"策略——不依赖重量级向量检索，而是用关键词检索 + 目录路由的组合方式。弱化向量是因为 Wiki 文档本身已经是 LLM 编译过的高质量文本，关键词匹配就能找到。
- **为什么重要**：向量检索有成本（embedding 计算+存储+检索延迟），对已经结构化很好的 Wiki 来说是过度设计。轻量索引更简单、更快、更可解释。

## 原理讲解

### 轻量化检索架构

```
用户查询 → 关键词提取 → 目录路由 → 精确匹配 Wiki 页
                ↓            ↓
          倒排索引      目录树（层级导航）
```

### 为什么弱化向量

| 场景 | 向量检索 | 关键词+目录 |
|------|---------|------------|
| Wiki 标题精确匹配 | ❌ 过度 | ✅ 直接命中 |
| 同义词搜索 | ✅ 好 | ❌ 可能漏 |
| 复杂语义查询 | ✅ 好 | ❌ 困难 |
| 可解释性 | ❌ "黑盒相似度" | ✅ "匹配了关键词X" |
| 维护成本 | ❌ 需维护向量库 | ✅ 纯文本，零维护 |

> LLM Wiki 中大部分查询是精确匹配（"XX问题怎么解决"），关键词+目录就行。少数语义查询退化到 Raw 层原始文本搜索。


### 3. 混合索引体系

```
查询 → BM25(标题/标签) ──┐
      → 向量(正文 chunk) ─┼→ RRF 融合 → Top-N Wiki 条目
```

**字段权重**：标题 3×、标签 2×、正文 1×。

### 4. 索引维护 cron

每日：新增/变更文档增量索引；每周：全量 rebuild 校验漂移。

## 代码实战

```python
# pip install — 本示例仅用 Python 3.10+ 标准库

"""轻量化索引——关键词检索+目录路由"""
from collections import defaultdict

class LightweightIndex:
    def __init__(self):
        self.inverted_index = defaultdict(set)  # {keyword: {doc_ids}}
        self.directory_tree = {}  # {category: [doc_ids]}
    
    def build_index(self, wiki: dict):
        """构建倒排索引"""
        for doc_id, doc in wiki.items():
            # 分词 + 建倒排
            words = set(doc["title"].lower().split() + 
                       doc["content"][:200].lower().split())
            for word in words:
                if len(word) >= 2:  # 过滤单字
                    self.inverted_index[word].add(doc_id)
    
    def build_directory(self, wiki: dict):
        """构建目录树"""
        for doc_id, doc in wiki.items():
            # 从文档标题提取分类（例如 "支付-支付宝-超时问题"）
            parts = doc["title"].split("-")
            category = parts[0] if parts else "未分类"
            if category not in self.directory_tree:
                self.directory_tree[category] = []
            self.directory_tree[category].append(doc_id)
    
    def search(self, query: str, wiki: dict) -> list[str]:
        """关键词检索 + 目录路由"""
        keywords = [w.lower() for w in query.split() if len(w) >= 2]
        
        # 1. 关键词精确匹配（倒排索引）
        matched_ids = set()
        for kw in keywords:
            matched_ids.update(self.inverted_index.get(kw, set()))
        
        # 2. 目录路由（查询是否匹配某个分类）
        for category, doc_ids in self.directory_tree.items():
            if category.lower() in query.lower():
                matched_ids.update(doc_ids)
        
        # 3. 排序——匹配关键词多的排前面
        scored = []
        for doc_id in matched_ids:
            content = wiki[doc_id]["content"].lower()
            score = sum(1 for kw in keywords if kw in content)
            scored.append((score, doc_id))
        
        scored.sort(reverse=True)
        return [wiki[s[1]]["title"] for s in scored[:5]]

# 使用
index = LightweightIndex()
index.build_index(wiki_manager.documents)
index.build_directory(wiki_manager.documents)
results = index.search("支付接口超时", wiki_manager.documents)
```

## 进阶方向

### 当前方案的局限性

- **分词粗糙**：用空格分割中文文本完全无效，需要 jieba 等中文分词工具
- **倒排索引内存膨胀**：文档量超过 10 万时倒排索引占用内存 GB 级，需要磁盘索引（如 Lucene）
- **无模糊匹配**：用户拼错关键词（"支负"→"支付"）完全搜不到，需要编辑距离或拼音搜索
- **目录树扁平**：只有一层分类，真实场景需要多层级目录和分类标签

### 工业界最佳实践

- **分层索引**：标题→倒排索引（快速匹配），正文→向量索引（语义匹配），元数据→SQL过滤
- **热词缓存**：高频查询结果缓存到内存，避免重复检索
- **索引质量监控**：统计"查询无结果率"，过高说明索引覆盖不足，触发索引重建

## 常见问题

### 小白最常踩的 3 个坑

**坑 1：中文不分词直接建索引**

把"支付接口超时问题"当作一个 token，用户搜"接口"搜不到。中文必须分词：["支付", "接口", "超时", "问题"]，每个词建独立索引。

**坑 2：倒排索引不设大小限制，内存撑爆**

每个文档产生 50-100 个 token，10 万文档就是 500 万+ 个倒排条目，Python 的 defaultdict(set) 可能占用 2-3GB 内存。应该设 max_index_entries 并按 LFU 淘汰低频词。

**坑 3：目录树和倒排索引独立建，两者不一致**

新文档加了索引但忘了更新目录树，或者反之。应该在 add_document 时同时更新索引和目录，保证原子性。

### 自检题

**Q1**：为什么LLM Wiki选择弱化向量检索？<details><summary>答案</summary>Wiki文档已被LLM编译为高质量结构文本，关键词精确匹配已能覆盖大部分查询。向量检索增加embedding成本、存储成本和检索延迟，且可解释性差。少数语义查询退化到Raw层搜索即可。</details>

**Q2**：轻量化索引的两大组件是什么？<details><summary>答案</summary>倒排索引（关键词→文档ID映射，精确匹配）+ 目录树（分类→文档列表，层级导航）。两者结合覆盖"搜索"和"浏览"两种使用模式。</details>

**Q3**：中文分词为什么是轻量索引的必备环节？<details><summary>答案</summary>中文没有空格作为天然分隔符，不分词会把整句话当做一个 token 建索引，用户搜任何子词都搜不到。必须用 jieba 等工具分词后才能建有效的倒排索引。</details>
