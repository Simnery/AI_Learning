> 5.6.2 换脸任务及算力分配 > 5.6 短剧视频换脸 > 5. 项目实战

## 核心概念

- **是什么**：算力分配是多 GPU 换脸系统的调度核心——如何将海量换脸任务高效分配给多张 GPU、控制队列长度、处理超时重试、缓存原始视频以节省流量。
- **为什么重要**：单 GPU 处理一部短剧可能需要数小时，多 GPU 调度不好可能导致资源浪费或任务饿死。合理的调度策略直接影响系统的吞吐量和成本。

## 原理讲解

### 1. 算力调度架构

```
用户提交任务 → 任务队列 → 调度器 → GPU池
                    ↓          ↓
              优先级+deadline   负载均衡
                                   ↓
                    GPU0  GPU1  GPU2  GPU3
```

### 2. 关键策略

| 策略 | 描述 | 参数 |
|------|------|------|
| 队列长度控制 | 防止任务堆积导致内存爆炸 | max_queue=100 |
| 超时重试 | 单帧处理超过阈值自动重试 | timeout=30s, max_retry=3 |
| 任务解耦 | 片段/全剧/预处理分离队列 | 独立优先级 |
| 视频缓存 | 热门素材缓存到本地SSD | LRU淘汰 |


### 3. GPU 队列与算力分配

```
任务队列(优先级, 帧数, 模型) → Scheduler
     → GPU-0: 推理卡 A100
     → GPU-1: 推理卡 A100
     → CPU 池: 检测/预处理
```

**策略**：短任务优先（SJF）防饿死；大任务可拆 batch 跨卡。

### 4. 容量估算

`总帧数 × 单帧推理 ms / GPU 数` ≈  wall time；队列深度 >5 时提示用户预计等待。

## 代码实战

```python
# pip install — 本示例仅用 Python 3.10+ 标准库

"""GPU算力分配与任务调度"""
import asyncio
from collections import deque

class GPUScheduler:
    def __init__(self, gpu_ids: list[int], max_queue: int = 100):
        self.gpus = {gid: {"busy": False, "queue": deque(maxlen=max_queue)} 
                     for gid in gpu_ids}
        self.video_cache = {}  # LRU 缓存
    
    def select_gpu(self) -> int:
        """选择最空闲的GPU（最少排队任务）"""
        return min(self.gpus, key=lambda g: len(self.gpus[g]["queue"]))
    
    async def submit(self, task: FaceSwapTask, priority: int = 0):
        gpu_id = self.select_gpu()
        self.gpus[gpu_id]["queue"].append((priority, task))
        task.status = TaskStatus.QUEUED
    
    async def process_loop(self, gpu_id: int):
        """GPU工作循环"""
        while True:
            if not self.gpus[gpu_id]["queue"]:
                await asyncio.sleep(1)
                continue
            
            self.gpus[gpu_id]["busy"] = True
            _, task = self.gpus[gpu_id]["queue"].popleft()
            
            try:
                # 检查缓存
                if task.video_path in self.video_cache:
                    task.video_path = self.video_cache[task.video_path]
                
                await asyncio.wait_for(
                    pipeline.process(task), timeout=task.params.get("timeout", 300)
                )
            except asyncio.TimeoutError:
                if task.params.get("retries", 0) < 3:
                    task.params["retries"] = task.params.get("retries", 0) + 1
                    self.gpus[gpu_id]["queue"].appendleft((1, task))  # 重新排队
                else:
                    task.status = TaskStatus.FAILED
                    task.error_msg = "超时重试3次后失败"
            finally:
                self.gpus[gpu_id]["busy"] = False
    
    def cache_video(self, video_path: str):
        """LRU缓存——最多缓存5个视频"""
        if len(self.video_cache) >= 5:
            oldest = next(iter(self.video_cache))
            del self.video_cache[oldest]
        self.video_cache[video_path] = video_path
```

## 进阶方向

### 当前方案的局限性

- **最少队列 ≠ 最优**：只看队列长度忽略了显存使用率和 GPU 温度，可能把任务发给"队列短但显存已满"的 GPU
- **固定超时**：短剧（10 分钟）和电影（120 分钟）用同一个超时阈值不合理
- **无优先级分级**：所有用户的任务同优先级，付费用户没有优先处理
- **视频缓存命中率低**：LRU 缓存只有 5 个槽位，10 个用户处理 10 部不同短剧时命中率为 0

## 常见问题

### 小白最常踩的 3 个坑

**坑 1：GPU 满了还在往队列塞任务**

不看 GPU 显存就 submit 任务——队列排到 50 个任务但 GPU 显存只够跑一个。结果是所有任务都 OOM。应该在 submit 前检查 GPU 可用显存。

**坑 2：process_loop 死循环 CPU 空转**

队列空时 `asyncio.sleep(1)` 每秒循环一次，单个 loop 还好，4 个 GPU 的 loop 就是每秒 4 次无意义轮询。应该用事件驱动——队列有任务时 notify loop 唤醒。

**坑 3：重试不加延迟直接重排**

任务失败后立即重新排队，可能瞬间又失败、又排、又失败——短时间内产生大量无效重试。重试之间应该加指数退避延迟（1s, 4s, 9s）。

### 自检题

**Q1**：如何选择最合适的GPU来分配任务？<details><summary>答案</summary>选择当前排队任务最少的GPU（最小负载优先）。可扩展为考虑GPU显存使用率、温度等指标。</details>

**Q2**：超时重试机制如何设计？<details><summary>答案</summary>单帧/单任务设置超时阈值（如30s），超时后重新排队最多3次，3次后标记失败并通知用户。重试任务优先级提高。</details>

**Q3**：为什么选择 GPU 不能只看队列长度？<details><summary>答案</summary>队列短不等于 GPU 空闲——可能队列只有 1 个任务但显存已被占满。需要综合评估显存使用率、GPU 利用率、温度等指标，做加权评分选择最优 GPU。</details>
