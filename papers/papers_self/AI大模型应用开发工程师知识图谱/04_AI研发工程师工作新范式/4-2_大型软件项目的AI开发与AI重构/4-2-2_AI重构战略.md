# 4.2.2 AI重构战略

> 4.2.2 AI重构战略 > 4.2 大型软件项目AI开发 > 4. AI研发工程师工作新范式

## 核心概念

- **是什么**：AI 重构战略是用 AI 系统性改造老旧/混乱代码库的方法论。"暴力拆解"是让 AI 快速分析和理解复杂代码，"精准缝合"是逐步替换旧代码为新实现，"绞杀模式"是让新旧代码并存、逐步迁移、最终替换。
- **为什么重要**：传统重构像拆炸弹——改一处可能炸一片，测试覆盖不够就不敢动。AI 重构让你有了"透视镜"（快速理解）和"安全网"（自动生成测试），把重构从高风险赌博变成可控的工程过程。
- **前置知识**：有大型项目维护经验，理解思维跃迁（参考 [4-2-1 思维跃迁](4-2-1_从代码实现者到系统架构师的思维跃迁.md)），了解审查闭环（参考 [4-1-4 审查闭环](../4-1_AI_Coding带来的范式变革/4-1-4_构建审查-优化闭环，把风险关在上线前.md)）。

### 重构三角

```
        理解
        /  \
       / AI \
      / 辅助  \
     /________\
  安全          执行
  (测试)       (AI生成)
```

AI 在三个顶点上都有独特优势：理解代码（暴力拆解）、生成新实现（精准缝合）、生成测试（安全保障）。

## 原理讲解

### 1. "暴力拆解"——让 AI 吃掉复杂代码

传统方式理解一个 2000 行的旧模块需要几天。AI 可以几分钟给出结构分析。

**暴力拆解三步**：

```
第 1 步: 结构拆解
  喂入整个模块 → AI 输出: 类/函数清单 + 调用关系图 + 数据流

第 2 步: 逻辑拆解  
  喂入关键函数 → AI 输出: 每个函数的伪代码 + 副作用 + 依赖

第 3 步: 风险拆解
  喂入 git 历史 → AI 输出: "经常出 bug 的区域" + "修改影响面"
```

### 2. "精准缝合"——用 AI 生成替换代码

拆解后知道了"旧代码在做什么"，下一步是用新代码替换它。关键是**接口不变**：

```
旧接口 ──→ 新实现
  ↓            ↓
输入A       输入A
  ↓            ↓
旧逻辑      新逻辑（更清晰/高效/可测试）
  ↓            ↓
输出B       输出B  ← 必须完全一致！
```

**缝合检查清单**：
- 输入类型是否完全一致？
- 输出类型是否完全一致？
- 副作用（数据库写入、文件操作、缓存更新）是否一致？
- 异常类型和错误信息是否兼容？

### 3. 渐进式现代化——绞杀模式

一次性重写整个系统 = 自杀。绞杀模式（Strangler Fig Pattern）让你逐步替换：

```
    旧系统
  ┌─────────┐
  │ 模块A   │ ← 第1周: AI 重写 → 新实现，代理切换
  │ 模块B   │ ← 第2周: AI 重写 → 新实现，代理切换
  │ 模块C   │ ← 第3周: 发现A有遗漏，AI补测试
  │ 模块D   │ ← 第4周: AI 重写 → 新实现
  │  ...    │
  └─────────┘
       ↓
  旧系统逐渐萎缩，新系统逐渐壮大
       ↓
  最终: 旧系统只剩空壳 → 删除
```

**AI 如何加速绞杀**：
1. **自动生成代理层**：AI 写路由代码，根据配置决定走新旧哪个实现
2. **自动对比验证**：同时调用新旧实现，对比输出，发现差异立即报警
3. **自动生成迁移测试**：为每个被替换的模块生成回归测试

### 4. 重构决策矩阵

| 代码状态 | 测试覆盖 | AI 重构策略 |
|---------|---------|------------|
| 清晰 + 有测试 | ✅ 好 | 直接让 AI 重构 + 现有测试兜底 |
| 清晰 + 无测试 | ⚠️ 中 | AI 先生成测试 → 再重构 |
| 混乱 + 有测试 | ⚠️ 中 | AI 暴力拆解理解 → 精准缝合 |
| 混乱 + 无测试 | 🔴 差 | AI 暴力拆解 → 补测试 → 小步重构 |

```python
"""
暴力拆解工具: 喂入混乱代码，AI 输出结构分析
"""

def analyze_legacy_module(file_path: str, llm_client) -> str:
    """AI 分析遗留模块的结构和风险"""
    
    code = Path(file_path).read_text()
    
    prompt = f"""你是遗留代码分析专家。分析以下代码，输出结构化报告。

代码文件: {file_path}
代码行数: {len(code.splitlines())}

{code[:12000]}  # 截断到 ~12KB

请输出:
### 1. 模块概述
这个模块做什么？（一句话）

### 2. 函数/类清单
| 名称 | 类型 | 行数 | 复杂度(高/中/低) | 副作用 |
|------|------|------|------------------|--------|

用文本画出关键调用链

### 4. 数据流
输入是什么 → 经过什么处理 → 输出是什么

### 5. 风险点
- 哪些函数最容易出 bug
- 哪些函数修改影响面最大
- 哪些地方最需要测试

### 6. 重构建议
按优先级排列，每个建议标注难度和工作量"""
    
    return llm_client.complete(prompt)
```

### 示例2：渐进式绞杀——代理层实现

```python
"""
绞杀模式实现: 新旧实现并存，逐步切换
"""

import functools
import logging

logger = logging.getLogger(__name__)

class StranglerProxy:
    """绞杀代理: 新旧实现对比验证，平滑切换"""
    
    def __init__(self, old_impl, new_impl, use_new: bool = False):
        self.old = old_impl
        self.new = new_impl
        self.use_new = use_new
    
    def _dual_run(self, func_name: str, *args, **kwargs):
        """同时运行新旧实现，对比结果"""
        old_result = getattr(self.old, func_name)(*args, **kwargs)
        
        try:
            new_result = getattr(self.new, func_name)(*args, **kwargs)
            
            if old_result != new_result:
                logger.warning(
                    f"差异检测! {func_name}: "
                    f"旧={old_result[:100]}, 新={new_result[:100]}"
                )
        except Exception as e:
            logger.error(f"新实现异常! {func_name}: {e}")
        
        # 当前仍返回旧实现结果（确保稳定性）
        return old_result
    
    def __getattr__(self, name: str):
        """代理所有方法调用"""
        old_attr = getattr(self.old, name)
        new_attr = getattr(self.new, name, None)
        
        if not callable(old_attr) or new_attr is None:
            return old_attr
        
        @functools.wraps(old_attr)
        def wrapper(*args, **kwargs):
            if self.use_new:
                return new_attr(*args, **kwargs)
            else:
                # 双跑模式：用旧结果，但后台验证新结果
                return self._dual_run(name, *args, **kwargs)
        
        return wrapper


# 使用示例
# old_payment = OldPaymentService()
# new_payment = NewPaymentService()
# 
# # 阶段1: 双跑验证（旧结果返回给用户，后台对比新旧）
# proxy = StranglerProxy(old_payment, new_payment, use_new=False)
# proxy.process_payment(order)
# 
# # 阶段2: 验证通过后，切换到新实现
# proxy.use_new = True
# proxy.process_payment(order)
# 
# # 阶段3: 确认稳定后，直接使用新实现，删除代理
# # payment = NewPaymentService()
```

### 示例3：AI 生成重构测试

```python
"""
AI 辅助生成重构安全网: 为旧代码自动生成回归测试
"""

def generate_refactor_tests(old_module_path: str, llm_client) -> str:
    """AI 为遗留代码生成回归测试"""
    
    old_code = Path(old_module_path).read_text()
    
    prompt = f"""为以下 Python 模块的每个公共函数生成 pytest 测试。

模块代码:
{old_code[:10000]}

要求:
1. 每个公共函数至少 3 个测试: 正常输入、边界条件、异常输入
2. 如果函数有副作用（数据库/文件/网络），用 unittest.mock  mock 掉
3. 测试文件命名为 test_{Path(old_module_path).stem}.py
4. 包含 conftest.py 级别的 fixture（如果需要）

输出完整可运行的测试代码。"""

    return llm_client.complete(prompt)
```

### 示例4：重构检查清单脚本

```python
"""
重构前/后检查: 确保行为不变
"""

class RefactorValidator:
    """重构验证器: 检查重构前后的接口兼容性"""
    
    def check_interface_compatibility(self, old_module, new_module) -> dict:
        """检查新旧模块的公共接口是否兼容"""
        import inspect
        
        old_api = self._extract_public_api(old_module)
        new_api = self._extract_public_api(new_module)
        
        report = {
            "removed": [],    # 旧有但新没有的
            "added": [],      # 旧没有但新有的
            "changed": [],    # 签名变更的
            "unchanged": [],  # 完全一致的
        }
        
        for name in old_api:
            if name not in new_api:
                report["removed"].append(f"❌ {name} 被删除!")
            elif old_api[name] != new_api[name]:
                report["changed"].append(
                    f"⚠️  {name}: {old_api[name]} → {new_api[name]}"
                )
            else:
                report["unchanged"].append(f"✅ {name}")
        
        for name in new_api:
            if name not in old_api:
                report["added"].append(f"➕ {name} 新增")
        
        return report
    
    def _extract_public_api(self, module) -> dict:
        """提取模块的公共接口"""
        import inspect
        api = {}
        for name, obj in inspect.getmembers(module):
            if name.startswith('_'):
                continue
            if inspect.isfunction(obj) or inspect.isclass(obj):
                try:
                    sig = str(inspect.signature(obj))
                except (ValueError, TypeError):
                    sig = "(无法获取签名)"
                api[name] = sig
        return api
```

## 常见问题

### 小白最常踩的坑

1. **想一次重写整个项目**：重构冲动 + AI 能力 = 想全部推倒重来。结果：项目停摆 3 个月，旧代码没人管，新代码没完成。正确做法：绞杀模式，一次一个模块
2. **跳过暴力拆解直接缝合**：没理解旧代码就开始写新代码。结果：遗漏关键逻辑。正确做法：先拆解 100%，再缝合
3. **没有对比验证**：重构后只跑单元测试，不对比新旧实际输出。正确做法：双跑模式（旧服务 + 新服务同时运行，对比结果）

### 自检题

**Q1**："暴力拆解"的三步分别是什么？每步输出什么？

<details><summary>答案</summary>

1. 结构拆解：输出类/函数清单 + 调用关系图 + 数据流
2. 逻辑拆解：输出关键函数的伪代码 + 副作用 + 依赖
3. 风险拆解：输出高 bug 率区域 + 修改影响面分析
</details>

**Q2**：绞杀模式（Strangler Fig）的核心思想是什么？AI 如何加速它？

<details><summary>答案</summary>

核心：新旧系统并存，逐步替换。新功能走新系统，旧功能逐一从旧系统迁移，最终旧系统萎缩消失。AI 加速方式：自动生成代理层（路由配置）、自动对比验证（双跑对比输出）、自动生成迁移测试。
</details>

**Q3**：重构前后必须验证的三个"一致性"是什么？

<details><summary>答案</summary>

1. 接口一致性：输入/输出类型和格式完全一致
2. 副作用一致性：数据库写入、文件操作、缓存更新、日志等行为一致
3. 异常一致性：相同错误输入产生相同类型的异常和相同格式的错误信息
</details>
