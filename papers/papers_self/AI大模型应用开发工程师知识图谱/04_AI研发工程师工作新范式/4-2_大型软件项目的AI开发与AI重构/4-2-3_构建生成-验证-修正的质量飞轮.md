# 4.2.3 构建"生成-验证-修正"的质量飞轮

> 4.2.3 构建生成-验证-修正的质量飞轮 > 4.2 大型软件项目AI开发 > 4. AI研发工程师工作新范式

## 核心概念

- **是什么**：质量飞轮是一个持续运转的 AI 开发循环——AI 生成代码 → 自动验证发现问题 → AI 修正问题 → 再次验证 → ... → 直到质量达标。与审查闭环（4-1-4）关注"上线前的门禁"不同，质量飞轮关注"开发过程中的持续质量提升"。
- **为什么重要**：大型项目的代码质量不是一次审查能保证的。单次 AI 生成的代码可能有 80 分，经过飞轮的 3-5 轮迭代可以达到 95 分。飞轮让"够用"变成"优秀"。
- **前置知识**：理解审查闭环（参考 [4-1-4 审查闭环](../4-1_AI_Coding带来的范式变革/4-1-4_构建审查-优化闭环，把风险关在上线前.md)），了解 AI 重构（参考 [4-2-2 AI重构战略](4-2-2_AI重构战略.md)）。

### 飞轮 vs 闭环

| 维度 | 审查闭环 (4-1-4) | 质量飞轮 (4-2-3) |
|------|-----------------|-----------------|
| 时机 | 提交前/上线前 | 开发过程中 |
| 粒度 | PR 级别 | 函数/模块级别 |
| 迭代 | 1-3 轮 | 3-10 轮 |
| 目标 | 阻断坏代码上线 | 把好代码变成优秀代码 |
| 比喻 | 门禁安检 | 健身房训练 |

## 原理讲解

### 1. 飞轮的三个核心轮齿

```
         ┌──────────────────┐
         │    AI 生成代码    │
         │  (Generation)    │
         └────────┬─────────┘
                  │
       ┌──────────▼──────────┐
       │    自动验证          │
       │  (Verification)     │
       │  测试/审查/分析     │
       └──────────┬──────────┘
                  │
          ┌───────▼───────┐
          │ 通过?          │
          │ 质量 >= 目标?  │
          └───┬───────┬───┘
              │NO     │YES
              ▼       ▼
    ┌─────────┐   ┌──────────┐
    │AI 修正   │   │ 人工终审  │
    │(Fix)    │   │ 合并代码  │
    └────┬────┘   └──────────┘
         │
         └──→ 回到 "AI 生成代码"
```

### 2. 每一轮迭代做什么

| 轮次 | 主要任务 | 质量目标 | 示例 |
|------|---------|---------|------|
| 第 1 轮 | 功能实现 | 能跑通 | 基础逻辑正确 |
| 第 2 轮 | 边界处理 | 不崩溃 | 处理 null/空/超大输入 |
| 第 3 轮 | 错误处理 | 鲁棒 | 合理的异常 + 错误信息 |
| 第 4 轮 | 性能优化 | 高效 | 减少不必要的查询/循环 |
| 第 5 轮 | 代码美化 | 可维护 | 命名/注释/结构优化 |

> 不是每段代码都需要 5 轮——核心业务逻辑多迭代，工具脚本 1 轮就够了。

### 3. 验证的三个维度

```
┌──────────────────────────────────────┐
│            验证三维度                 │
├────────────┬────────────┬────────────┤
│ 功能验证   │ 质量验证   │ 兼容验证   │
├────────────┼────────────┼────────────┤
│ 单元测试   │ 静态分析   │ 接口兼容   │
│ 集成测试   │ AI 审查    │ 性能回归   │
│ E2E 测试   │ 覆盖率检查 │ 行为对比   │
└────────────┴────────────┴────────────┘
```

### 4. 从 "超级个体" 到 "超级团队" 的工程文化

质量飞轮能在团队层面运转，需要文化支撑：

| 旧文化（超级个体） | 新文化（超级团队） |
|-------------------|-------------------|
| "我写的代码我负责" | "AI 生成的代码团队共同审查" |
| 代码审查是"挑刺" | 代码审查是"共同成长" |
| 测试是"额外负担" | 测试是"飞轮的燃料" |
| 知识藏在脑子里 | 知识在 Prompt 库和规则里 |
| 速度 > 质量 | 速度 × 质量（飞轮保证） |

**三个关键习惯**：
1. **Show your prompt**：分享代码时附带所用的 prompt，让队友能复现和改进
2. **Blame the prompt, not the person**：AI 代码出问题时，改进 prompt/规则，而不是指责使用者
3. **Celebrate iterations**：认可"经过 5 轮飞轮优化"的努力，而不只是"快速完成"

## 代码实战

### 示例1：质量飞轮引擎

```python
# pip install — 本示例仅用 Python 3.10+ 标准库，无需额外安装

# 标准库示例，无需额外 pip 依赖
"""
QualityFlywheel: 自动化"生成→验证→修正"循环
"""

import subprocess
from typing import Callable

class QualityFlywheel:
    def __init__(self, llm_client, target_quality: float = 0.9):
        self.llm = llm_client
        self.target = target_quality
        self.history = []  # 记录每轮的改进
    
    def generate(self, spec: str, context: str = "") -> str:
        """生成代码"""
        prompt = f"""根据以下需求生成 Python 代码。
{context}
需求: {spec}
输出完整代码，包含 type hints 和 docstring。"""
        return self.llm.complete(prompt)
    
    def verify(self, code: str, tests: str) -> dict:
        """验证代码质量"""
        result = {
            "tests_pass": False,
            "coverage": 0.0,
            "lint_issues": 0,
            "ai_score": 0,
            "issues": []
        }
        
        # 1. 运行测试
        Path("_temp_code.py").write_text(code)
        Path("_temp_test.py").write_text(tests)
        
        r = subprocess.run(
            ["pytest", "_temp_test.py", "--tb=short", "-q"],
            capture_output=True, text=True, timeout=30
        )
        result["tests_pass"] = r.returncode == 0
        if not result["tests_pass"]:
            result["issues"].append(f"测试失败:\n{r.stdout[-500:]}")
        
        # 2. Lint 检查
        r = subprocess.run(
            ["ruff", "check", "_temp_code.py"],
            capture_output=True, text=True
        )
        result["lint_issues"] = len([l for l in r.stdout.split("\n") if l.strip()])
        
        # 3. AI 评审
        ai_review = self._ai_review(code)
        result["ai_score"] = ai_review.get("score", 0)
        result["issues"].extend(ai_review.get("issues", []))
        
        # 4. 计算综合质量分
        test_score = 1.0 if result["tests_pass"] else 0.0
        lint_score = max(0, 1.0 - result["lint_issues"] * 0.1)
        result["quality"] = (test_score * 0.4 + lint_score * 0.2 + result["ai_score"] * 0.4)
        
        return result
    
    def fix(self, code: str, verification: dict, spec: str) -> str:
        """根据验证结果修正代码"""
        issues_text = "\n".join(f"- {i}" for i in verification["issues"])
        
        prompt = f"""修复以下代码的问题。

原始需求: {spec}

当前代码:
{code}

发现的问题:
{issues_text}

当前质量分: {verification['quality']:.2f} (目标: {self.target})

请输出修复后的完整代码。"""
        
        return self.llm.complete(prompt)
    
    def _ai_review(self, code: str) -> dict:
        """AI 评审代码"""
        prompt = f"""评审以下代码，输出 JSON。

代码:
{code}

输出格式:
{{"score": <0-1 质量评分>,
 "issues": ["问题1", "问题2"]}}"""
        
        import json
        try:
            return json.loads(self.llm.complete(prompt))
        except json.JSONDecodeError:
            return {"score": 0.7, "issues": ["AI 评审解析失败"]}
    
    def run(self, spec: str, tests: str, max_rounds: int = 5) -> tuple[str, list]:
        """运行完整质量飞轮"""
        code = self.generate(spec)
        self.history = []
        
        for round_num in range(1, max_rounds + 1):
            print(f"\n{'='*40}")
            print(f"第 {round_num} 轮")
            
            # 验证
            verification = self.verify(code, tests)
            self.history.append({
                "round": round_num,
                "quality": verification["quality"],
                "issues": len(verification["issues"])
            })
            
            print(f"质量分: {verification['quality']:.2f} (目标: {self.target})")
            
            if verification["quality"] >= self.target:
                print(f"✅ 达到目标质量! 共 {round_num} 轮")
                return code, self.history
            
            if round_num < max_rounds:
                print(f"问题数: {len(verification['issues'])}，进入修正...")
                code = self.fix(code, verification, spec)
        
        print(f"⚠️  达到最大轮次 {max_rounds}，当前质量: {verification['quality']:.2f}")
        return code, self.history

# 使用

# )
# print(f"最终代码: {len(code)} 字符, 经过 {len(history)} 轮优化")
```

### 示例2：从超级个体到超级团队的 Prompt Share 工具

```python
"""
PromptShare: 团队 prompt 共享和版本管理
"""

class PromptShare:
    """团队 prompt 资产管理"""
    
    def __init__(self, repo_path: str = ".claude/prompts"):
        self.repo = Path(repo_path)
        self.repo.mkdir(parents=True, exist_ok=True)
    
    def save(self, name: str, prompt: str, author: str, result_quality: int):
        """保存一个有效的 prompt（附带效果评分）"""
        entry = f"""# {name}
> 作者: {author} | 质量评分: {result_quality}/10 | 日期: {datetime.now().strftime('%Y-%m-%d')}

{prompt}
"""
        (self.repo / f"{name}.md").write_text(entry, encoding="utf-8")
        print(f"✅ Prompt '{name}' 已保存 (评分: {result_quality}/10)")
    
    def search(self, keyword: str) -> list[str]:
        """搜索相关 prompt"""
        results = []
        for f in self.repo.glob("*.md"):
            content = f.read_text(encoding="utf-8")
            if keyword.lower() in content.lower():
                results.append(f.stem)
        return results
    
    def get(self, name: str) -> str:
        """获取指定 prompt"""
        filepath = self.repo / f"{name}.md"
        if filepath.exists():
            return filepath.read_text(encoding="utf-8")
        return f"Prompt '{name}' 不存在"


# 团队使用流程
share = PromptShare()

# 张三发现一个好用的 prompt
share.save(
    name="fastapi-crud-endpoint",
    prompt="""生成 FastAPI CRUD 端点:
- 路径: {resource}
- 使用 SQLAlchemy 异步 session
- Pydantic v2 做请求/响应验证
- 每个操作有独立的路由函数
- 错误返回统一格式 {{"detail": "..."}}""",
    author="张三",
    result_quality=9
)

# 李四使用
prompt = share.get("fastapi-crud-endpoint")
# code = llm.complete(prompt.format(resource="users"))
```

## 进阶方向

- 飞轮迭代过多会导致 token 成本线性增长（权衡迭代次数和质量目标）
- 验证维度的权重需要根据项目调优（核心库重可靠性，营销页面重体验）
- 超级团队文化需要自上而下的推动，纯自底向上难以落地

## 常见问题

### 小白最常踩的坑

1. **追求完美第一轮**：给 AI 写 500 字 prompt 期望一次完美。正确做法：先让 AI 生成一个能跑的版本，再用飞轮迭代优化
2. **飞轮跑太多轮**：核心逻辑飞了 10 轮还在微调变量命名。正确做法：设定明确的质量目标和最大轮次
3. **不记录飞轮过程**：只知道最终代码好，不知道为什么好。正确做法：记录每轮发现的问题和改进，沉淀到规则库

### 自检题

**Q1**：质量飞轮和审查闭环（4-1-4）的核心区别是什么？

<details><summary>答案</summary>

- 时序：飞轮在开发过程中持续运转，闭环在提交/上线前一次性检查
- 粒度：飞轮在函数/模块级迭代，闭环在 PR 级别
- 目标：飞轮把代码从 80 分提升到 95 分，闭环阻断 60 分以下的代码
</details>

**Q2**：质量飞轮迭代 5 轮的典型分工是什么？

<details><summary>答案</summary>

第 1 轮功能实现（能跑通）→ 第 2 轮边界处理（不崩溃）→ 第 3 轮错误处理（鲁棒）→ 第 4 轮性能优化（高效）→ 第 5 轮代码美化（可维护）
</details>

**Q3**：从"超级个体"到"超级团队"需要哪三个关键习惯？

<details><summary>答案</summary>

1. Show your prompt：分享代码时附带 prompt，让队友能复现和改进
2. Blame the prompt, not the person：AI 代码出问题改进 prompt/规则，不指责人
3. Celebrate iterations：认可多轮优化的努力，不只是快速完成
</details>
