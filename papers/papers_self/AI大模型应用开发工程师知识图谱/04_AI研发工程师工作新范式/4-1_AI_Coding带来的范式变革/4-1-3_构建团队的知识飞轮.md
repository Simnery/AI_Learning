# 4.1.3 构建团队的"知识飞轮"

> 4.1.3 构建团队的知识飞轮 > 4.1 AI Coding带来的范式变革 > 4. AI研发工程师工作新范式

## 核心概念

- **是什么**：知识飞轮是让团队 AI 协作经验**正向循环**的机制——每一次和 AI 的交互成果（好的 prompt、规则、代码模板）都沉淀下来，让下一次交互更好，就像飞轮越转越快。
- **为什么重要**：没有知识飞轮，每个团队成员都在重复踩同样的坑、写相似的 prompt、犯相同的错误。有了它，团队经验从"散落在聊天记录里"变成"可搜索、可复用、可进化的资产"。
- **前置知识**：有 AI Coding 实践经验，理解 Spec Coding（参考 [4-1-1 用确定性驾驭概率性](4-1-1_用确定性驾驭概率性.md)），了解流水线概念（参考 [4-1-2 构建AI流水线](4-1-2_构建你的AI流水线.md)）。

### 知识飞轮三环

```
         ┌──────────────┐
        ╱     积累       ╲
       /   记录每次AI交互  \
      │    的有效产出      │
      │                   │
      │  ┌───────────┐   │
      │  │  知识飞轮  │   │
      │  └───────────┘   │
      │                   │
       \    复用         /
        ╲  注入到下次   ╱
         \  AI交互中   /
          └──────────┘
              ↓
           进化
         （持续优化规则和模板）
```

## 原理讲解

### 1. 飞轮运转的三个阶段

**阶段一：积累（Capture）**

每次和 AI 交互后，提取可复用的部分：

| 积累物 | 示例 | 存储位置 |
|--------|------|---------|
| 成功的 Spec | "写 Python API 时加上 pydantic 验证" | Spec 库 |
| 高质量 Prompt | 项目特定的 prompt 模板 | .claude/prompts/ |
| 编码规则 | "本项目禁止使用 mutable 默认参数" | .cursorrules / CLAUDE.md |
| AI 生成的测试 | 验证通过的测试用例 | 测试套件 |

**阶段二：复用（Reuse）**

把积累物注入到后续的 AI 交互中：

```
新任务 → 从知识库检索相关 Spec/Prompt/规则 → 注入当前上下文 → AI 交互 → 产出
```

**阶段三：进化（Evolve）**

定期审查并优化积累物：

```
规则库 review（每两周）
  ├── 规则是否还有效？
  ├── 有没有更好的写法？
  ├── 合并重复，删除废弃
  └── 更新版本号
```

### 2. 解决"代码考古"难题

"代码考古"指的是接手一段没有文档、没有注释、原作者已离职的代码时，需要像考古学家一样慢慢挖掘理解。

**传统方式**（痛苦）：
```
读代码 → 猜逻辑 → 看 git log → 问同事（可能已离职） → 局部理解 → 再读 → ...
需要数天到数周
```

**知识飞轮方式**（高效）：
```
把代码 + git 历史 + Issue 讨论 → 喂给 AI → AI 生成代码地图
↓
相当于有了一个"知道所有历史和上下文"的资深同事
需要数小时
```

### 3. 知识飞轮的具体产出物

```
project-root/
├── .claude/
│   ├── CLAUDE.md              # 项目总体规则和约定（飞轮的核心载体）
│   ├── prompts/
│   │   ├── code-review.md     # 代码审查专用 prompt
│   │   ├── new-feature.md     # 新功能开发 prompt
│   │   └── refactor.md        # 重构 prompt 模板
│   └── rules/
│       ├── api-design.md      # API 设计规范
│       ├── error-handling.md  # 错误处理约定
│       └── testing.md         # 测试规范
├── docs/
│   └── architecture.md        # AI 生成的架构文档（持续更新）
└── specs/
    ├── feature-001.md         # 已完成的 Spec 存档
    └── feature-002.md
```

### 4. 上下文注入策略

不是把所有规则一股脑塞给 AI（反而干扰），而是按需注入：

```python
def select_context(task_type: str, knowledge_base: dict) -> str:
    """根据任务类型，智能选择要注入的上下文"""
    context_map = {
        "new_feature": ["spec_template", "api_rules", "test_rules"],
        "bug_fix": ["error_handling_rules", "related_code", "git_history"],
        "refactor": ["code_style_rules", "architecture_doc", "test_suite"],
        "code_review": ["security_rules", "naming_convention", "performance_rules"],
    }
    selected = context_map.get(task_type, [])
    return "\n".join(knowledge_base[k] for k in selected if k in knowledge_base)
```

### 示例1：构建团队的 Prompt 模板库

```python
# pip install — 本示例仅用 Python 3.10+ 标准库，无需额外安装

# 标准库示例，无需额外 pip 依赖
"""
团队 Prompt Template 管理器
每个成员提交好用的 prompt → 审核入库 → 全团队复用
"""

import json
from datetime import datetime
from pathlib import Path

class PromptLibrary:
    def __init__(self, library_path: str = "./prompts"):
        self.path = Path(library_path)
        self.path.mkdir(exist_ok=True)
    
    def add_prompt(self, name: str, template: str, 
                   author: str, tags: list[str], variables: list[str]):
        """添加一个新的 prompt 模板"""
        entry = {
            "name": name,
            "template": template,
            "author": author,
            "tags": tags,
            "variables": variables,  # 模板中的 {变量}
            "usage_count": 0,
            "success_rate": 0.0,
            "created_at": datetime.now().isoformat(),
            "examples": []
        }
        
        filepath = self.path / f"{name}.json"
        filepath.write_text(json.dumps(entry, ensure_ascii=False, indent=2))
        return entry
    
    def find_prompts(self, task: str, tags: list[str] = None) -> list[dict]:
        """根据任务描述查找相关 prompt"""
        results = []
        for file in self.path.glob("*.json"):
            entry = json.loads(file.read_text())
            # 按标签或关键词匹配
            if tags and any(t in entry["tags"] for t in tags):
                results.append(entry)
            elif any(kw in entry["name"] for kw in task.split()):
                results.append(entry)
        # 按使用次数排序
        return sorted(results, key=lambda x: x["usage_count"], reverse=True)
    
    def fill_template(self, name: str, **kwargs) -> str:
        """填充模板变量"""
        entry = json.loads((self.path / f"{name}.json").read_text())
        template = entry["template"]
        try:
            result = template.format(**kwargs)
            # 记录使用
            entry["usage_count"] += 1
            (self.path / f"{name}.json").write_text(
                json.dumps(entry, ensure_ascii=False, indent=2)
            )
            return result
        except KeyError as e:
            return f"模板变量缺失: {e}。需要: {entry['variables']}"


# 使用示例
lib = PromptLibrary()

# 团队成员 A 添加一个好用的 prompt
lib.add_prompt(
    name="python-api-endpoint",
    template="""实现一个 FastAPI 端点:
- 路径: {path}
- 方法: {method}
- 请求体: {request_body}
- 响应: {response}
- 使用 Pydantic 做请求/响应验证
- 错误时返回 {error_format}
- 数据库操作通过 {db_service} 完成""",
    author="张三",
    tags=["python", "api", "fastapi"],
    variables=["path", "method", "request_body", "response", "error_format", "db_service"]
)

# 团队成员 B 使用
filled = lib.fill_template(
    "python-api-endpoint",
    path="/users",
    method="POST",
    request_body="{\"email\": str, \"name\": str}",
    response="{\"id\": int, \"email\": str, \"name\": str}",
    error_format="{\"error\": str}",
    db_service="UserService"
)
```

### 示例2：知识驱动的 AI 交互——上下文自动注入

```python
"""
SmartContextBuilder: 根据任务自动选择和注入相关知识
"""

class SmartContextBuilder:
    def __init__(self, rules_dir: str = ".claude/rules"):
        self.rules_dir = Path(rules_dir)
        self.rules_cache = {}
        self._load_rules()
    
    def _load_rules(self):
        """加载所有规则文件"""
        if self.rules_dir.exists():
            for f in self.rules_dir.glob("*.md"):
                self.rules_cache[f.stem] = f.read_text(encoding="utf-8")
    
    def build_context(self, task: str, task_type: str = "auto") -> str:
        """构建 AI 交互的上下文"""
        context_parts = []
        
        # 1. 项目基本规则（始终包含）
        if "project" in self.rules_cache:
            context_parts.append("## 项目规则\n" + self.rules_cache["project"])
        
        # 2. 根据任务类型注入专项规则
        type_rules = {
            "api": ["api-design", "error-handling"],
            "database": ["database", "orm"],
            "frontend": ["component", "styling"],
            "refactor": ["code-style", "testing"],
            "bug_fix": ["debugging", "error-handling"],
        }
        
        for rule_key in type_rules.get(task_type, []):
            if rule_key in self.rules_cache:
                context_parts.append(
                    f"## {rule_key}规范\n" + self.rules_cache[rule_key]
                )
        
        # 3. 注入相关 Spec（从完成的 Spec 库检索）
        relevant_specs = self._find_relevant_specs(task)
        if relevant_specs:
            context_parts.append("## 相关 Spec 参考\n" + "\n---\n".join(relevant_specs))
        
        return "\n\n".join(context_parts)
    
    def _find_relevant_specs(self, task: str, max_results: int = 2) -> list[str]:
        """从 Spec 库检索相关文档（简化版关键词匹配）"""
        specs_dir = Path("specs")
        if not specs_dir.exists():
            return []
        
        results = []
        keywords = set(task.lower().split())
        for f in specs_dir.glob("*.md"):
            content = f.read_text(encoding="utf-8")
            # 简单关键词匹配（生产环境用 embedding + 向量检索）
            score = sum(1 for kw in keywords if kw in content.lower())
            if score > 0:
                results.append((score, content[:500]))  # 截取前500字
        
        results.sort(key=lambda x: x[0], reverse=True)
        return [r[1] for r in results[:max_results]]
```

### 示例3：代码考古——让 AI 帮助理解遗留代码

```python
"""
Code Archaeologist: 利用 AI + git 历史生成代码地图
"""

import subprocess

def generate_code_map(repo_path: str, target_file: str):
    """为一团迷雾的遗留代码生成理解地图"""
    
    # 1. 读取文件内容
    content = Path(repo_path, target_file).read_text()
    
    # 2. 获取 git 历史（谁改的、什么时候、为什么）
    log = subprocess.run(
        ["git", "-C", repo_path, "log", "--oneline", "--follow", "-20", target_file],
        capture_output=True, text=True
    ).stdout
    
    # 3. 获取相关的 commit diff
    diffs = subprocess.run(
        ["git", "-C", repo_path, "log", "-p", "--follow", "-5", target_file],
        capture_output=True, text=True
    ).stdout[:8000]  # 截断
    
    # 4. 喂给 AI，生成代码地图
    prompt = f"""你是一位资深代码考古学家。这是一段遗留代码，原作者已离职。

### 代码文件: {target_file}
{content}

### Git 历史（最近的修改）
{log}

### 最近的 Diff
{diffs}

请生成一份"代码地图"，包含：
1. **这个文件是干什么的**（一句话）
2. **核心数据流**：数据从哪里来，经过什么处理，到哪里去
3. **关键函数/类**：每个的作用和依赖关系
4. **已知坑点**：从 git history 看，哪些地方经常出 bug
5. **修改指南**：如果要改 XXX 功能，应该从哪里入手"""
    
    # return llm.complete(prompt)
    pass
```

## 常见问题

### 小白最常踩的坑

1. **规则写成散文**：CLAUDE.md 里写了几百字的故事。正确做法：用列表、代码示例、硬约束，AI 更容易理解
2. **一次写太多规则**：上来写 50 条规则 → AI 被淹没 → 反而忽略关键规则。正确做法：从 5 条开始，逐周增加
3. **只积累不清理**：半年前的规则还留着，反而误导。必须定期 review 和删除过时规则

### 自检题

**Q1**：知识飞轮的三个阶段是什么？每个阶段做什么？

<details><summary>答案</summary>

1. **积累（Capture）**：记录每次 AI 交互的有效产出（Spec、Prompt、规则、测试）
2. **复用（Reuse）**：从知识库检索相关内容，注入到新的 AI 交互中
3. **进化（Evolve）**：定期审查、优化、删除过时规则，保持知识库的有效性
</details>

**Q2**：什么是"代码考古"？知识飞轮如何解决它？

<details><summary>答案</summary>

"代码考古"指接手没有文档/注释、原作者离职的代码时，需要大量时间理解。知识飞轮通过：把代码 + git 历史 + Issue 讨论 + 设计文档喂给 AI，让 AI 生成代码地图（文件作用、数据流、已知坑点），数小时完成原本数天的理解工作。
</details>

**Q3**：为什么做上下文注入时不能把所有规则都塞给 AI？

<details><summary>答案</summary>

- Token 浪费（成本高）
- 信息过载（AI 被大量无关规则干扰，输出质量下降）
- 规则冲突（不同场景的规则可能互相矛盾）

应该根据任务类型按需注入（如新建 API 只注入 API 设计规则，不注入前端样式规则）。
</details>
