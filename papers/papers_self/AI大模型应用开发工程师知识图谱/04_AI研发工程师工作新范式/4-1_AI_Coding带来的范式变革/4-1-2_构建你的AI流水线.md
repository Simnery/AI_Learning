# 4.1.2 构建你的AI流水线

> 4.1.2 构建你的AI流水线 > 4.1 AI Coding带来的范式变革 > 4. AI研发工程师工作新范式

## 核心概念

- **是什么**：AI 流水线是一套自动化的软件开发流程——把代码编写、测试、审查、部署等环节串联起来，每个环节由 AI Agent 负责或辅助，人只做关键决策和最终把关。
- **为什么重要**：单个 AI 对话窗口只能做点状任务。流水线把多个 AI 能力编排成持续运转的生产线，从"AI 辅助写一段代码"升级到"AI 驱动整个开发流程"。
- **前置知识**：理解 Spec Coding（参考 [4-1-1 用确定性驾驭概率性](4-1-1_用确定性驾驭概率性.md)），了解 Agent 和多智能体概念（参考 [2-2-2 Agent自主规划与工具开发](../../02_AI应用技术/2-2_Agent理论知识+案例详解+实操/2-2-2_Agent的自主规划与工具开发.md)）。

### AI 流水线的核心价值

| 维度 | 人肉开发 | AI 流水线 |
|------|---------|----------|
| 代码生成 | 人写 | AI 生成 + 人审核 |
| 测试 | 人写测试 | AI 生成测试 + 自动运行 |
| 审查 | 人逐行看 | AI 先扫 → 人看关键决策 |
| 文档 | 写完补 | AI 同步生成 |
| 部署 | 手动操作 | CI/CD 自动触发 |

## 原理讲解

### 1. 流水线架构

```
┌─────────────────────────────────────────────────────────┐
│                     AI 开发流水线                        │
│                                                         │
│  [需求] → [Spec Agent] → [Code Agent] → [Test Agent]    │
│                ↓               ↓              ↓         │
│           结构化需求       生成代码       自动测试       │
│                                                         │
│  → [Review Agent] → [Doc Agent] → [Deploy Agent]        │
│         ↓               ↓               ↓               │
│    代码审查         文档生成         自动部署            │
│                                                         │
│  全程: Human-in-the-loop（关键节点人工确认）              │
└─────────────────────────────────────────────────────────┘
```

### 2. 多智能体协作模式

| 模式 | 原理 | 适用场景 |
|------|------|---------|
| **串行流水线** | A 的输出是 B 的输入 | 需求→开发→测试，流程固定的场景 |
| **并行分工** | 多个 Agent 同时干不同模块 | 前端/后端/测试同时开发 |
| **辩论模式** | 多个 Agent 对同一问题给出方案，人选择 | 架构设计、技术选型 |
| **Critic 模式** | 一个 Agent 写，另一个专门挑刺 | 代码审查、安全检查 |

### 3. 自循环——流水线自动修复

流水线的关键特性是 **自动回环**：当一个环节失败时，自动回到上游修复。

```
Code Agent 生成 → Test Agent 运行 → ❌ 失败
                                      ↓
                         自动回传错误信息给 Code Agent
                                      ↓
                         Code Agent 修复 → 重新测试 → ✅ 通过
```

### 4. 上下文传递——流水线的血液

每个 Agent 需要的上下文不同，流水线要负责裁剪和传递：

```
完整上下文 (所有文件+需求+历史)
    ↓ 裁剪
Spec Agent 需要: 需求文档 + 数据模型 + API 规范
Code Agent 需要: Spec + 相关代码文件 + 项目配置
Test Agent 需要: 实现代码 + 测试框架 + 边界条件
Review Agent 需要: Diff + 编码规范 + 安全规则
```

## 代码实战

### 示例1：搭建最小 AI 流水线（Python 脚本编排）

```python
"""
一个最简单的 AI 开发流水线:
需求 → 生成代码 → 运行测试 → 通过则输出，失败则自动修复
"""

import subprocess
import sys

class AIPipeline:
    def __init__(self, llm_client):
        self.llm = llm_client  # 可以是 OpenAI/Claude/本地模型
    
    def step1_generate_code(self, spec: str) -> str:
        """步骤1: 根据 Spec 生成代码"""
        prompt = f"""根据以下需求生成 Python 代码。
只输出代码，不要解释，不要 markdown 代码块标记。

需求: {spec}

要求:
- 包含 type hints
- 处理常见边界条件
- 函数名和变量名见名知意"""
        
        code = self.llm.complete(prompt)
        return code.strip().removeprefix("```python").removesuffix("```")
    
    def step2_run_tests(self, code: str, test_file: str) -> tuple[bool, str]:
        """步骤2: 运行测试"""
        # 写入代码文件
        with open("generated.py", "w") as f:
            f.write(code)
        
        # 运行 pytest
        result = subprocess.run(
            ["pytest", test_file, "-v", "--tb=short"],
            capture_output=True, text=True, timeout=30
        )
        
        return result.returncode == 0, result.stdout + result.stderr
    
    def step3_auto_fix(self, code: str, test_output: str, spec: str) -> str:
        """步骤3: 自动修复失败的代码"""
        prompt = f"""以下代码的测试失败了。请修复代码。

原始需求: {spec}

当前代码:
{code}

测试失败信息:
{test_output}

请输出修复后的完整代码，只输出代码。"""
        
        return self.llm.complete(prompt).strip()
    
    def run(self, spec: str, test_file: str, max_retries: int = 3) -> str:
        """运行完整流水线"""
        print(f"[Pipeline] 开始处理: {spec[:50]}...")
        
        code = self.step1_generate_code(spec)
        print(f"[Pipeline] 代码已生成 ({len(code)} 字符)")
        
        for attempt in range(max_retries):
            passed, output = self.step2_run_tests(code, test_file)
            
            if passed:
                print(f"[Pipeline] ✅ 测试通过 (尝试 {attempt + 1} 次)")
                return code
            
            print(f"[Pipeline] ❌ 测试失败 (尝试 {attempt + 1})，自动修复中...")
            code = self.step3_auto_fix(code, output, spec)
        
        print(f"[Pipeline] ❌ 达到最大重试次数 {max_retries}")
        return code


# 使用示例

# )

```

### 示例2：多 Agent 协作——Code + Review 双 Agent

```python
"""
双 Agent 协作: Code Agent 写代码 → Review Agent 审查 → Code Agent 修复
"""

class DualAgentPipeline:
    def __init__(self, llm):
        self.llm = llm
    
    def code_agent(self, spec: str, context: str = "") -> str:
        """Code Agent: 负责生成代码"""
        prompt = f"""你是一个资深 Python 工程师。根据需求生成代码。

需求: {spec}
{context}

要求:
- 代码简洁清晰，注释只在必要时写
- 包含完整的错误处理
- 符合 PEP 8 规范"""
        return self.llm.complete(prompt)
    
    def review_agent(self, code: str, spec: str) -> dict:
        """Review Agent: 负责审查代码"""
        prompt = f"""审查以下代码，按照标准输出 JSON。

代码:
{code}

原始需求: {spec}

输出格式:
{{
    "score": <1-10 整体评分>,
    "issues": [
        {{"severity": "critical|warning|suggestion", "line": <行号>, "description": "问题描述"}}
    ],
    "security_concerns": ["安全问题1", "安全问题2"],
    "approved": true/false
}}

只输出 JSON，不要其他内容。"""
        
        import json
        result = self.llm.complete(prompt)
        return json.loads(result)
    
    def run(self, spec: str, min_score: int = 7, max_iterations: int = 3) -> str:
        iteration = 0
        code = ""
        
        while iteration < max_iterations:
            iteration += 1
            print(f"--- 第 {iteration} 轮 ---")
            
            # Code Agent 生成
            if iteration == 1:
                code = self.code_agent(spec)
            else:
                context = f"\n上一轮审查意见:\n{feedback}"
                code = self.code_agent(spec, context)
            
            print(f"[Code Agent] 生成了 {len(code)} 字符代码")
            
            # Review Agent 审查
            review = self.review_agent(code, spec)
            print(f"[Review Agent] 评分: {review['score']}/10")
            
            if review.get("issues"):
                for issue in review["issues"]:
                    print(f"  [{issue['severity']}] {issue['description']}")
            
            # 判断是否通过
            if review.get("approved") and review["score"] >= min_score:
                print(f"✅ 审查通过! (评分 {review['score']})")
                return code
            
            feedback = "\n".join(
                f"- [{i['severity']}] {i['description']}"
                for i in review.get("issues", [])
            )
            print(f"❌ 审查未通过，进入修复轮次\n")
        
        return code
```

### 示例3：自动化 CI/CD AI 流水线（GitHub Actions 风格）

```yaml
# .github/workflows/ai-pipeline.yml
# AI 驱动的 CI/CD 流水线

name: AI-Powered Pipeline

on:
  pull_request:
    types: [opened, synchronize]

jobs:
  ai-review:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      
      # Step 1: AI 代码审查
      - name: AI Code Review
        run: |
          python scripts/ai_review.py \
            --pr-number ${{ github.event.pull_request.number }} \
            --output review.md
      
      # Step 2: AI 生成测试
      - name: AI Generate Tests
        run: |
          python scripts/ai_generate_tests.py \
            --changed-files "${{ steps.changed-files.outputs.all_changed_files }}"
      
      # Step 3: 运行所有测试
      - name: Run Tests
        run: pytest --cov=. --cov-report=xml
      
      # Step 4: AI 分析测试覆盖率
      - name: AI Coverage Analysis
        run: |
          python scripts/ai_coverage_check.py \
            --coverage-xml coverage.xml \
            --threshold 80
      
      # Step 5: 发布 AI 审查报告到 PR
      - name: Post Review Report
        uses: actions/github-script@v7
        with:
          script: |
            const fs = require('fs');
            const review = fs.readFileSync('review.md', 'utf8');
            github.rest.issues.createComment({
              owner: context.repo.owner,
              repo: context.repo.repo,
              issue_number: context.issue.number,
              body: review
            });
```

## 进阶方向

### 1. 当前局限性

- 流水线的 token 消耗大（每个环节都要传上下文），成本是单次对话的 3-10 倍
- Agent 间的上下文传递会累积误差——上游的微小偏差到下游可能被放大
- 复杂流水线的调试困难——出问题后难以定位是哪个 Agent 的责任

### 3. 落地建议

- **先跑通一个最小流水线**：需求→代码→测试三步，证明可行性后再加环节
- **从低成本 Agent 开始**：用 GPT-3.5/Claude Haiku 做低风险环节（代码格式化、测试生成），用强模型做核心环节
- **关键环节保留人工确认**：部署到生产、修改数据库 schema、安全相关变更

## 常见问题

### 小白最常踩的坑

1. **把所有环节都给一个 Agent**：一个 Agent 的上下文窗口有限，角色混杂导致每个环节都不精。正确的做法是每个环节用独立 Agent + 专门 prompt
2. **自动修复死循环**：AI 反复生成错误代码 → 反复修复 → 反复失败。必须设置 `max_retries` 上限，达到后人工介入
3. **忽略上下文裁剪**：把全部项目文件传给每个 Agent。Token 浪费且干扰判断。务必做上下文裁剪

### 自检题

**Q1**：AI 流水线的"自动回环"是什么意思？为什么重要？

<details><summary>答案</summary>

自动回环是当流水线某个环节失败时，自动把失败信息反馈给上游环节进行修复。比如测试失败 → 错误信息回传给 Code Agent → 修复代码 → 重新测试。它让流水线具备了自愈能力，不需要人手动介入每次失败。
</details>

**Q2**：多智能体协作的四种模式分别是什么？

<details><summary>答案</summary>

1. **串行流水线**：A→B→C，流程固定，上游输出是下游输入
2. **并行分工**：多个 Agent 同时工作在不同模块（如前端+后端+测试）
3. **辩论模式**：多个 Agent 给出不同方案，人选最佳
4. **Critic 模式**：一个写，另一个专门挑刺（生成+审查双Agent）
</details>

**Q3**：为什么流水线中要做上下文裁剪？不裁剪会怎样？

<details><summary>答案</summary>

每个 Agent 角色不同，需要的信息也不同。不裁剪导致：Token 浪费（成本高）、信息过载（Agent 被无关信息干扰，输出质量下降）、可能泄露敏感信息给不该看到的 Agent。
</details>
