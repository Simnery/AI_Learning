# 4.1.1 用确定性驾驭概率性

> 4.1.1 用确定性驾驭概率性 > 4.1 AI Coding带来的范式变革 > 4. AI研发工程师工作新范式

## 核心概念

- **是什么**：AI 编程的核心矛盾——LLM 本质是概率系统（同样的 prompt 可能生成不同的代码），但软件工程要求确定性（同样输入必须同样输出）。"用确定性驾驭概率性"就是你如何设计约束和流程，让概率性的 AI 产出确定性的结果。
- **为什么重要**：不理解这一点，AI 编程就是"抽卡"——有时惊艳，有时灾难。掌握了驾驭方法，AI 就从一个随机生成器变成了可控的生产力工具。
- **前置知识**：有实际编程经验（至少能独立读懂代码），用过至少一款 AI 编程工具（Copilot/Cursor/Claude Code）。

### 核心公式

```
AI 编程质量 = 需求明确度 × 上下文精准度 × 验证严格度
```

三个因子中任何一个为 0，最终质量都是 0。这就是为什么"随便说一句需求"的产出远不如"写好 Spec + 给足上下文 + 严格测试"。

## 原理讲解

### 1. "Spec Coding" 三铁律

三铁律是用确定性约束 AI 的核心手段，每次和 AI 交互编程时必须遵守：

```
铁律 1: 精准定义边界
┌─────────────────────────────┐
│ ❌ "写一个登录功能"           │
│ ✅ "写一个登录 API，POST /login│
│    接收 {email, password}，   │
│    返回 {token, user}，       │
│    密码用 bcrypt 验证，        │
│    token 用 JWT，有效期 7 天， │
│    email 格式用正则验证，      │
│    错误时返回 401{error: msg}" │
└─────────────────────────────┘

铁律 2: 分步验证
┌─────────────────────────────┐
│ 不让 AI 一次写完整个功能后检查  │
│ 而是：写接口 → 验证 → 写逻辑   │
│ → 验证 → 写测试 → 验证 → ...  │
└─────────────────────────────┘

铁律 3: 测试先行
┌─────────────────────────────┐
│ 先让 AI 写测试用例，再写实现   │
│ 测试 = 可执行的确定性约束     │
│ 实现 = AI 的概率性输出       │
└─────────────────────────────┘
```

### 2. 渐进式复杂度管理

一次性让 AI 理解整个复杂系统是不现实的。把大象装进冰箱分三步：

```
第 1 步: 建立心像（Context Building）
  让 AI 理解系统的整体结构和当前文件在整个系统中的位置
  方法: 提供架构图、目录结构、相关文件内容

第 2 步: 小步快跑（Incremental Delivery）
  每次只实现一个最小可验证单元
  方法: 拆解任务到 < 50 行代码的粒度，每步验证

第 3 步: 层层加固（Layer Hardening）
  每完成一层，立即写测试、加错误处理、补充文档
  方法: 功能代码 → 测试代码 → 边界条件 → 文档注释
```

### 3. Prompt 的确定性控制技巧

| 技巧 | 示例 | 原理 |
|------|------|------|
| 约束输出格式 | "只输出 JSON，不要额外解释" | 限制 AI 的自由度 |
| 给定模板/骨架 | "用以下类结构填充实现..." | 提供确定性的框架 |
| 负面示例驱动 | "不要用全局变量，不要用 eval" | 排除已知的错误模式 |
| 温度控制 | temperature=0（推理）/0.7（生成） | 直接控制随机性 |
| 多轮精炼 | "第一步: 写骨架 → 第二步: 填充逻辑 → ..." | 逐步消除歧义 |

### 4. 验证金字塔——确定性保障的最后一道防线

```
           ┌───────┐
           │ E2E   │  端到端测试：模拟真实用户操作
           │ 测试  │
         ┌─┴───────┴─┐
         │  集成测试   │  模块间协同是否正确
         │            │
       ┌─┴────────────┴─┐
       │   单元测试       │  每个函数是否正确
       │                 │
     ┌─┴─────────────────┴─┐
     │  静态分析 (Lint/Type) │  语法和类型错误
     │                      │
   ┌─┴──────────────────────┴─┐
   │  AI 生成代码 → 人工 Review │  逻辑审查
   └──────────────────────────┘
```

AI 生成的代码必须经过每层验证，越底层的问题越容易被 AI 自动修复。

### 示例1：差的 Spec vs 好的 Spec

```python
# pip install — 本示例仅用 Python 3.10+ 标准库，无需额外安装

# 标准库示例，无需额外 pip 依赖
"""
【差的 Spec——AI 容易跑偏】
"写一个函数处理用户数据"

AI 可能的产出: 各种奇怪实现，不知道你想干什么
"""

"""
【好的 Spec——AI 一次输出到位】
写一个 Python 函数 `validate_and_format_user(email: str, age: int) -> dict`：

要求：
1. email 必须包含 @，且 @ 后至少有 1 个 .，否则抛出 ValueError("Invalid email")
2. age 必须是 0-150 之间的整数，否则抛出 ValueError("Invalid age")
3. 返回 {"email": email.lower().strip(), "age": age, "is_adult": age >= 18}
4. 函数体内不要加 try/except，让调用方处理异常
5. 包含 type hints 和 docstring

不要输出测试代码，只输出这个函数。
"""

# AI 按好 Spec 生成的代码：
def validate_and_format_user(email: str, age: int) -> dict:
    """验证并格式化用户信息。
    
    Args:
        email: 用户邮箱地址
        age: 用户年龄（整数）
    
    Returns:
        dict: 包含格式化后的 email、age 和 is_adult 字段
    
    Raises:
        ValueError: 当 email 格式无效或 age 超出范围时
    """
    if "@" not in email:
        raise ValueError("Invalid email")
    
    local, domain = email.rsplit("@", 1)
    if "." not in domain:
        raise ValueError("Invalid email")
    
    if not isinstance(age, int) or age < 0 or age > 150:
        raise ValueError("Invalid age")
    
    return {
        "email": email.lower().strip(),
        "age": age,
        "is_adult": age >= 18
    }
```

### 示例2：分步验证的工作流

```python
"""
实战演示：和 AI 协作开发一个"Markdown 标题提取器"

步骤1: 先写测试（铁律3：测试先行）
"""
# test_md_extractor.py —— 先写好，作为确定性约束
import pytest

def test_extracts_h1():
    result = extract_headers("# 标题")
    assert result == [{"level": 1, "text": "标题"}]

def test_extracts_multiple_levels():
    md = """# H1
### H2
### H3"""
    result = extract_headers(md)
    assert len(result) == 3
    assert result[0] == {"level": 1, "text": "H1"}
    assert result[2] == {"level": 3, "text": "H3"}

def test_ignores_non_headers():
    md = """普通文本
# 标题
又是普通文本"""
    result = extract_headers(md)
    assert len(result) == 1

def test_empty_input():
    assert extract_headers("") == []
    assert extract_headers("没有标题的文本") == []

"""
步骤2: 给 AI 看测试 + Spec，生成实现
Spec: "根据以下测试用例，实现 extract_headers(md: str) -> list[dict]，
      用正则提取所有 # 开头的行，返回 [{'level': N, 'text': '...'}]
      level 是 # 的个数"
"""

# AI 生成的实现（经过验证）：
import re

def extract_headers(md: str) -> list[dict]:
    """从 Markdown 文本中提取所有标题。"""
    headers = []
    for line in md.split("\n"):
        match = re.match(r"^(#{1,6})\s+(.+)", line)
        if match:
            headers.append({
                "level": len(match.group(1)),
                "text": match.group(2).strip()
            })
    return headers

"""
步骤3: 运行测试验证（铁律2：分步验证）
$ pytest test_md_extractor.py -v
→ 全部通过！
"""
```

### 示例3：渐进式构建——从骨架到完整功能

```python
"""
场景：和 AI 协作开发一个简单的 CLI 待办事项工具

第 1 步: 骨架（建立心像）
"用 Python 写一个 TodoCLI 类的骨架，包含 add/list/done 三个空方法，
 数据存在内存 list 中，每个 todo 是 dict{'task': str, 'done': bool}"
"""

class TodoCLI:
    def __init__(self):
        self.todos: list[dict] = []
    
    def add(self, task: str) -> str:
        """添加待办事项"""
        pass
    
    def list(self) -> str:
        """列出所有待办"""
        pass
    
    def done(self, index: int) -> str:
        """标记完成"""
        pass

"""
第 2 步: 逐个实现（小步快跑）
"实现 TodoCLI.add: 添加 task 到 self.todos，
 返回 f'已添加: {task}'，task 为空时返回'任务不能为空'"
"""

# ... AI 实现 add()，验证通过

"""
第 3 步: 层层加固
"为 TodoCLI 添加：1) 输入验证 2) done 的 index 越界检查 3) list 的空列表提示"
"""

# 最终产物（层层加固后）：
class TodoCLI:
    def __init__(self):
        self.todos: list[dict] = []
    
    def add(self, task: str) -> str:
        task = task.strip()
        if not task:
            return "错误: 任务不能为空"
        if len(task) > 200:
            return "错误: 任务不能超过200字"
        self.todos.append({"task": task, "done": False})
        return f"已添加 [#{len(self.todos)}]: {task}"
    
    def list(self) -> str:
        if not self.todos:
            return "暂无待办事项"
        lines = []
        for i, t in enumerate(self.todos, 1):
            status = "✓" if t["done"] else "○"
            lines.append(f"  [{status}] {i}. {t['task']}")
        return "待办列表:\n" + "\n".join(lines)
    
    def done(self, index: int) -> str:
        if index < 1 or index > len(self.todos):
            return "错误: 序号无效"
        self.todos[index - 1]["done"] = True
        return f"已完成: {self.todos[index - 1]['task']}"
```

## 进阶方向

### 1. 当前局限性

- Spec 写得太死可能限制 AI 的创造性（某些场景需要 AI 自由发挥）
- 小步快跑适用于确定性功能，探索性编码（如实验新算法）需要不同的策略
- 过度验证会增加流程负担——根据任务重要性选择合适的验证层级

## 常见问题

### 小白最常踩的坑

1. **一次性要求太多**：写 500 行的需求文档让 AI 一次实现。正确做法：拆成 10 个 50 行的子任务逐步完成
2. **不测试就信任**：AI 代码跑过一遍就当合格。正确做法：至少覆盖正常/边界/异常三种情况
3. **模糊的需求描述**："写个好用的函数"。正确做法：参照示例1的"好的 Spec"格式

### 自检题

**Q1**：Spec Coding 三铁律是什么？每条的作用是什么？

<details><summary>答案</summary>

1. **精准定义边界**：用具体的输入/输出/约束消除需求歧义，让 AI 一次输出到位
2. **分步验证**：每完成一小步立即验证，问题早发现早修正，避免错误累积
3. **测试先行**：测试用例 = 可执行的确定性约束，用确定性约束驾驭概率性代码生成
</details>

**Q2**：为什么"温度"（temperature）可以直接控制 AI 输出的确定性？推理和生成分别用什么温度？

<details><summary>答案</summary>

Temperature 控制 token 采样的随机性。temperature=0 时模型总是选概率最高的 token（确定性输出），temperature 越高分布越平均（更多样但更不可控）。推理/代码生成用 temperature=0-0.3，创意写作用 0.7-1.0。
</details>

**Q3**：渐进式复杂度管理的三步是什么？每步的目的？

<details><summary>答案</summary>

1. **建立心像**：让 AI 理解系统全貌和当前任务在全局中的位置
2. **小步快跑**：每次只实现最小可验证单元，降低出错概率和修复成本
3. **层层加固**：每完成一层立即加测试/错误处理/文档，保证每一步的产出都是可用的
</details>
