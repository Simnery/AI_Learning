## 核心概念

### 什么是 RAG 调优

**一句话**：RAG 搭出来只是开始——从"能跑"到"好用"，需要对检索、召回、生成每个环节做系统优化。

```
能跑(60分): 用户问 → 向量检索 → 拼接 → LLM 回答
好用(90分): 用户问 → 查询改写 → 混合检索 → 重排序 → 置信度评估 → LLM 回答 + 引用标注
```

**类比**：搭 RAG 像是在图书馆里给一个聪明的朋友配了索引卡——调优就是不断优化"索引卡怎么组织"和"朋友怎么使用这些卡"的过程。

### 为什么需要调优

| 问题 | 表现 | 根因 |
|------|------|------|
| 检索偏差 | 搜"价格"返回一堆架构文档 | 纯向量检索被语义误导 |
| 精确匹配失败 | 搜"API-V3"找不到 | 向量不擅长精确字符串匹配 |
| 低质量文档污染 | 回答一半是废话 | 检索到不相关的碎片 |
| 答案不完整 | 漏掉关键信息 | 只搜到一个文档的片段 |

## 原理讲解

### 混合检索：向量 + 关键词，取长处补短处

**单一方式的致命缺陷**：

```
问题: "请查阅 API-V3-2024 的错误码定义"

🎯 精确匹配需求
  BM25: "API-V3-2024" → ✅ 精确命中 → 高分
  向量: "API-V3-2024" → 语义相近... "API 版本 文档" → ❌ 找不到

🌐 语义理解需求
  向量: "用户登录失败怎么排查" → ✅ 搜到"认证错误排查指南"
  BM25: 只有"认证错误排查指南" → ❌ 没有"登录失败"这几个字
```

**结论**：BM25 擅长精确匹配（专有名词、编号、代码），向量检索擅长语义理解。两条路都搜，结果合并——这就是混合检索。

**混合检索的架构**：

```
用户查询 "API-V3 的价格是多少"
  │
  ├─→ BM25 检索(关键词) → [DocA(0.9), DocB(0.7), DocC(0.5)]
  ├─→ 向量检索(语义)  → [DocD(0.85), DocB(0.6), DocE(0.55)]
  │
  ↓ 结果融合(加权求和/倒数排名融合RRF)
  │
  └─→ 融合结果: [DocB(高分), DocA, DocD, DocC, DocE]
       ↓
       Rerank 重排序(BGE-Reranker / Cohere)
       ↓
       最终 Top-3: [DocB, DocA, DocD]
```

**结果融合的两种方法**：

| 方法 | 做法 | 适用场景 |
|------|------|---------|
| **加权求和** | `score = w1*vec_score + w2*bm25_score` | 需要细调权重比例 |
| **RRF**（倒数排名融合） | `score = 1/(k+rank_vec) + 1/(k+rank_bm25)` | 无需调权重，直接就融合 |

**推荐**：RRF 最省心，k 通常取 60。不需要调权重，直接融合排名，效果通常不输手调权重。

### 重排序（Rerank）：在 Top-K 中再做一次"精挑"

向量检索返回的 Top-10 只是"大致相关"，Rerank 用更强的模型重新打分。

```
向量检索 Top-10: [Doc1, Doc2, ..., Doc10]
  → Rerank 模型逐一精细打分
  → 保留 Top-3，丢弃低分 → [Doc3, Doc7, Doc1]
```

**常用 Rerank 模型**：

| 模型 | 特点 | 推荐场景 |
|------|------|---------|
| **BGE-Reranker-v2-m3** | 智源出品，多语言，开源 | 通用中文 RAG，首选 |
| **Cohere Rerank** | 商用 API，高精度 | 追求极致效果，有预算 |
| **BGE-Reranker-v2-minicpm** | 轻量级(0.5B)，速度快 | 延迟敏感场景 |

**Rerank 的价值**：常见场景下仅 Rerank 这一步就能将 RAG 精度提升 10-20%。

### RAG 效果评估：从"感觉"到"数字"

做优化必须量化——你不知道当前几秒几分，就不知道优化是不是在进步。

**RAGAS 评估框架（行业标准）**：

| 指标 | 测什么 | 怎么算 |
|------|--------|--------|
| **Faithfulness（忠实度）** | 回答是否基于检索文档，而非模型编造 | LLM 逐句检查是否有依据 |
| **Answer Relevancy（答案相关性）** | 回答是否切题 | LLM 对答案反向生成问题，看是否接近原问题 |
| **Context Precision（上下文精度）** | 检索到的文档中，相关的排在第几位 | 信号/噪音比，排名越靠前越好 |
| **Context Recall（上下文召回）** | 回答所需的信息，检索到的文档是否全覆盖 | 对比"需要的信息"和"检索到信息"的交集 |

### 持续优化的三层循环

```
外层(月度): 评估指标 → 发现薄弱环节 → 调整策略
中层(周度): A/B 测试 → 对比方案 → 选择最优
内层(实时): Bad Case → 问题定位 → 快速修复

三个 Bad Case 分析模板:
  ┌─ 为什么没有检索到？
  │  → 切分/Embedding/Query改写 哪个环节出的问题
  ├─ 为什么检索到了但没用上？
  │  → Rerank/Prompt设计/上下文窗口
  └─ 为什么用上了但答案错误？
     → LLM 能力/文档矛盾/引用缺失
```

### 实战 1：混合检索（BM25 + 向量）+ RRF 融合

```python
"""混合检索: BM25 + 向量检索，RRF 融合 + Rerank"""
import numpy as np
from rank_bm25 import BM25Okapi
from sentence_transformers import SentenceTransformer, CrossEncoder

# ===== 准备数据 =====
documents = [
    "DeepSeek API-V3 的价格为每百万 token 输入 1 元，输出 2 元。",
    "DeepSeek V3 模型采用 MoE 架构，总参数 671B。",
    "API 调用需要先在 platform.deepseek.com 注册获取 API Key。",
    "DeepSeek R1 支持 JSON Mode 结构化输出。",
    "出现 401 错误码表示 API Key 无效或已过期。",
    "出现 429 错误码表示请求频率超限，需要降低并发。",
    "DeepSeek 支持函数调用(Function Calling)功能。",
    "vLLM 部署 DeepSeek 需要 8 张 A100 GPU。",
]

# ===== BM25 索引 =====
tokenized_docs = [doc.split() for doc in documents]
bm25 = BM25Okapi(tokenized_docs)

# ===== 向量索引 =====
embed_model = SentenceTransformer("BAAI/bge-small-zh-v1.5")
doc_embeddings = embed_model.encode(documents, normalize_embeddings=True)

import faiss
dim = doc_embeddings.shape[1]
index = faiss.IndexFlatIP(dim)
index.add(doc_embeddings.astype('float32'))

# ===== RRF 融合 =====
def rrf_fusion(bm25_results, vector_results, k=60):
    """倒数排名融合: 不需要权重，直接合并两个排名的分数"""
    scores = {}
    for rank, (doc_idx, _) in enumerate(bm25_results):
        scores[doc_idx] = scores.get(doc_idx, 0) + 1.0 / (k + rank + 1)
    for rank, (doc_idx, _) in enumerate(vector_results):
        scores[doc_idx] = scores.get(doc_idx, 0) + 1.0 / (k + rank + 1)
    # 按融合分排序
    return sorted(scores.items(), key=lambda x: x[1], reverse=True)

# ===== 检索 =====
query = "API 调用报错 401 怎么处理"

# BM25 检索
tokenized_query = query.split()
bm25_scores = bm25.get_scores(tokenized_query)
bm25_ranked = sorted(enumerate(bm25_scores), key=lambda x: x[1], reverse=True)

# 向量检索
q_emb = embed_model.encode([query], normalize_embeddings=True).astype('float32')
distances, indices = index.search(q_emb, len(documents))
vec_ranked = [(idx, dist) for dist, idx in zip(distances[0], indices[0])]

# RRF 融合
fused = rrf_fusion(bm25_ranked, vec_ranked)

print(f"查询: {query}\n")
print("=== BM25 排名 ===")
for rank, (idx, score) in enumerate(bm25_ranked[:5]):
    print(f"  {rank+1}. [{score:.2f}] {documents[idx][:50]}")
print("\n=== 向量检索排名 ===")
for rank, (idx, score) in enumerate(vec_ranked[:5]):
    print(f"  {rank+1}. [{score:.4f}] {documents[idx][:50]}")
print("\n=== RRF 融合排名(最终) ===")
for rank, (idx, score) in enumerate(fused[:5]):
    print(f"  {rank+1}. [{score:.4f}] {documents[idx][:50]}")
```

### 实战 2：Rerank 重排序

```python
"""用 CrossEncoder 做 Rerank，提升最终结果精度"""
from sentence_transformers import CrossEncoder

# 加载 Rerank 模型(首次自动下载)
reranker = CrossEncoder("BAAI/bge-reranker-v2-m3", max_length=512)

# 从混合检索拿到候选 Top-10，Rerank 精排
# 接上面代码...
candidate_indices = [idx for idx, _ in fused[:10]]  # Top-10 候选

# 构造 (query, doc) 对
pairs = [[query, documents[idx]] for idx in candidate_indices]

# Rerank 打分
rerank_scores = reranker.predict(pairs)

# 按 Rerank 分重排
reranked = sorted(
    zip(candidate_indices, rerank_scores),
    key=lambda x: x[1], reverse=True
)

print("=== Rerank 后最终 Top-3 ===")
for rank, (idx, score) in enumerate(reranked[:3]):
    print(f"  {rank+1}. [Rerank={score:.4f}] {documents[idx]}")

# 预期效果: Rerank 后 "401错误码" 的文档排第一
# 即使它和 query 的措辞不完全一致
```

## 进阶方向

### RAG 调优策略速查表

| 症状 | 可能原因 | 调优方向 |
|------|---------|---------|
| 检索不到相关文档 | 切分粒度不对/Embedding 模型不匹配 | 调整 chunk_size、换 Embedding 模型 |
| 检索到无关文档多 | 纯向量检索被语义误导 | 加 BM25 混合检索、提高相似度阈值 |
| Top-K 有答案但排序靠后 | 一级检索不够精细 | 加 Rerank 模型 |
| 答案有编造(幻觉) | Prompt 约束不够 | 加强"不知道就说不知道"指令 |
| 答案前后矛盾 | 多文档信息冲突 | 加文档来源过滤、时效性权重 |
| 专业术语检索差 | Embedding 模型领域不匹配 | 用领域微调的 Embedding 模型 |

### 工业界调优 Checklist

```
检索阶段
  ✅ 文档是否按语义切分（非暴力截断）
  ✅ 是否启用混合检索（向量 + BM25）
  ✅ 是否有相似度阈值过滤
  ✅ 是否引入 Rerank 精排

生成阶段
  ✅ Prompt 是否要求引用出处
  ✅ Prompt 是否包含"不知道就说不准"指令
  ✅ 是否限制回答范围（不低于检索内容）

评估阶段
  ✅ 是否定期运行 RAGAS 评估
  ✅ 是否有 Bad Case 收集机制
  ✅ 是否有 A/B 测试框架
```

## 常见问题

### 小白最常踩的 3 个坑

1. **只调 Top-K 数值，不调检索策略**
   - 错误：Top-5 效果不好就改 Top-10，再不好就 Top-20，靠堆数量
   - 正确：先检查检索策略（混合检索 > 纯向量），再检查切分（语义切分 > 固定切分），Top-K 只是最后微调的旋钮

2. **不做评估就"凭感觉"调优**
   - 错误："改了 chunk_size=512，感觉效果好了一点"
   - 正确：建立评估集（至少 50 个问题+标准答案），每次改动后跑 RAGAS，用数据说话

3. **Rerank 后还保留所有文档**
   - 错误：Rerank 只改变排序，还是把所有文档塞进 Prompt
   - 正确：Rerank 后只取前 3-5 条，丢弃低分文档。减少噪音比增加信息更重要

### 自检题

**Q1**：什么情况下 BM25 比向量检索更有优势？举一个具体例子。

> **答案**：当查询包含精确的专有名词、编号、代码时。例如搜"Error-401-V3"，BM25 精确匹配能直接命中，而向量检索会把"401"理解为数字，"V3"理解为版本号，被语义误导到其他版本相关文档。混合检索的核心价值就是 BM25 补上向量检索不擅长精确匹配的短板。

**Q2**：RAGAS 的 "Faithfulness" 和 "Answer Relevancy" 分别衡量什么？哪个指标更重要？

> **答案**：Faithfulness（忠实度）衡量回答是否基于检索文档（有没有编造），Answer Relevancy（答案相关性）衡量回答是否切中问题（有没有跑题）。RAG 场景下 Faithfulness 更重要——一个"不忠实但相关"的回答（看起来对但实际上是编的）比"忠实但不相关"更有害，因为用户会相信看似正确的幻觉。

**Q3**：你有一个 RAG 系统，发现用户搜"部署"时经常返回"架构设计"文档而漏掉"部署教程"文档。应该从哪个环节入手修复？

> **答案**：首先检查 Embedding 模型是否对"部署/架构"区分度不足（换模型或加同义词扩展）；其次加 Query 改写——把"部署"改写为"部署教程 部署步骤 环境配置"；最后能加混合检索更好，BM25 可以精确命中"部署"这个关键词。
