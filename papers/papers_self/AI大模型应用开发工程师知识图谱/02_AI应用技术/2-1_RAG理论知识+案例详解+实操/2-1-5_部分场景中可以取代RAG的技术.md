## 核心概念

### RAG 不是银弹

**一句话**：RAG 解决了很多问题，但不是所有场景的最优解。长上下文模型、Agentic Search、LLM Wiki 三条路线正在不同场景下取代传统 RAG。

```
传统 RAG: 检索 → 拼接 → 生成（"先检索，后思考"）
新范式:   思考 → 决策 → 行动（"边思考边找信息"）
```

**类比**：RAG 是去图书馆借书（检索→阅读→回答），长上下文是直接带整座图书馆进场（全量注入），Agentic Search 是自己会逛图书馆的 AI 助手（什么时候找、找什么、怎么找自己决定）。

### 什么时候该考虑替换 RAG

| 场景 | RAG 的问题 | 替代方案 |
|------|-----------|---------|
| 知识总量不大（<1M token） | 检索有遗漏，不如全量给 | Long-Context LLM |
| 需要多步搜索和交叉验证 | 单次检索不够 | Agentic Search |
| 知识需持续自我进化 | 人工维护知识库成本高 | LLM Wiki |
| 需要深度分析和总结 | 文档碎片丢失全局视角 | Long-Context / GraphRAG |

## 原理讲解

### Long-Context LLM：上下文就是知识库

2024 年，Gemini 1.5 Pro 支持 1M token（≈75 万字），Claude 支持 200K，GPT-4o 支持 128K。当 context window 足够大，"检索"这个步骤直接可以被跳过。

**核心思想**：

```
传统方案(小context, <32K):
  文档库(1000本) → 检索(选10本) → 塞入context → LLM回答
  问题: 检索可能漏掉关键信息

新方案(大context, >128K):
  文档库(如果总量< context window) → 全量塞入context → LLM回答
  优势: 不会遗漏任何信息，LLM "看见"完整上下文
```

**实战架构：全量知识注入**：

```python
# 伪代码: 长上下文模型替代 RAG
def long_context_qa(query, all_documents):
    # 不再检索，直接把所有文档当作上下文
    prompt = f"""以下是全部知识库内容（共{len(all_documents)}篇）:
    
    {all_documents}  # 所有文档全量注入
    
    基于以上全部内容回答问题: {query}
    """
    return llm.generate(prompt)  # 需要模型支持大 context
```

**成本与性能的权衡**：

| 维度 | RAG | Long-Context |
|------|:---:|:------------:|
| 检索遗漏 | 可能 | 不会 |
| Token 消耗 | 低（只传检索结果） | 高（每次传全量） |
| 延迟 | 快（检索后只传 <5K token） | 慢（传 100K+ token） |
| 适用数据量 | 任意大小 | < context window 上限 |
| 多跳推理 | 弱（跨文档片段难关联） | 强（LLM 看见完整上下文） |

**决策建议**：
- 知识总量 < 100K token → 直接 Long-Context，不需要 RAG
- 知识总量 > 1M token → RAG 为主，特定分析任务用 Long-Context
- 混合方案（推荐） → RAG 粗筛 + Long-Context 精读

### Agentic Search：让 AI 自己决定什么时候搜、搜什么

传统 RAG 的问题：**人类要预设"什么时候该检索"**——但 AI 应该能自己判断。

**从"Just-In-Case"到"Just-In-Time"**：

```
传统 RAG(Just-In-Case):
  不管什么问题，都先检索一遍(以防万一需要外部知识)
  → 浪费: 简单问题不需要检索也白搜了

Agentic Search(Just-In-Time):
  AI 先想:"这个问题我知道答案吗？"
  → 知道 → 直接回答(零检索开销)
  → 不知道 → 自动去搜索 → 再回答
  → 不确定 → 搜一下确认 → 再回答
```

**核心工作流：ReAct（Reasoning + Acting）**：

```
用户: "DeepSeek V3 和 Qwen 2.5 在 MATH 基准上谁更强？"

Agent 思考链:
  💭 我需要找到两个模型在 MATH 基准上的分数
  🎬 搜索: "DeepSeek V3 MATH benchmark score"
  👁 结果: DeepSeek V3 MATH 得分 90.2
  💭 还需要 Qwen 2.5 的数据
  🎬 搜索: "Qwen 2.5 MATH benchmark score"
  👁 结果: Qwen 2.5 MATH 得分 85.6
  💭 现在可以比较了
  🎬 回答: "DeepSeek V3(90.2) 高于 Qwen 2.5(85.6)"
```

**Plan-and-Solve 架构**：

```
复杂问题先做计划，再分步执行:

用户: "对比 2024 年三家云厂商的 AI 推理服务价格"

Agent:
  Plan:
    1. 列出三家厂商(AWS/阿里云/腾讯云)
    2. 分别查 AI 推理服务定价
    3. 按相同规格(如 8×A100)归一化
    4. 生成对比表
  Execute: 按计划逐步执行，每步验证结果
```

### LLM Wiki：让知识库自己进化

传统 RAG 的致命缺陷——**知识库是"死"的**：

```
问题: 人工维护知识库 →
  - 新知识来了要手动更新
  - 旧知识过期了没人删
  - 矛盾的知识没人发现和修正
  → 知识库越来越"脏"，回答越来越差
```

**LLM Wiki 范式**：知识库不再由人维护，而是 LLM 自己读、自己总结、自己更新。

```
传统模式:
  人写文档 → 人更新知识库 → RAG 检索

LLM Wiki 模式:
  对话/文档 → LLM 自动提取知识 → 存入结构化 Wiki
  → LLM 自动检测矛盾 → 标记/合并/更新
  → LLM 自动发现知识空白 → 提问/搜索补全
```

**自进化机制的四个步骤**：

```
1. 吸收 — 从对话和文档中提取结构化知识
2. 检测 — 发现矛盾(新旧结论冲突)、碎片(相关知识点分散在多处)
3. 进化 — 合并冲突、链接碎片、标记置信度
4. 反哺 — 进化后的知识反哺给下一次对话
```

### 实战 2：Agentic Search（ReAct 模式）

```python
"""用 ReAct 模式实现自主搜索 Agent"""
import json

# 模拟搜索工具
def web_search(query):
    """模拟搜索(实际使用 duckduckgo-search 等)"""
    mock_knowledge = {
        "DeepSeek": "DeepSeek V3: MoE架构, 671B参数, MATH得分90.2, API价格输入1元/1M token",
        "Qwen": "Qwen2.5-72B: 72B参数, MATH得分85.6, 开源免费",
        "GPT-4o": "GPT-4o: 未公开参数量, MATH得分88.7, API价格输入$2.5/1M token",
    }
    for key, value in mock_knowledge.items():
        if key.lower() in query.lower():
            return value
    return f"未找到关于 '{query}' 的信息"

def react_agent(question, max_steps=5):
    """ReAct: 思考-行动-观察 循环"""
    history = []
    print(f"用户: {question}\n")

    for step in range(max_steps):
        # 构造思考 Prompt
        prompt = f"""问题: {question}

已有信息:
{json.dumps(history, ensure_ascii=False, indent=2)}

你需要决定下一步:
- 如果已有足够信息回答: 输出 {{"action": "answer", "content": "你的回答"}}
- 如果还需要搜索: 输出 {{"action": "search", "query": "搜索关键词"}}

当前步骤 {step+1}/{max_steps}:"""

        print(f"--- Step {step+1} ---")
        print(f"Prompt 已准备好({len(prompt)} 字符)")

        # 这里实际调用 LLM(DeepSeek/GPT)做决策
        # response = llm.chat(prompt)
        # decision = json.loads(response)

        # 模拟 ReAct 决策(实际中由 LLM 自主判断)
        if step == 0:
            decision = {"action": "search", "query": "DeepSeek V3 MATH 得分 价格"}
        elif step == 1:
            decision = {"action": "search", "query": "Qwen2.5 MATH 得分"}
        elif step == 2:
            decision = {"action": "search", "query": "GPT-4o MATH 得分"}
        else:
            math_scores = [("DeepSeek V3", 90.2), ("GPT-4o", 88.7), ("Qwen2.5-72B", 85.6)]
            best = max(math_scores, key=lambda x: x[1])
            decision = {
                "action": "answer",
                "content": f"数学能力最强的是 {best[0]}，MATH 得分 {best[1]}。DeepSeek V3 API 价格为输入 1元/1M token，性价比最高。"
            }

        if decision["action"] == "search":
            result = web_search(decision["query"])
            print(f"  搜索: {decision['query']}")
            print(f"  结果: {result}")
            history.append({"step": step+1, "action": "search",
                          "query": decision["query"], "result": result})
        elif decision["action"] == "answer":
            print(f"\n最终回答: {decision['content']}")
            return decision["content"]

        print()

    return "达到最大搜索步数"

# 测试
react_agent("对比 DeepSeek、Qwen、GPT-4o 的数学能力和价格")
```

### 实战 3：LLM Wiki 的自进化演示

```python
"""LLM Wiki: 从对话中自动提取和进化知识"""
from dataclasses import dataclass, field
from typing import Dict, List

@dataclass
class KnowledgeEntry:
    """知识条目"""
    topic: str
    fact: str
    confidence: float   # 置信度 0-1
    source: str          # 来源
    version: int = 1

@dataclass
class LLMWiki:
    """简易 LLM Wiki 实现"""
    entries: Dict[str, List[KnowledgeEntry]] = field(default_factory=dict)

    def absorb(self, topic, fact, confidence, source):
        """吸收新知识"""
        entry = KnowledgeEntry(topic, fact, confidence, source)
        if topic not in self.entries:
            self.entries[topic] = []
        self.entries[topic].append(entry)
        print(f"  ✅ 吸收: [{topic}] {fact} (置信度: {confidence})")

    def detect_conflict(self, topic):
        """检测同一主题下的矛盾知识"""
        if topic not in self.entries or len(self.entries[topic]) < 2:
            return None
        facts = self.entries[topic]
        # 检测明显矛盾
        for i, f1 in enumerate(facts):
            for f2 in facts[i+1:]:
                if f1.fact != f2.fact:
                    return (f1, f2)
        return None

    def evolve(self, topic):
        """进化: 解决冲突，保留最新/置信度最高的"""
        conflict = self.detect_conflict(topic)
        if conflict:
            old, new = conflict
            if new.confidence >= old.confidence:
                print(f"  🔄 进化: {old.fact} → {new.fact} (v{old.version}→v{new.version})")
                old.version += 1  # 标记旧版本
        # 简化为保留最新

# === 演示 ===
wiki = LLMWiki()

print("=== 第1天: 初始知识 ===")
wiki.absorb("DeepSeek价格", "输入 0.5元/1M token", 0.8, "官网2024.6")

print("\n=== 第2天: 价格更新 ===")
wiki.absorb("DeepSeek价格", "输入 1元/1M token", 0.9, "官网2024.12")

print("\n=== 冲突检测 ===")
conflict = wiki.detect_conflict("DeepSeek价格")
if conflict:
    f1, f2 = conflict
    print(f"  ⚠️ 发现矛盾: '{f1.fact}' vs '{f2.fact}'")
    print(f"  来源: {f1.source} vs {f2.source}")

print("\n=== 自进化 ===")
wiki.evolve("DeepSeek价格")
print("  结果: 保留最新+高置信度版本")
```

## 进阶方向

### 三条路线的对比与选择

| 维度 | RAG | Long-Context | Agentic Search | LLM Wiki |
|------|:---:|:------------:|:--------------:|:--------:|
| 信息遗漏 | 可能有 | 无 | 极少 | 可能有 |
| Token 效率 | 高 | 低 | 中 | 高 |
| 自主性 | 无 | 无 | 高 | 中 |
| 知识进化 | 手动 | 手动 | 无 | 自动 |
| 实现复杂度 | 中 | 低 | 高 | 高 |
| 最佳场景 | 海量文档QA | 小知识库深度分析 | 复杂多步调研 | 持续积累的知识库 |

### 混合方案（推荐最佳实践）

```
智能路由:
  用户提问
    ├→ 简单事实性问题 → 直接回答(不走任何增强)
    ├→ 知识库可覆盖 → RAG 检索
    ├→ 需要深度分析 → Long-Context 全量注入
    ├→ 需要多步搜索 → Agentic Search
    └→ 新知识 → LLM Wiki 自动吸收
```

## 常见问题

### 小白最常踩的 3 个坑

1. **上下文越大 = 效果越好？**
   - 错误：不管什么场景都想着"砸全量进去"
   - 正确：长上下文有"迷失中间"问题——模型对长文本中间部分的信息处理最弱。关键信息放开头或结尾。且长上下文 Token 成本可观(100K token 输入可能是检索的 20 倍费用)

2. **Agentic Search 不设最大步数限制**
   - 错误：Agent 无限制搜索，陷入"搜了又搜"的死循环
   - 正确：设置 max_steps（通常 5-8 步），超时强制输出。每次搜索前让 Agent 自问"这个搜索是必需的吗？"

3. **LLM Wiki 盲目信任模型提取的知识**
   - 错误：LLM 自动提取的知识都当作事实
   - 正确：所有自动提取的知识都标记置信度，设置阈值。低于阈值的不进知识库，人工审核后再入库

### 自检题

**Q1**：什么时候用 Long-Context 全量注入比 RAG 更好？给出一个具体判断标准。

> **答案**：当知识总量 < context window 的 70% 且需要跨文档综合分析时。判断标准：如果任务要求"通读所有资料才能正确回答"（如对比分析、总结归纳），RAG 的碎片化检索反而会丢失全局视角，此时全量注入更合适。

**Q2**：ReAct 和 Plan-and-Solve 两种 Agentic Search 架构的核心区别是什么？

> **答案**：ReAct 是"走一步看一步"——每步思考后决定下一步行动，灵活但可能缺乏全局规划。Plan-and-Solve 是"先做计划再执行"——先规划完整步骤链，再逐步执行，适合目标明确的多步任务，但遇到计划外情况调整不如 ReAct 灵活。

**Q3**：LLM Wiki 相比于传统 RAG 知识库，最核心的改进是什么？

> **答案**：传统 RAG 知识库是"死的"——人写、人维护。LLM Wiki 是"活的"——从对话和文档中自动吸收新知识，自动检测矛盾（新旧信息冲突），自动进化（合并、更新、补全）。知识库不再需要人工维护就能持续更新。
