## 核心概念

### 什么是 RAG

**一句话**：RAG（Retrieval-Augmented Generation，检索增强生成）是让 LLM 在回答问题前先去"翻书"——从外部知识库检索相关信息，再结合检索结果生成回答。

```
没有 RAG：用户问 → LLM 直接回答（靠训练记忆，可能产生幻觉）
有 RAG：  用户问 → 检索相关文档 → 把文档+问题一起给 LLM → LLM 基于文档回答
```

**类比**：LLM 是一个很聪明但记忆有限的"考生"。RAG 就是给了这个考生一本"开卷参考书"——不用背，翻书找答案就行。

### 大模型开发的三种范式

| 范式 | 做法 | 成本 | 适用场景 |
|------|------|:----:|---------|
| **Prompt Engineering** | 直接写提示词，不修改模型 | 最低 | 简单任务、通用问答 |
| **RAG** | 不改模型，外挂知识库 | 中等 | 知识密集型、需最新信息 |
| **Fine-tuning** | 修改模型参数 | 最高 | 定制风格、特定领域深度优化 |

**选择决策**：
```
需要模型"知道"新知识？
  ├─ 知识是事实/数据，经常更新 → RAG（不改模型，知识放向量库）
  ├─ 知识是思维方式/专业术语风格 → Fine-tuning（内化进模型）
  └─ 两者都要 → RAG + Fine-tuning 组合
```

### 为什么 RAG 重要

1. **解决幻觉**：给 LLM 提供事实依据，大幅减少编造
2. **知识实时更新**：知识库更新即生效，不需要重新训练模型
3. **可溯源**：每个答案都能追溯到具体文档，比"模型内部记忆"更可信
4. **低成本**：不需要 GPU 集群做微调，CPU 也能跑向量检索

## 原理讲解

### RAG 的完整流程（五步）

```
用户查询: "DeepSeek 有哪些版本？"
  │
  ▼
┌──────────────────────────────────────────────────┐
│ 1. Query 处理                                    │
│    改写/扩写: "DeepSeek 版本 型号 发布"          │
│    目的: 短查询 → 更具检索性的表达              │
└───────────────────┬──────────────────────────────┘
                    ▼
┌──────────────────────────────────────────────────┐
│ 2. 检索 (Retrieval)                              │
│    将 Query Embedding → 向量库搜索 Top-K 相关文档 │
│    返回: 5 条相关文档片段                        │
└───────────────────┬──────────────────────────────┘
                    ▼
┌──────────────────────────────────────────────────┐
│ 3. 重排序 (Rerank) [可选但推荐]                  │
│    用更强的模型对 Top-K 结果重新打分排序           │
│    过滤低相关度的, 保留最相关的 3 条              │
└───────────────────┬──────────────────────────────┘
                    ▼
┌──────────────────────────────────────────────────┐
│ 4. 构建 Prompt                                   │
│    拼接: System Prompt + 检索文档 + 用户问题      │
│    "基于以下参考资料回答问题: ..."                │
└───────────────────┬──────────────────────────────┘
                    ▼
┌──────────────────────────────────────────────────┐
│ 5. 生成 (Generation)                             │
│    LLM 基于拼接的 Prompt 生成最终回答             │
│    输出: "DeepSeek 有 V2/V3/R1 等版本..."        │
└──────────────────────────────────────────────────┘
```

### Native RAG vs Advanced RAG

**Native RAG（基础版）**：用户问什么就 Embedding 什么，直接检索 → 直接生成。简单粗暴，容易出问题。

**Advanced RAG（进阶版）**：在 Native 基础上加了多个优化环节：

| 阶段 | Native RAG | Advanced RAG |
|------|-----------|-------------|
| 检索前 | 无 | Query 改写、Query 分解、HyDE（生成假设文档再检索） |
| 检索中 | 单一向量检索 | 混合检索（向量+关键词）、多路召回 |
| 检索后 | 无 | Rerank 重排序、上下文压缩 |
| 生成 | 直接拼接 | 引用标注、置信度评估、拒答机制 |

### Query 改写：凭什么要改写用户的问题

用户输入的查询往往很糟糕：
- "那个模型怎么样？"（缺少主语）
- "v3"（太简短）
- "和昨天那个比较一下"（依赖上下文）

Query 改写让 LLM 把模糊查询变成精确的检索语句：

```
原始: "v3 vs r1"
改写: "DeepSeek V3 和 DeepSeek R1 在性能、架构、应用场景上的对比"
```

这个改写过程本身调用一次轻量 LLM（甚至本地小模型），但对检索质量的提升是决定性的。

### 知识库处理的四个关键场景

| 场景 | 做什么 | 为什么重要 |
|------|--------|-----------|
| **1. 问题生成与检索优化** | 为每个文档片段预生成"可能被问到的问题" | 用户问法千变万化，提前覆盖多种问法 |
| **2. 对话知识沉淀** | 把对话中的有用结论写入知识库 | 知识库持续进化，越用越好 |
| **3. 知识库健康度检查** | 定期检测：死链、重复、矛盾、过期内容 | 脏数据 = 坏答案 |
| **4. 版本管理与对比** | A/B 测不同切分策略、不同 Embedding 模型 | 找到最佳配置组合 |

### 如何系统提升 RAG 质量

从五个维度优化：

```
1. 数据质量（最重要）
   垃圾进垃圾出 → 清洗、去重、结构化原始文档

2. 切分策略（最容易被忽视）
   切太碎 → 语义不完整；切太大 → 噪音多
   → 语义切分（按段落/章节）、递归切分、小粒度+父文档回填

3. 检索策略（最直接影响效果）
   → 多路召回（向量+BM25）+ Rerank

4. Prompt 设计（成本最低的优化）
   → 明确引用要求、"不知道就说不知道"、限制回答范围

5. 评估体系（保证迭代方向正确）
   → RAGAS（RAG Assessment）: 忠实度、相关性、上下文精度
```

### 实战 1：从零搭建本地知识库 RAG（DeepSeek + FAISS）

```python
"""
最小 RAG 系统：50 行代码实现"文档检索 + LLM 生成"
"""
import numpy as np
from sentence_transformers import SentenceTransformer

# ========== 1. 准备知识库 ==========
knowledge_base = [
    "DeepSeek V3 发布于 2024 年 12 月，采用 MoE 架构，总参数 671B，激活参数 37B。",
    "DeepSeek R1 是推理增强模型，擅长数学和编程，采用强化学习训练。",
    "DeepSeek V2 引入 MLA（Multi-head Latent Attention）机制，大幅降低 KV Cache。",
    "DeepSeek API 价格：输入 1 元/百万 token，输出 2 元/百万 token。",
    "FAISS 是 Meta 开源的高效向量检索库，支持 GPU 加速。",
]

# ========== 2. 构建向量索引 ==========
embed_model = SentenceTransformer("BAAI/bge-small-zh-v1.5")
doc_embeddings = embed_model.encode(knowledge_base, normalize_embeddings=True)

import faiss
dim = doc_embeddings.shape[1]
index = faiss.IndexFlatIP(dim)
index.add(doc_embeddings.astype('float32'))

# ========== 3. 检索函数 ==========
def retrieve(query, top_k=3):
    """检索最相关的文档片段"""
    q_emb = embed_model.encode([query], normalize_embeddings=True).astype('float32')
    distances, indices = index.search(q_emb, top_k)
    results = []
    for dist, idx in zip(distances[0], indices[0]):
        if dist > 0.3:  # 相似度阈值，过滤噪音
            results.append((dist, knowledge_base[idx]))
    return results

# ========== 4. 构建 RAG Prompt ==========
def build_rag_prompt(query, retrieved_docs):
    """拼接检索结果到 Prompt"""
    context = "\n".join([f"- {doc}" for _, doc in retrieved_docs])
    return f"""基于以下参考资料回答问题。如果资料中没有相关信息，请明确说"资料中未找到相关信息"。

【参考资料】
{context}

【用户问题】
{query}

【回答】"""

# ========== 5. 测试 ==========
query = "DeepSeek V3 用了什么架构？"
docs = retrieve(query)
print(f"检索到 {len(docs)} 条相关文档:")
for score, doc in docs:
    print(f"  [{score:.4f}] {doc}")

prompt = build_rag_prompt(query, docs)
print(f"\n--- 完整 Prompt ---\n{prompt}")

# 预期输出:
# 检索到 X 条相关文档:
#   [0.8xxx] DeepSeek V3 发布于 2024 年 12 月，采用 MoE 架构...
# --- 完整 Prompt ---
# 基于以下参考资料回答问题...
# 【参考资料】
# - DeepSeek V3 发布于 2024 年 12 月，采用 MoE 架构...
```

**上述代码实际调用 LLM 时，只需用 OpenAI/DeepSeek API 发送 `prompt` 即可**：

```python
# 接上面代码，实际调用 LLM
from openai import OpenAI

client = OpenAI(
    api_key="your-deepseek-api-key",
    base_url="https://api.deepseek.com"
)

response = client.chat.completions.create(
    model="deepseek-chat",
    messages=[{"role": "user", "content": prompt}],
    temperature=0.3
)
print(response.choices[0].message.content)
```

### 实战 2：Query 改写

```python
"""
用 LLM 改写用户查询，提升检索效果
"""
from openai import OpenAI

client = OpenAI(
    api_key="your-api-key",
    base_url="https://api.deepseek.com"
)

def rewrite_query(original_query, history=None):
    """将模糊查询改写为精确的检索查询"""
    system_prompt = """你是一个查询改写助手。将用户的问题改写为更适合做知识库检索的形式。

改写规则：
1. 补全代词和省略（"它"→具体对象名）
2. 扩展缩写和简称（"v3"→"DeepSeek V3"）
3. 增加同义词覆盖（"咋用"→"如何使用 使用方法 使用教程"）
4. 返回 2 个改写版本，用换行分隔
5. 只输出改写结果，不要解释"""

    messages = [{"role": "system", "content": system_prompt}]
    if history:
        messages.insert(0, {"role": "system",
            "content": f"对话历史: {history}"})
    messages.append({"role": "user", "content": original_query})

    response = client.chat.completions.create(
        model="deepseek-chat",
        messages=messages,
        temperature=0.3,
        max_tokens=200
    )
    return response.choices[0].message.content.strip().split("\n")

# 测试
test_queries = [
    "v3 和 r1 有什么区别",
    "那它的价格呢",  # 依赖上下文
    "咋部署",
]

for q in test_queries:
    rewritten = rewrite_query(q)
    print(f"原始: {q}")
    for i, rw in enumerate(rewritten, 1):
        print(f"  改写{i}: {rw}")
    print()

# 预期输出示例:
# 原始: v3 和 r1 有什么区别
#   改写1: DeepSeek V3 和 DeepSeek R1 在性能架构应用场景上的对比区别
#   改写2: DeepSeek V3 对比 DeepSeek R1 差异比较优缺点
```

### 实战 3：知识库文档切分

```python
"""
文档切分策略对比：固定长度 vs 递归切分 vs 语义切分
"""
from langchain.text_splitter import (
    RecursiveCharacterTextSplitter,
    CharacterTextSplitter,
)

sample_text = """
第一章：深度学习基础
深度学习是机器学习的一个分支，它使用多层神经网络来学习数据的表示。
与传统的机器学习方法不同，深度学习可以自动从原始数据中提取特征。

1.1 神经网络
神经网络由多个神经元层组成。每个神经元接收输入，应用权重和偏置，然后通过激活函数输出。
常见的激活函数包括 ReLU、Sigmoid 和 Tanh。

1.2 反向传播
反向传播是训练神经网络的核心算法。它通过计算损失函数对每个参数的梯度，使用链式法则从输出层向输入层逐层传递误差。
梯度下降优化器（如 Adam、SGD）使用这些梯度来更新模型参数。

第二章：Transformer 架构
Transformer 是当前大语言模型的基础架构，由 Vaswani 等人在 2017 年提出。
它完全基于注意力机制，摒弃了传统的循环神经网络结构。
"""

# 方案 A: 固定字符切分（简单但有缺陷）
fixed_splitter = CharacterTextSplitter(
    separator="\n",
    chunk_size=200,
    chunk_overlap=0,
)
chunks_a = fixed_splitter.split_text(sample_text)
print(f"方案A 固定切分: {len(chunks_a)} 块")
for i, c in enumerate(chunks_a):
    print(f"  块{i}: {c[:60]}...")

# 方案 B: 递归切分（推荐）
recursive_splitter = RecursiveCharacterTextSplitter(
    separators=["\n\n", "\n", "。", ".", " ", ""],
    chunk_size=300,
    chunk_overlap=50,
)
chunks_b = recursive_splitter.split_text(sample_text)
print(f"\n方案B 递归切分: {len(chunks_b)} 块")
for i, c in enumerate(chunks_b):
    print(f"  块{i}: {c[:80]}...")

# 要点：递归切分按优先级尝试分隔符，尽可能保持语义完整
# chunk_overlap 保证边界处信息不被丢失
```

## 进阶方向

### RAG 技术的局限

1. **检索失败**：知识库有答案但没搜到（召回率不足）
2. **上下文窗口有限**：检索到的文档太长，可能超出 LLM 的 context window
3. **多跳推理困难**：问题需要综合多个文档的信息时效果下降
4. **知识冲突**：检索到的多个文档相互矛盾时，LLM 难以判断

### 工业界最佳实践

| 实践 | 说明 |
|------|------|
| **混合检索** | 向量检索 + BM25 关键词检索，互补覆盖 |
| **Rerank 模型** | 用 BGE-Reranker / Cohere Rerank 对 Top-K 重排序 |
| **小粒度+父文档** | 检索用小粒度片段，生成时回填完整段落 |
| **元数据过滤** | 按时间/类别/来源过滤，缩小检索范围 |
| **缓存热门查询** | 相同问题直接返回缓存结果，节省成本 |

### 从 RAG 到更高级的范式

- **Graph RAG**：知识图谱代替文档，实现多跳推理 → [2.1.5 部分场景中可以取代RAG的技术](2-1-5_部分场景中可以取代RAG的技术.md)
- **Agentic RAG**：RAG 步骤由 Agent 自主决策（何时检索、检索什么、如何验证）→ [2.2.1 Function Calling与MCP](../2-2_Agent理论知识+案例详解+实操/2-2-1_Function_Calling与MCP.md)
- **Long Context 模型**：部分场景可替代 RAG → [2.1.5](2-1-5_部分场景中可以取代RAG的技术.md)

## 常见问题

### 小白最常踩的 3 个坑

1. **检索出来直接拼接，不做过滤**
   - 错误：把所有 Top-K 结果都塞进 Prompt，包括相似度 0.1 的噪音文档
   - 正确：设置相似度阈值（如 ≥0.4），过滤低相关文档。如果所有文档都被过滤，明确告知用户"未找到相关信息"

2. **Query 不做任何处理就检索**
   - 错误：用户问"V3 多少钱"，直接 Embedding "V3 多少钱"去检索
   - 正确：先做 Query 改写 → "DeepSeek V3 价格 API 收费标准"，再检索。特别是依赖对话上下文的查询（"那它呢"）必须补全

3. **切分策略不考虑文档结构**
   - 错误：把所有文档按 500 字符硬切，不管句子是否被截断
   - 正确：按文档实际结构（段落、章节）切分，保持语义完整。长代码/表格不要横切

### 自检题

**Q1**：一个用户问"请解释 RAG 的原理"，RAG 系统检索到一段讲"Embedding 原理"的文档。这段文档应该被使用吗？为什么？

> **答案**：不应该。虽然 Embedding 是 RAG 的组成部分，但用户问的是 RAG 整体原理，不是 Embedding。这暴露了纯向量检索的问题——"相关"不等于"有用"。可以通过提高相似度阈值、使用 Rerank 模型重新排序来解决。

**Q2**：为什么 Advanced RAG 比 Native RAG 多"Query 改写"这一步？直接用用户原话去检索有什么问题？

> **答案**：用户原话通常存在三大问题：(1) 口语化、不完整；(2) 缺少关键词和同义词；(3) 依赖上下文（如"它的价格"）。直接检索可能导致召回率低。Query 改写将口语查询变成"搜索引擎友好"的精确表达，显著提升检索质量。

**Q3**：什么场景下 RAG 比 Fine-tuning 更合适？给出一个具体例子。

> **答案**：当知识频繁更新且需要可溯源性时，RAG 更合适。例如：公司内部知识库问答系统——每天有新的技术文档、政策更新，如果用 Fine-tuning 需要频繁重训（成本高），而 RAG 只需把新文档加入向量库即可（秒级生效），并且每个答案都能追溯到具体文档来源。
