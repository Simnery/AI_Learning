## 核心概念

- **是什么**：RAG 是 AI 大模型应用开发工程师面试的高频考点。面试官考察的不只是"你用过什么"，更是你对 RAG 系统设计的深度理解——从文档解析到检索策略到评估体系。
- **为什么重要**：RAG 是大多数 AI 应用工程师日常工作的核心。面试中的 RAG 问题答得好，基本就拿到了 50% 的分数。

## 原理讲解

面试中的 RAG 问题可按深度分为三层：
- **基础层**：RAG 是什么、架构怎么画、核心组件有哪些（考察知识面）
- **设计层**：为什么选这个 Embedding/向量库/分块策略、做过什么调优（考察工程经验）
- **反思层**：RAG 的局限性、什么场景可能不适用、下一步演进方向（考察架构能力）

面试官通过追问来判断你在哪一层。从"我用过 RAG"→"我设计过 RAG 系统"→"我知道什么时候不该用 RAG"，层层深入。

## 代码实战

```python
"""面试模拟器——用 AI 做 RAG 面试练习"""

# pip install openai

from openai import OpenAI

class RAGInterviewSimulator:
    """RAG 面试模拟——帮你练习常见面试问题"""
    
    def __init__(self, llm_endpoint: str = "http://localhost:8000/v1"):
        self.client = OpenAI(base_url=llm_endpoint, api_key="local")
        self.questions = [
            "大模型应用开发有哪三种主要模式？各自适用什么场景？",
            "文档分块策略怎么选？chunk_size 过大或过小各有什么问题？",
            "画出 RAG 架构图，并解释每个组件的设计决策",
            "向量检索的缺点是什么？什么是混合检索？",
            "RAG 效果不好时，你的调优顺序是什么？",
        ]
    
    def simulate(self, question_index: int = 0):
        """模拟面试——提问→你的回答→AI 点评"""
        q = self.questions[question_index % len(self.questions)]
        print(f"面试官: {q}")
        
        # 这里你可以输入你的回答
        your_answer = input("\n你的回答: ")
        
        # AI 点评
        feedback = self.client.chat.completions.create(
            model="local-model",
            messages=[{
                "role": "system",
                "content": "你是面试教练。评估候选人的回答，指出优点、不足和改进建议。"
            }, {
                "role": "user",
                "content": f"面试问题: {q}\n候选人回答: {your_answer}\n请给出点评。"
            }],
            temperature=0,
        )
        print(f"\n点评: {feedback.choices[0].message.content}")


# 使用:
# sim = RAGInterviewSimulator()
# sim.simulate(0)
```


### 预期输出

交互式运行后：终端打印 `面试官: …`，输入回答后输出 `点评: …`。无本地 LLM 时可只打印问题，用纸笔自评。

## 常见问题

### 面试题与回答要点（精选）

### Q1: 大模型应用开发的三种主要模式？

**答案框架**：1）**Prompt工程**——不修改模型，通过设计提示词引导输出，适合简单任务 2）**RAG**——外挂知识库，检索+生成，适合需要事实准确性的场景 3）**微调**——修改模型权重，适合需要特定风格/领域知识的场景。选择依据：数据量、实时性、成本、准确性需求。

### Q2: 文档分块策略？怎么选？

**答案要点**：三种策略——固定大小（简单但可能切断语义）、语义分块（按段落/句子，效果好但实现复杂）、递归分块（折中方案，我的项目用这个）。我的选择逻辑：chunk_size=300-500（平衡精确性和上下文完整性）、overlap=10-20%（防止关键信息被切断）。

### Q3: 画出RAG架构图并解释

**答案框架**：文档加载→文本解析(MinerU/Docling)→分块→Embedding→向量库(FAISS/LanceDB)→用户提问→向量检索→重排序(Rerank)→拼接上下文→LLM生成。关键技术决策：为什么用这个Embedding模型（MTEB榜单+中文性能）、为什么用混合检索而非纯向量检索。

### Q4: 为什么选这个Embedding模型？

**回答要点**：我参考 MTEB 中文榜单，选择 BGE/M3E。考量因素：1）中文语义理解能力 2）向量维度（768 vs 1536，平衡精度和存储）3）最大序列长度（512 vs 8192，长文档友好）。实践验证：在我的测试集上，M3E准确率85%，BGE-large 88%。

### Q5: RAG效果差，怎么调？

**调试清单（从快到慢）**：1）检查文档解析质量（PDF解析是不是丢失了表格）2）调chunk大小（太大不精准，太小缺上下文）3）优化检索（加混合检索+重排序）4）优化Query（改写模糊提问）5）优化Prompt模板（是否充分使用了检索到的上下文）6）换更强的Embedding/LLM模型。

### Q6: 多轮对话中RAG怎么优化？

**方案**：1）Query改写：将"那上次那个呢"改写为"上次讨论的XX问题" 2）上下文继承：将上一轮的检索结果/关键实体注入当前轮 3）对话历史压缩：用LLM总结历史关键点作为检索查询。

### Q7: 向量检索的缺点？什么是混合检索？

**回答**：向量检索缺点——对精确匹配不友好（"订单号ORD-12345"）、同义词可能检索到不相关内容。混合检索=向量检索(BM25)+关键词检索，用RRF融合排序。我的实践：混合检索比纯向量检索准确率提升10-15%。

### Q8: 20条文档选最好的3条？(Rerank策略)

**回答**：两阶段策略——粗排（向量检索Top-20）→精排（LLM Rerank或Cross-Encoder Reranker选Top-3）。为什么不用简单Top-K：向量相似≠语义相关，Rerank能过滤掉"相似但不相关"的噪声。

### Q9: 知识库维护和迭代？

**回答**：1）增量更新：新文档自动解析入库（非全量重建）2）质量监控：追踪无答案率、用户反馈（赞/踩）3）过期处理：标记时效性，定期清理过期内容 4）A/B测试：新分块策略/新Embedding模型效果对比。

### 面试常见误区

**误区 1：背答案，不说设计决策**

面试官问"RAG 架构是什么"，很多候选人把架构图画一遍就完了。更好的回答："我们的设计是 X，当时考虑了 Y 和 Z 两种方案，选择 X 的原因是..."——展示你的决策能力而非记忆能力。

**误区 2：项目描述太笼统**

"我搭建了一个企业知识库"不如"我用 Docling+M3E+LanceDB 搭建了 RAG 系统，文档解析准确率 92%，检索延迟 <200ms，支持 50 并发"——量化结果让面试官能评估你的水平。

**误区 3：不懂的强行编**

面试官通常比你资深，编造的技术细节 3 个追问就露馅。坦诚说"这个我没深入，但我理解核心思路是..."比瞎编好得多。
