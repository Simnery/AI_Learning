## 原理讲解

框架面试的核心在于"理解设计哲学而非记忆 API"：
- **为什么会有框架**：解决 LLM 开发的通用问题（模型切换、链式编排、记忆管理），避免每次从零写胶水代码
- **框架选型逻辑**：不同框架有不同的"中心抽象"——LangChain 以 Chain 为中心、LlamaIndex 以 Index 为中心、AutoGen 以 Agent 对话为中心
- **自己设计框架的能力**：能说出框架的核心抽象（LLM 接口、工具协议、编排引擎、记忆管理），说明你不仅会用还能造

"""200行极简 LLM 框架 —— 面试中展示框架设计理解"""

from abc import ABC, abstractmethod
from typing import Any

# ===== 1. LLM 统一接口 =====
class BaseLLM(ABC):
    @abstractmethod
    def complete(self, prompt: str, **kwargs) -> str:
        pass

class OpenAILLM(BaseLLM):
    def __init__(self, model: str = "gpt-3.5-turbo", api_key: str = ""):
        self.model = model
        self.api_key = api_key
    
    def complete(self, prompt: str, **kwargs) -> str:
        # 实际调用 OpenAI API（这里简化）:
        return f"[{self.model}]: {prompt[:50]}..."

# ===== 2. 工具协议 =====
class Tool:
    def __init__(self, name: str, description: str, handler: callable):
        self.name = name
        self.description = description
        self.handler = handler
    
    def execute(self, **params) -> Any:
        return self.handler(**params)

# ===== 3. 链式编排 =====
class Chain:
    def __init__(self, steps: list[callable]):
        self.steps = steps
    
    def run(self, input_data: Any) -> Any:
        result = input_data
        for step in self.steps:
            result = step(result)
        return result

# 使用演示:
# llm = OpenAILLM(model="deepseek-v4")
# tools = [Tool("search", "Search the web", lambda q: f"Found: {q}")]
# chain = Chain([lambda x: llm.complete(x), lambda x: x.upper()])
# print(chain.run("Hello"))

## 常见问题

### 面试题与回答要点（精选）

### Q1: LangChain解决了什么问题？

**一句话**：让 LLM 开发从"手写胶水代码"变成"标准化组件拼装"。具体：统一LLM接口（切换模型不改代码）、提供RAG/Agent/Memory等开箱即用模块、LCEL声明式编排。

### Q2: 六大核心组件及交互

Models（LLM接口）→ Prompts（模板管理）→ Chains（串联执行）→ Agents（自主决策）→ Memory（上下文存储）→ Indexes（文档检索）。交互：Chains 固定编排组件，Agents 动态决定调用顺序。

### Q3: 用LangChain构建RAG系统

**流程**：DocumentLoader→TextSplitter→Embeddings→VectorStore→Retriever→PromptTemplate→LLM→OutputParser。关键代码：`RunnablePassthrough` + LCEL管道串联。

### Q4: Memory机制如何实现？

**原理**：每次对话后将消息存入 Memory 对象（Buffer/Summary/Window），下次调用时自动注入到 Prompt 的对话历史中。SummaryMemory 额外用 LLM 压缩历史到固定长度。

### Q5: LangChain vs LlamaIndex 核心定位

**LangChain**：通用LLM应用框架，以Chain为中心。**LlamaIndex**：数据密集型RAG，以Index为中心。前者重编排，后者重数据。实际项目中经常混用——LlamaIndex做索引，LangChain做编排。

### Q6: 何时选LangChain，何时选LlamaIndex？

选LangChain：需要灵活编排多种LLM模式（RAG+Agent+Memory混用）。选LlamaIndex：数据连接器需求多（100+种格式）、索引结构复杂（树索引/关键词索引）、RAG是最核心需求。

### Q7: AutoGen多智能体vs单Agent区别

AutoGen核心是"Agent对话即工作流"——不需要显式定义调用顺序，Agent自己商量。单Agent（LangChain Agent）是"一个大脑+多个工具"——由Orchestrator控制流程。选AutoGen当任务需要多个Agent自主协商时。

### Q8: 从零设计LLM框架的组件抽象

**三层抽象**：1）基础设施层（LLM统一接口、工具协议、记忆存储）2）编排层（Chain/Graph/Agent执行引擎）3）应用层（RAG构建器、Agent工厂、API服务）。核心原则：接口先于实现、可插拔、可观测。

### Q9: 可插拔Tool/Plugin系统设计

**关键**：Tool 注册（name+description+schema+handler）→ Tool 发现（Agent按需匹配）→ 沙箱执行（隔离+超时）→ 结果标准化（统一JSON格式）。参考MCP协议（Model Context Protocol）的标准化思路。

### Q10: HuggingFace Pipelines优势与局限

**优势**：零代码推理、自动匹配模型+Tokenizer、支持20+任务类型。局限：黑盒（无法定制预处理）、不支持复杂训练流程、不适合生产级自定义pipeline。

### Q11-13: 微调/Tokenizers/PyTorch动态图

- **Tokenizer不匹配后果**：输出乱码/OOD token，因为不同tokenizer的词表不同
- **全参数微调缺点**：显存爆炸（7B需56GB）、灾难性遗忘、训练时间长
- **PyTorch动态图**：定义即执行（vs TF 1.x先定义再执行），调试友好，每个前向传播重新构建计算图


### 面试常见误区

**误区 1：记 API 不记原理**

面试官问"LangChain 的 Memory 怎么实现的"，不要只回答"用 ConversationBufferMemory"。要解释底层机制——每次调用时自动将历史消息注入到 Prompt 的对话历史字段中，SummaryMemory 额外用 LLM 压缩。

**误区 2：只会一个框架，对其他框架一无所知**

被问"LlamaIndex 和 LangChain 有什么不同"，回答"我没用过 LlamaIndex"就结束了。至少要能说出一两个核心差异——LangChain 以 Chain 为中心重编排，LlamaIndex 以 Index 为中心重数据。

**误区 3：框架对比变成"踩一捧一"**

"LangChain 太笨重了，不如自己写"——面试官会觉得你偏激且不专业。正确态度："每个框架有适用场景，我在项目中选择 X 因为...但 Y 在 Z 场景下也很好用"。
