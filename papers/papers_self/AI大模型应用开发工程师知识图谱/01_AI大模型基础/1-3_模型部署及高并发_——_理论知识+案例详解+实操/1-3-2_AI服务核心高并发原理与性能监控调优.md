### 高并发的本质问题

AI 推理和传统 Web 服务不同——每个请求不仅消耗 CPU，更消耗**大量 GPU 显存**。高并发的核心挑战是：**如何让有限的显存同时服务尽可能多的请求？**

打个比方：传统 Web 服务器像一个餐厅，多来几个客人多摆几张桌子就行。AI 推理像一个手术室——设备（GPU）只有一套，多台手术（请求）同时做需要精密的调度。

### 关键性能指标

| 指标 | 含义 | 目标值 |
|------|------|--------|
| **TTFT** (Time To First Token) | 发出请求到收到第一个 token 的时间 | < 500ms（实时交互） |
| **TPOT** (Time Per Output Token) | 每个输出 token 的间隔时间 | < 50ms（流畅阅读） |
| **Throughput** | 每秒处理的总 token 数 | 越高越好，受 GPU 上限约束 |
| **QPS** (Queries Per Second) | 每秒完成的请求数 | 取决于模型大小和并发 |
| **GPU 利用率** | GPU 计算核心的使用率 | > 80%（越高越不浪费） |
| **显存占用率** | GPU 显存的使用比例 | 85-95%（留一点余量防止 OOM） |

## 原理讲解

### KV Cache 为什么成为瓶颈

LLM 是自回归的——每生成一个 token，都要重新计算之前所有 token 的注意力。如果不缓存，第 N 个 token 的注意力计算量是 O(N²)。

**KV Cache 的作用**：把每个 token 的 Key 和 Value 向量存起来，下一个 token 只需要算增量。

```
请求1："请介绍 Transformer"（1000 tokens 上下文）
  模型权重：14 GB
  KV Cache：0.5 GB      ← 每个请求独立占用！
  
请求2：同问
  模型权重：14 GB（共享，只加载一次）
  KV Cache：0.5 GB      ← 又占用 0.5 GB！
  
10 个并发请求 → 14 + 0.5×10 = 19 GB 显存
```

**传统方案的问题**：KV Cache 是**连续分配**的——每个请求预先分配一大块连续显存，但实际可能只用了一半（因为生成长度不确定）。碎片严重，30% 显存可能浪费。

### PagedAttention 怎么解决

PagedAttention 借鉴了操作系统**虚拟内存分页**的思想：

```
传统方案：                           PagedAttention：
┌─────────────────────┐              ┌──┬──┬──┬──┬──┬──┐
│ 请求1: ██████░░░░░░  │ (浪费40%)     │ 1│ 1│  │ 2│ 1│ 3│  ← 按页分配
│ 请求2: ████░░░░░░░░  │ (浪费60%)     ├──┼──┼──┼──┼──┼──┤
│ 请求3: ████████░░░░  │ (浪费33%)     │ 3│ 2│ 1│  │ 2│  │  ← 不连续存储
└─────────────────────┘              └──┴──┴──┴──┴──┴──┘
                                      页表记录每个请求的页映射
```

三个核心优势：
1. **不连续分配**：KV Cache 不需要一大块连续显存，碎片大幅减少
2. **按需分配**：需要多少分配多少，不用预分配最大值
3. **前缀共享**：两个请求有相同的 System Prompt → 共享同一组 KV Cache 页（SGLang 的 RadixAttention）

### Continuous Batching（连续批处理）

传统批处理：等一批请求**全部完成**才发下一批。问题：如果这批里有一个生成长回答，其他短回答要干等。

```
传统批处理：                Continuous Batching：
时间 ─────────────────→     时间 ─────────────────→
Req A: ████████████         Req A: ████████████
Req B: ████                 Req B: ████
Req C: ██████               Req C: ██████
Req D: （空闲等待）          Req D:    ████  ← 随时插入！
         
完成 A+B+C 后才能发 D        B 完成后立即插入 D
```

vLLM 实现：每完成一个 token 生成步骤，检查是否有请求完成、有新请求到达，动态调整 batch。

```python
# benchmark_vllm.py — 压测 vLLM 服务
import time
import asyncio
from openai import AsyncOpenAI

client = AsyncOpenAI(
    base_url="http://localhost:8000/v1",
    api_key="not-needed"
)

async def single_request(idx: int) -> tuple[int, float]:
    """单个请求，返回 (序号, 耗时秒)"""
    start = time.time()
    await client.chat.completions.create(
        model="Qwen/Qwen2.5-7B-Instruct",
        messages=[{"role": "user", "content": "用 100 字介绍深度学习"}],
        max_tokens=100
    )
    elapsed = time.time() - start
    return idx, elapsed

async def benchmark(concurrency: int = 10, total_requests: int = 50):
    """并发压测"""
    tasks = [single_request(i) for i in range(total_requests)]
    
    print(f"压测开始：并发={concurrency}，总请求={total_requests}")
    start = time.time()
    results = await asyncio.gather(*tasks)
    total_time = time.time() - start
    
    latencies = [r[1] for r in results]
    latencies.sort()
    
    print(f"总耗时：{total_time:.1f}s")
    print(f"P50 延迟：{latencies[len(latencies)//2]:.2f}s")
    print(f"P95 延迟：{latencies[int(len(latencies)*0.95)]:.2f}s")
    print(f"P99 延迟：{latencies[int(len(latencies)*0.99)]:.2f}s")
    print(f"QPS：{total_requests/total_time:.1f}")

asyncio.run(benchmark(concurrency=10, total_requests=50))
```

```python
# gpu_monitor.py — Python 程序化监控 GPU
import subprocess
import json
import time

def get_gpu_stats():
    """获取 GPU 状态"""
    result = subprocess.run([
        "nvidia-smi",
        "--query-gpu=index,name,utilization.gpu,memory.used,memory.total,temperature.gpu",
        "--format=csv,noheader,nounits"
    ], capture_output=True, text=True)
    
    stats = []
    for line in result.stdout.strip().split("\n"):
        idx, name, util, mem_used, mem_total, temp = [x.strip() for x in line.split(",")]
        stats.append({
            "index": idx, "name": name,
            "gpu_util": f"{util}%",
            "memory": f"{mem_used}/{mem_total} MB",
            "temp": f"{temp}°C"
        })
    return stats

while True:
    for gpu in get_gpu_stats():
        print(f"GPU{gpu['index']} {gpu['name']}: 利用率={gpu['gpu_util']}, "
              f"显存={gpu['memory']}, 温度={gpu['temp']}")
    time.sleep(2)
```

## 常见问题

**Q1：TTFT 和 TPOT 哪个更重要？**

取决于场景。聊天应用：TTFT 重要（用户不想等第一句话）。批量翻译：TPOT 重要（整体速度优先）。流式输出场景：TTFT < 300ms，TPOT < 30ms 是体验标杆。

**Q2：怎么判断是否该加 GPU？**

GPU 利用率持续 > 90% 且 P95 延迟超过目标值 → 加卡。显存占用率 > 90% 且 OOM 频繁 → 加显存更大的卡。注意：利用率高不一定是坏事——理想情况 GPU 就应该在 80-90% 利用率。

**Q3：量化（Quantization）会影响准确性吗？**

INT8 量化：几乎无影响（<1% 精度损失）。INT4 量化：轻微影响（1-3%），大多数场景可接受。低于 4-bit：精度下降明显，不推荐生产环境。显存不够时优先考虑量化而不是加卡。

### 自检题

1. 为什么 KV Cache 大小和并发数有直接的线性关系？
   <details><summary>答案</summary>每个请求独立维护一份 KV Cache，10 个并发 = 10 份。KV Cache 总占用 = 单请求 KV Cache × 并发数，与模型权重（所有请求共享）不同</details>

2. PagedAttention 的"分页"和操作系统虚拟内存的分页有什么异同？
   <details><summary>答案</summary>相同：都是通过页表映射实现不连续物理存储 → 连续逻辑视图。不同：PagedAttention 的页大小按 token 块（如 16 tokens/页），OS 按固定字节（如 4KB/页）；PagedAttention 可共享前缀页，OS 可共享代码页</details>

3. Continuous Batching 为什么比传统批处理效率高？
   <details><summary>答案</summary>传统批处理要等最慢的请求完成才发下一批，GPU 会空转。Continuous Batching 在每个 token 生成步骤后检查，有完成的就插入新请求，GPU 几乎不空转</details>
