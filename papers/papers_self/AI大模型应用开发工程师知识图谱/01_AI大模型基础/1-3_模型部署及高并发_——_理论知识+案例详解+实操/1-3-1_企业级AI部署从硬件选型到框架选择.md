### 什么企业级 AI 部署

**企业级 AI 部署** 不是"下个 Ollama 跑起来就行"，而是要考虑：几百人同时用能不能扛住？一张卡坏了服务会不会断？模型更新要不要停服？

四个核心决策维度：

```
企业级部署 = 硬件选型 + 推理框架 + 并发策略 + 运维体系
                ↓            ↓          ↓          ↓
              选什么GPU   用什么跑   怎么抗压    怎么持续跑
```

### 关键术语

| 术语 | 解释 |
|------|------|
| **吞吐量（Throughput）** | 每秒能处理多少个请求（tokens/s 或 requests/s） |
| **延迟（Latency）** | 单个请求从发出到收到首个 token 的时间（TTFT） |
| **并发** | 同时处理多少个请求而不相互阻塞 |
| **显存** | GPU 上的内存，决定能加载多大的模型 |
| **KV Cache** | 推理时缓存的注意力计算结果，占用显存大头 |

## 原理讲解

### GPU 选型决策

| GPU | 显存 | FP16 算力 | 参考价(万元) | 适合场景 |
|-----|------|----------|-------------|---------|
| RTX 4090 | 24GB | 82 TFLOPS | ~1.5 | 个人开发、微调小模型（7B） |
| L40S | 48GB | 91 TFLOPS | ~8 | 中小规模推理、微调 |
| A100 80GB | 80GB | 312 TFLOPS | ~15 | 大规模推理、训练 70B 模型 |
| H100 | 80GB | 989 TFLOPS | ~25 | 大规模训练、最高吞吐推理 |
| H200 | 141GB | 989 TFLOPS | ~35 | 超大模型推理（405B） |

**选型公式**：

```
所需显存 ≈ 模型参数量 × 2（FP16） + KV Cache × 并发数

例：7B 模型，FP16
  模型权重：7 × 2 = 14 GB
  KV Cache（单请求，4096 tokens）：~0.5 GB
  并发 10 请求：14 + 0.5 × 10 = 19 GB
  → RTX 4090 (24GB) 勉强够用
```

### CPU / 内存 / 网络配比

| 组件 | 作用 | 推荐配比 |
|------|------|---------|
| **CPU** | Tokenization 预处理、请求调度、后处理 | 每 GPU 配 8-16 核 |
| **内存** | 模型加载前的暂存区、系统缓存 | 至少 = 显存总和 × 1.5 |
| **网络（单机）** | 无特殊要求 | PCIe 4.0 x16 即可 |
| **网络（多机）** | 张量并行需要极低延迟 | RoCE（RDMA）或 InfiniBand |

**多机推理**时网络是关键瓶颈——如果 GPU 间通信延迟高，张量并行的加速效果会被网络等待吃掉。

### 三大推理框架对比

| | Ollama | vLLM | SGLang |
|---|--------|------|--------|
| **定位** | 个人/小团队 | 高吞吐 API 服务 | 复杂 Agent 流程 |
| **易用性** | 一条命令启动 | 需写 Python 脚本 | 需写结构化程序 |
| **核心优化** | llama.cpp（CPU友好） | PagedAttention（显存效率） | RadixAttention（前缀共享） |
| **并发能力** | 低（单请求阻塞） | 高（连续批处理） | 高 + 程序化控制 |
| **适合场景** | 本地开发、Demo | 生产 API、大批量推理 | RAG、Agent、CoT 复杂流程 |

**选型建议**：
- 刚入门 → Ollama，零配置体验
- 做 API 服务 → vLLM，吞吐量最优
- Agent/RAG 场景 → SGLang，控制流灵活

```python
from openai import OpenAI

client = OpenAI(
    base_url="http://localhost:8000/v1",
    api_key="not-needed"
)

# 并发测试：同时发 5 个请求
import concurrent.futures

def ask(question: str) -> str:
    response = client.chat.completions.create(
        model="Qwen/Qwen2.5-7B-Instruct",
        messages=[{"role": "user", "content": question}],
        max_tokens=100
    )
    return response.choices[0].message.content

questions = [f"用一句话介绍第{i}种编程语言" for i in range(5)]
with concurrent.futures.ThreadPoolExecutor(max_workers=5) as executor:
    results = list(executor.map(ask, questions))

for q, a in zip(questions, results):
    print(f"Q: {q}\nA: {a}\n")
```

### 3. SGLang：复杂控制流

```bash
pip install "sglang[all]"
```

```python
# sglang_demo.py — SGLang 的 RadixAttention 演示
import sglang as sgl

# 定义结构化生成流程
@sgl.function
def multi_turn_qa(s, question: str):
    """多轮问答：先检索再回答"""
    s += sgl.system("你是一个技术专家。")
    
    # 第一轮：分析问题
    s += sgl.user(question)
    s += sgl.assistant(sgl.gen("analysis", max_tokens=200))
    
    # 第二轮：基于分析给出结论
    s += sgl.user("请基于以上分析，用一段话给出最终结论。")
    s += sgl.assistant(sgl.gen("conclusion", max_tokens=150))

# 运行
backend = sgl.Runtime(model_path="Qwen/Qwen2.5-7B-Instruct")
sgl.set_default_backend(backend)

result = multi_turn_qa.run(question="vLLM 和 SGLang 的核心差异是什么？")
print(f"分析：{result['analysis']}")
print(f"结论：{result['conclusion']}")
```

### 4. 关键概念图解：PagedAttention

```
传统 KV Cache 管理：          PagedAttention：
┌──────────────────┐          ┌──┬──┬──┬──┐
│ 连续内存块         │          │页│页│页│页│  ← 像 OS 分页一样
│ ████████████████ │          ├──┼──┼──┼──┤
│ 碎片化严重         │          │页│  │页│  │  ← 按需分配
│ 显存利用率 ~30%   │          ├──┼──┼──┼──┤
└──────────────────┘          │  │页│  │页│  ← 可动态回收
                               └──┴──┴──┴──┘
                               显存利用率 ~90%
```

PagedAttention 把 KV Cache 像操作系统管理内存一样分页管理——不连续分配、不用的页可以释放、请求间可以共享相同的 prompt 前缀页。这就是 vLLM 能达到 10-20 倍吞吐提升的核心原因。

### 工业界实践
- **多级缓存**：对于高频重复问题（如"公司简介"），加一层语义缓存（semantic cache），相同/相似问题直接返回缓存结果，不调模型
- **动态批处理（Continuous Batching）**：不等一批请求全部完成再发下一批，随时有完成的就插入新请求，vLLM 已内置
- **模型热更新**：用蓝绿部署策略，新模型先启动，验证通过后切换流量，旧模型再下线

## 常见问题

**Q1：个人开发者用 RTX 4090 够吗？**

够。24GB 显存可以跑 7B 模型（FP16）+ 5-10 并发。13B 模型需要 4-bit 量化后才能跑。70B 模型不够（需要 2 张以上的卡或用云 GPU）。

**Q2：Ollama 和 vLLM 底层有什么区别？**

Ollama 基于 llama.cpp（C++ 推理引擎，CPU 友好，GPU 加速有限）。vLLM 基于 PyTorch + 自研 PagedAttention（GPU 优先，吞吐量优势大）。用汽车比喻：Ollama 是家用轿车（方便好用），vLLM 是重卡（拉货能力强）。

**Q3：什么时候需要多机部署？**

模型放不下一张 GPU（如 70B FP16 需要 ~140GB 显存）→ 需要张量并行跨多卡。单机能扛住的并发不够（如 >100 QPS）→ 需要多机做数据并行，前端加负载均衡。

### 自检题

1. 为什么 vLLM 的吞吐量比 Ollama 高那么多？核心技术是什么？
   <details><summary>答案</summary>核心技术是 PagedAttention（分页 KV Cache 管理）+ Continuous Batching（连续批处理）。PagedAttention 让显存利用率从 ~30% 提升到 ~90%，Continuous Batching 让 GPU 不空转等待</details>

2. 选 GPU 时除了显存大小，还有哪些关键指标？
   <details><summary>答案</summary>算力（TFLOPS，决定推理速度）、显存带宽（决定数据传输速度）、互连带宽（NVLink/PCIe，多卡通信瓶颈）、功耗和散热（TDP，影响机房租用成本）</details>

3. 多机推理为什么网络很关键？
   <details><summary>答案</summary>张量并行时，GPU 间需要频繁交换中间计算结果。如果网络延迟高（如用普通 TCP 跨机器），通信时间可能超过计算时间，多卡加速比急剧下降。所以需要 RDMA/InfiniBand 这种低延迟网络</details>
