### 什么是 AI 编程

**AI 编程**不是"用 AI 写代码就不用学编程了"，而是"让 AI 成为你的结对编程伙伴"——你负责架构决策和审查，AI 负责代码生成、重构和测试。

| 传统编程 | AI 辅助编程 |
|---------|-----------|
| 逐行手写代码 | 用自然语言描述需求，AI 生成代码 |
| 查文档、搜 StackOverflow | 直接在编辑器里问 AI |
| 调试靠 print + 肉眼 | AI 分析错误栈，定位问题 |
| 重构手动改几十个文件 | AI 理解上下文，批量修改 |

### 主流 AI 编程工具

| 工具 | 厂商 | 特点 | 适合场景 |
|------|------|------|---------|
| **Cursor** | Cursor Inc | 基于 VS Code，规则系统强大 | 专业开发者日常编码 |
| **Trae** | 字节跳动 | 免费，中文优化 | 国内开发者，初学者 |
| **CodeBuddy** | 腾讯 | 内嵌微信开发者工具 | 小程序开发 |

| **Claude Code** | Anthropic | 终端原生，Agent 式编程 | 命令行工作流 |

### 核心概念：Cursor Rules

Cursor Rules 是 Cursor 的"System Prompt"——定义 AI 在整个项目中的行为规范。

```
项目级 Rules（.cursorrules）  →  设定基础规则（语言、框架、代码风格）
目录级 Rules（.cursor/rules/）→  细化到模块（API 层、前端层不同规则）
```

## 原理讲解

### AI 编程工具的共同工作模式

所有 AI 编程工具本质上是同一个模式：

```
   用户输入（问题描述 / 选中代码 / 报错信息）
                ↓
   ┌────────────────────────────┐
   │ AI 编程工具                 │
   │                            │
   │ 1. 收集上下文                │
   │    当前文件 + 相关文件 + Rules│
   │                            │
   │ 2. 构建提示词               │
   │    系统指令 + Rules + 上下文  │
   │                            │
   │ 3. 调用 LLM                │
   │    GPT-4 / Claude / 自研    │
   │                            │
   │ 4. 后处理                   │
   │    格式化、合并、去重         │
   │                            │
   │ 5. 返回结果                 │
   │    Diff 视图 / 直接插入 / 建议│
   └────────────────────────────┘
```

**关键差异在于第 1 步"收集上下文"**：Cursor 通过分析项目结构自动找到相关文件，Copilot 主要看当前文件，Claude Code 可以主动探索整个代码库。

### Cursor Rules 设计原则

一个典型的 `.cursorrules` 文件：

```ini
# .cursorrules — Cursor AI 行为规则

# 1. 技术栈声明
你正在使用 Python 3.10+, FastAPI, SQLAlchemy, PostgreSQL
前端使用 Vue 3 + TypeScript + Element Plus

# 2. 代码风格
- 函数和类必须有类型注解
- 使用 async/await 而非同步调用
- 遵循 PEP 8，行宽不超过 100 字符
- 错误处理：显式捕获异常，不使用 bare except

# 3. 项目约定
- API 路由放在 api/routes/，数据模型放在 models/
- 数据库迁移使用 Alembic，禁止直接改表
- 配置从环境变量读取，代码中不要硬编码

# 4. 响应风格
- 给出修改时，使用 diff 格式展示变更
- 复杂重构先给方案再写代码
- 生成代码后，自动添加关键注释
```

**设计原则**：Rules 不是越长越好。聚焦于 AI 容易"猜错"的地方——项目特有的约定、非标准的技术选型。

## 代码实战

### 1. Excel 报表处理（AI 编程实战）

以下是用 AI 编程工具处理多条 Excel 的典型流程。假想你在 Cursor/Trae 中描述需求：

```
自然语言需求（写给 AI）：
"我有个 sales/ 目录，里面有 12 个月份的 Excel（1月.xlsx ~ 12月.xlsx）。
每个文件结构相同：A列=产品名，B列=销售额，C列=数量。
请写一个 Python 脚本：
1. 合并所有月份的文件
2. 计算每个产品的年度总销售额
3. 输出 Top 10 产品的条形图
4. 结果保存到 annual_report.xlsx"
```

AI 生成的代码：

```bash
pip install pandas openpyxl matplotlib
```

```python
# sales_report.py — AI 编程生成的 Excel 合并分析脚本
import pandas as pd
import matplotlib.pyplot as plt
from pathlib import Path

# 1. 合并12个月的Excel
sales_dir = Path("sales")
all_data = []

for month_file in sorted(sales_dir.glob("*.xlsx")):
    df = pd.read_excel(month_file)
    df["月份"] = month_file.stem  # 从文件名提取月份
    all_data.append(df)

merged = pd.concat(all_data, ignore_index=True)
print(f"合并完成：{len(merged)} 条记录，{merged['产品名'].nunique()} 个产品")

# 2. 计算年度总销售额 Top 10
yearly = merged.groupby("产品名")["销售额"].sum().sort_values(ascending=False)
top10 = yearly.head(10)
print("\n年度销售 Top 10：")
print(top10)

# 3. 条形图
top10.plot(kind="barh", title="年度销售额 Top 10", xlabel="销售额(元)")
plt.tight_layout()
plt.savefig("top10_chart.png", dpi=150)
print("\n图表已保存到 top10_chart.png")

# 4. 保存报告
with pd.ExcelWriter("annual_report.xlsx") as writer:
    merged.to_excel(writer, sheet_name="明细", index=False)
    top10.to_excel(writer, sheet_name="Top10")
print("报告已保存到 annual_report.xlsx")
```

## 进阶方向

### AI 编程的三层境界

| 层 | 能力 | 例子 |
|----|------|------|
| L1：代码补全 | AI 帮你写完当前这行 | 输入 `def cal` → 自动补全 `calculate_total` |
| L2：任务级生成 | AI 根据描述生成完整功能 | "写一个用户登录接口，含 JWT token" |
| L3：Agent 式编程 | AI 自主探索代码库、跨文件修改、跑测试 | "把这个项目的 ORM 从 SQLAlchemy 迁移到 Peewee"（Claude Code 级别） |

### 当前局限
- **跨文件修改容易遗漏**：AI 可能改 A 文件但忘记改依赖 A 的 B 文件
- **对项目特定约定不了解**：这就是 Rules 的价值所在
- **长上下文可能失焦**：改一个 2000 行的文件时，中间部分的逻辑可能被忽略

## 常见问题

**Q1：AI 写的代码能直接上生产吗？**

不能。AI 是"最佳猜测引擎"，不是"正确性保证引擎"。每条 AI 生成的代码都要经过：人工审查 → 单元测试 → 集成测试 → Code Review，和人工写的代码一视同仁。

**Q2：Cursor、Trae、CodeBuddy 底层用的什么模型？**

Cursor 默认用 GPT-4o + Claude Sonnet，支持自定义 API Key 接入自己的模型。Trae 用字节自研模型（基于豆包）。CodeBuddy 用腾讯混元。但核心差异不在于模型，而在于**上下文收集和交互设计**。

**Q3：Rules 写多长合适？**

100-500 行为佳。太短 AI 会"猜"，太长 AI 会"忽略"。关键是写 AI 容易猜错的约定，不写显而易见的通用规则（如"遵循 PEP 8"——AI 默认就会）。

### 自检题

1. Cursor Rules 和 System Prompt 在概念上有什么相同点和不同点？
   <details><summary>答案</summary>相同：都是预设规则约束 AI 行为。不同：System Prompt 是单次对话的，Cursor Rules 是整个项目的持久化配置，影响所有对话</details>

2. 为什么 AI 编程工具要在 Rules 中声明技术栈，而不能让 AI 自己从代码中推断？
   <details><summary>答案</summary>AI 推断可能出错（版本号、包名混淆），且每次对话都重新推断浪费 token。显式声明保证一致性，且 Rules 可以约束 AI 不乱用其他框架</details>

3. AI 编程工具处理"合并 12 个 Excel 文件"这个任务时，AI 内部经过了哪些步骤？
   <details><summary>答案</summary>①分析需求→②检查 Rules 中的技术栈要求→③搜寻项目上下文（已有代码、目录结构）→④生成代码方案→⑤格式化为可应用的 Diff→⑥带到编辑器供用户审查</details>
