## 核心概念

### 是什么

**提示工程（Prompt Engineering）** 是通过精心设计输入文本，引导大语言模型（LLM）产生高质量输出的技术。**RAG（Retrieval-Augmented Generation，检索增强生成）** 则是在 LLM 生成答案前，先从外部知识库检索相关信息，将检索结果与用户问题一起喂给模型，让模型基于"参考资料"作答。

|  | 纯 LLM | RAG |
|---|--------|-----|
| 知识来源 | 训练数据（静态） | 训练数据 + 外部知识库（动态） |
| 更新方式 | 重新训练/微调 | 更新知识库即可 |
| 幻觉风险 | 高（编造不存在的事实） | 低（输出受检索结果约束） |
| 适用场景 | 通用对话、创意写作 | 企业客服、法律/医疗问答 |

### 为什么重要

LLM 有两个绕不开的问题：**知识截止日期**（训练后发生的事情不知道）和**幻觉**（对不熟悉的问题会编造答案）。RAG 用"先查后答"的思路同时解决了这两个问题——让模型像考试时可以翻书一样，先找到相关资料再回答。

## 原理讲解

### 提示工程的三个层次

可以把提示工程想象成"和模特沟通"：

| 层次 | 类比 | 示例 |
|------|------|------|
| **基础提示** | "帮我写个东西" | `写一篇关于AI的文章` |
| **结构化提示** | "角色+任务+格式" | `你是一个技术博主，用500字介绍Transformer，分三点，每点有例子` |
| **上下文工程** | "给足背景和约束" | 提供 10 页产品文档 + 3 个客户案例，要求生成 FAQ，严格基于给定材料 |

**上下文工程（Context Engineering）** 是提示工程的进阶——不只设计问题，还管理整个"对话上下文"，包括：
- 系统提示词（System Prompt）：定义模型角色和规则
- Few-shot 示例：给出几个问答范例
- 对话历史管理：控制上下文窗口，避免超长对话中模型"遗忘"
- 工具调用指令：告诉模型何时、如何调用外部工具

### RAG 的工作流（四步法）

```
用户提问 "公司年假政策是什么？"
         │
    ┌────▼────┐
    │ 1. 检索  │  将问题转为向量 → 在向量数据库中搜索最相关的文档片段
    └────┬────┘
         │ Top-3 结果: "年假制度.pdf §3", "员工手册 p12", "FAQ 第5条"
    ┌────▼────┐
    │ 2. 增强  │  将检索结果 + 用户原始问题拼接成一个增强版提示词
    └────┬────┘
         │ "根据以下参考资料回答用户问题：
         │   [资料1] ... [资料2] ... [资料3] ...
         │   用户问题：公司年假政策是什么？
         │   如果参考资料没有答案，请明确说明。"
    ┌────▼────┐
    │ 3. 生成  │  LLM 基于增强提示词生成答案
    └────┬────┘
         │ "根据公司手册，您享有每年10天带薪年假..."
    ┌────▼────┐
    │ 4. 引用  │  (可选) 标注答案来源，便于用户查证
    └─────────┘
```

**关键设计点**：
- **分块策略**：文档切多大块？通常 256-1024 tokens，块之间留 10%-20% 重叠防止信息断裂
- **检索数量**：返回 Top-K（通常 3-10 个），太多会稀释关键信息，太少覆盖不全
- **重排序（Re-ranking）**：用更强的模型对初检结果二次排序，提升精确度

### 1. 基础：提示工程入门

先安装依赖（项目根目录执行）：


```python
# prompt_basics.py — 从基础提示到结构化提示
from openai import OpenAI

client = OpenAI()  # 默认读取 OPENAI_API_KEY 环境变量

# 层次1：基础提示（效果差）
bad_prompt = "写一篇关于AI的文章"
bad_response = client.chat.completions.create(
    model="gpt-4o-mini",
    messages=[{"role": "user", "content": bad_prompt}]
)
# 预期：得到一篇泛泛而谈、主题不清的文章

# 层次2：结构化提示（效果显著提升）
good_prompt = """你是一位AI科普作家，请完成以下任务：
1. 用小学生能听懂的语言解释什么是RAG
2. 举一个生活的例子（比如：去图书馆查资料）
3. 不超过200字
"""
good_response = client.chat.completions.create(
    model="gpt-4o-mini",
    messages=[{"role": "user", "content": good_prompt}]
)
print(good_response.choices[0].message.content)
```

### 2. 进阶：简易 RAG 管道

```bash
pip install openai chromadb sentence-transformers
```

```python
# simple_rag.py — 最小可运行的 RAG 示例
from openai import OpenAI
import chromadb

# ===== 步骤1：准备知识库 =====
documents = [
    "公司年假制度：入职满1年享有5天年假，满3年享有10天年假，满5年享有15天年假。",
    "请假流程：员工需提前3天在OA系统提交请假申请，经直属领导审批后生效。",
    "病假需要提供医院开具的病假条，急诊可事后补交。"
]

# 创建向量数据库（内存模式）
chroma_client = chromadb.Client()
collection = chroma_client.create_collection(name="company_policy")

# 将文档转为向量并存入数据库
# 注：ChromaDB 默认使用 all-MiniLM-L6-v2 做 embedding
for i, doc in enumerate(documents):
    collection.add(
        documents=[doc],
        ids=[f"doc_{i}"]
    )

# ===== 步骤2：检索相关文档 =====
user_question = "我工作了2年，有几天年假？"
results = collection.query(
    query_texts=[user_question],
    n_results=2  # 返回最相关的2条
)
retrieved_docs = results['documents'][0]
print(f"检索到的资料：{retrieved_docs}")
# 预期输出：['公司年假制度：入职满1年享有5天年假...']

# ===== 步骤3：增强 + 生成 =====
client = OpenAI()
augmented_prompt = f"""你是一个公司政策问答助手。
请严格根据以下参考资料回答问题。如果资料中没有答案，请说"资料中未提及"。

参考资料：
{chr(10).join(f'- {doc}' for doc in retrieved_docs)}

用户问题：{user_question}"""

response = client.chat.completions.create(
    model="gpt-4o-mini",
    messages=[{"role": "user", "content": augmented_prompt}]
)
print(f"回答：{response.choices[0].message.content}")
# 预期输出：根据公司年假制度，您入职满1年，享有5天年假。
```

### 3. 关键代码解读

| 代码段 | 作用 | 为什么这样写 |
|--------|------|-------------|
| `ChromaDB` 内存模式 | 快速原型验证 | 生产环境替换为持久化存储（如 `PersistentClient`） |
| `n_results=2` | 控制检索数量 | 太少丢失信息，太多引入噪声，RAG 中一般 3-5 最佳 |
| 提示词中的 `严格根据以下参考资料` | 防止幻觉 | 约束模型只在给定材料中作答，这是 RAG 的核心技巧 |
| 默认 embedding 模型 | `all-MiniLM-L6-v2` | 384 维向量，速度快，中英文都可用，适合入门 |

## 进阶方向

### 当前局限
- **基础 RAG 无法处理复杂推理**：用户问"比较 A 和 B"时，A 和 B 分别检索到，但模型可能不会自动对比
- **检索质量决定上限**：垃圾进垃圾出，文档切分和 embedding 模型选择比 LLM 本身更关键
- **更新不及时**：知识库需要独立的更新管道（增量索引）

### 工业界最佳实践
- **混合检索**：向量检索（语义相似）+ 关键词检索（BM25）结合，取各自 top-K 再合并排序

- **引用溯源**：返回答案时标注来源文档，让用户可验证

## 常见问题

**Q1：提示工程是不是就是"会写 prompt"？跟普通使用有什么区别？**

不是。普通使用是"我今天想吃什么，让 AI 帮我查"，提示工程是"我要做一个客服系统，全年 7x24 准确率 95% 以上"。前者随便写写就行，后者需要系统化的 prompt 设计、测试、迭代——像一个 QA（质量保证）流程。

**Q2：有向量数据库了还学 RAG 干什么？直接用不行吗？**

向量数据库只管"找到相似的文本"，不管"生成答案"。RAG 把检索和生成串起来，还需要设计提示词模板、处理多轮对话、控制引用格式。就像你有了一本百科全书（向量数据库），但还需要一套方法（RAG）教 AI 怎么"查书→理解→回答"。

**Q3：我的文档全是中文，用什么 embedding 模型好？**

推荐 `BAAI/bge-large-zh-v1.5`（智源研究院开源），在中文 MTEB 榜单上排名靠前，支持 1024 维向量。轻量级可选 `BAAI/bge-small-zh-v1.5`（512 维，速度快 3 倍）。

### 自检题

1. RAG 把答案生成分成了哪四步？每一步的核心操作是什么？
   <details><summary>答案</summary>检索（问题→向量→搜索文档）→ 增强（拼接资料+问题）→ 生成（LLM 基于资料作答）→ 可选引用（标注来源）</details>

2. 为什么 RAG 能减少幻觉，而纯 LLM 不行？
   <details><summary>答案</summary>RAG 把答案限制在检索到的参考资料范围内，模型不会凭空编造；纯 LLM 只能靠训练记忆，对不知道的内容倾向于"编一个合理的答案"</details>

3. 提示词中为什么要加"如果参考资料中没有答案，请明确说明"？
   <details><summary>答案</summary>这是 RAG 的防护措施。不加这句话，LLM 可能忽略检索结果限制，用自己的训练知识补充答案，重新引入幻觉</details>
