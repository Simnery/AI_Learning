# 测试反模式

**在以下情况加载此参考:** 编写或修改测试、添加 mock、或试图向产品代码添加仅用于测试的方法时。

## 概览

测试必须验证真实行为，而不是 mock 行为。Mock 是隔离的工具，不是被测试的东西。

**核心原则:** 测试测的是代码做什么，不是 mock 做什么。

## 铁律

```
1. 永远不要测试 mock 行为
2. 永远不要向产品类添加仅用于测试的方法
3. 永远不要在不理解依赖的情况下 mock
```

## 反模式 1: 测试 Mock 行为

**为什么是错的:** 你在验证 mock 能工作，而不是组件能工作。

**修复方法:** 测试真实组件 或 不要 mock 它。不要在 mock 上做断言。

## 反模式 2: 产品代码中的仅测试方法

**为什么是错的:** 产品类被仅用于测试的代码污染。违反 YAGNI。

**修复方法:** 让测试工具类处理测试清理。将仅测试用的方法移到测试工具类中。

## 反模式 3: 不理解就 Mock

**为什么是错的:** Mock 掉的方法有测试依赖的副作用。过度 mock 破坏实际行为。

**修复方法:** 在正确的层级 mock。Mock 之前理解测试依赖哪些副作用。

## 反模式 4: 不完整的 Mock

**为什么是错的:** 部分 mock 隐藏了结构假设。下游代码可能依赖你没包含的字段。

**铁则:** Mock 完整的数据结构，与实际情况一致。

## 反模式 5: 集成测试作为事后补充

**为什么是错的:** 测试是实现的一部分，不是可选的后续工作。

**修复方法:** TDD 循环: 写失败测试 → 实现 → 重构 → 然后才声称完成。

## 快速参考

| 反模式 | 修复方法 |
|--------------|-----|
| 对 mock 元素做断言 | 测试真实组件 或 取消 mock |
| 产品代码中的仅测试方法 | 移到测试工具类 |
| 不理解就 mock | 先理解依赖，最小化 mock |
| 不完整的 mock | 完整镜像真实 API |
| 测试作为事后补充 | TDD - 测试先行 |
| 过度复杂的 mock | 考虑集成测试 |

## 红旗

- 断言检查 `*-mock` 测试 ID
- 方法只在测试文件中被调用
- Mock 设置占测试的 50% 以上
- 移除 mock 后测试失败
- 无法解释为什么需要 mock
- "为了安全起见" 而 mock

## 底线

**Mock 是隔离的工具，不是被测试的东西。**

如果 TDD 揭示你正在测试 mock 行为，你就走偏了。

修复: 测试真实行为，或质疑你为什么需要 mock。
