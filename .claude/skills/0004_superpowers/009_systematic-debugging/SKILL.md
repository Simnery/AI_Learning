# Systematic Debugging

## 概述

随机修复浪费时间并产生新 bug。快速补丁掩盖了根本问题。

**核心原则：** 在尝试修复之前，始终先找到根本原因。修复表象是失败。

**违反本流程的字面规定，就是违反调试精神本身。**

## 铁律

```
没有根本原因调查，就没有修复方案
```

如果你还没有完成阶段一，就不能提出修复方案。

## 何时使用

遇到任何技术问题时使用：
- 测试失败
- 生产环境 bug
- 非预期行为
- 性能问题
- 构建失败
- 集成问题

**在以下情况尤其要使用：**
- 时间紧迫（紧急情况容易让人想猜测）
- "简单快速修一下"看起来很明显
- 你已经尝试过多次修复
- 之前的修复无效
- 你并没有完全理解问题

**以下情况不要跳过：**
- 问题看起来很简单（简单的 bug 也有根本原因）
- 你很着急（仓促必然导致返工）
- 管理者要求立刻修复（系统化比反复折腾更快）

## 四个阶段

你必须完成每个阶段后才能进入下一阶段。

### 阶段一：根本原因调查

**在尝试任何修复之前：**

1. **仔细阅读错误信息**
   - 不要跳过错误或警告
   - 它们往往包含确切的解决方案
   - 完整阅读堆栈追踪
   - 注意行号、文件路径、错误码

2. **稳定复现**
   - 能否可靠触发？
   - 确切步骤是什么？
   - 每次都会发生吗？
   - 如果不能复现 → 收集更多数据，不要猜测

3. **检查近期变更**
   - 什么变更可能导致此问题？
   - Git diff、最近的提交
   - 新依赖、配置变更
   - 环境差异

4. **在多组件系统中收集证据**

   **当系统包含多个组件时（CI → 构建 → 签名，API → 服务 → 数据库）：**

   **在提出修复方案之前，添加诊断性探针：**
   - 记录每个组件输入的数据
   - 记录每个组件输出的数据
   - 验证环境/配置的传播
   - 检查每一层的状态

   运行一次收集证据，显示哪里出问题，然后分析证据确定故障组件，然后调查该特定组件。

5. **追踪数据流**

   **当错误在调用栈深处时：**
   - 坏值从哪里产生？
   - 谁用坏值调用了这里？
   - 持续向上追踪，直到找到源头
   - 在源头修复，而非在表象修复

### 阶段二：模式分析

**在修复之前找到模式：**

1. **寻找可工作的实例** — 在同一代码库中定位类似的可工作代码
2. **对照参考实现** — 完整阅读参考实现。不要快速浏览。
3. **识别差异** — 列出每一项差异，无论多小。不要假设"这不可能有关系"
4. **理解依赖** — 此组件需要哪些其他组件？它做了哪些假设？

### 阶段三：假设与测试

**科学方法：**

1. **形成单一假设** — 清晰陈述："我认为 X 是根本原因，因为 Y"
2. **最小化测试** — 做尽可能最小的改动。一次只改一个变量。
3. **验证后再继续** — 有效吗？是 → 阶段四。否 → 形成新假设。
4. **当你不懂时** — 说"我不理解 X"。不要假装。请求帮助。

### 阶段四：实现

**修复根本原因，而非表象：**

1. **创建失败的测试用例** — 修复前必须要有。使用 `0004_superpowers:004_test-driven-development` skill。
2. **实现单一修复** — 一次一个改动。不要"顺便"改进其他东西。
3. **验证修复** — 测试通过了？没有其他测试被破坏？问题确实解决了？
4. **如果修复无效** — 停止。如果尝试 < 3 次：回到阶段一。**如果 ≥ 3 次：质疑架构。**
5. **如果 3 次以上修复失败：质疑架构** — 这个模式在根本上是否合理？与你的合作伙伴讨论。

## 红旗警告——停止并按流程执行

- "先快速修一下，之后再调查"
- "试试改一下 X，看看行不行"
- "加多个改动，一起跑测试"
- "跳过测试，我手动验证"
- "应该就是 X，我来修复它"
- "我不完全理解，但这可能有效"
- 在追踪数据流之前就提出解决方案
- "再试一次修复"（已经试了 2 次以上）
- 每次修复都在不同地方揭示新问题

**以上所有情况都意味着：停止。回到阶段一。**

**如果 3 次以上修复失败：** 质疑架构。

## 常见借口

| 借口 | 现实 |
|------|------|
| "问题很简单，不需要流程" | 简单问题也有根本原因。流程对简单 bug 也很快。 |
| "紧急情况，没时间按流程" | 系统化调试比盲目猜测反复折腾更快。 |
| "先试试这个，然后再调查" | 第一次修复就定下了基调。从一开始就做对。 |
| "确认修复有效后我再写测试" | 未经测试的修复不能持久。先写测试来证明。 |
| "一次修复多处更省时间" | 无法隔离哪个有效。还会产生新 bug。 |
| "我看到问题了，让我修" | 看到表象 ≠ 理解根本原因。 |

## 快速参考

| 阶段 | 关键活动 | 成功标准 |
|------|----------|----------|
| **1. 根本原因** | 阅读错误、复现、检查变更、收集证据 | 理解"是什么"和"为什么" |
| **2. 模式** | 找到可工作实例、对比 | 识别差异 |
| **3. 假设** | 形成理论、最小化测试 | 确认或形成新假设 |
| **4. 实现** | 创建测试、修复、验证 | bug 解决，测试通过 |

## 当流程揭示"没有根本原因"时

如果系统化调查表明问题确实是环境性、时序依赖或外部的：
1. 记录你调查的内容
2. 实现适当的处理（重试、超时、错误信息）
3. 添加监控/日志以便日后调查

**但是：** 95% 的"没有根本原因"案例都是不完整的调查。

## 真实世界效果

- 系统化方法：15-30 分钟修复
- 随机修复方法：2-3 小时反复折腾
- 首次修复成功率：95% vs 40%
- 引入新 bug：几乎为零 vs 常见
