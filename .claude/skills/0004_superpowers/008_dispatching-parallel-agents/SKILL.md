# Dispatching Parallel Agents

## 概述

当方案中有多个互不依赖的任务时，并行派发。每个任务一个独立子代理同时运行。全部完成后收集结果，然后合并。

**核心原则:** 独立 = 并行。依赖 = 顺序。

## 何时使用

- 方案中有 3+ 个独立任务
- 任务间不共享可变状态或无数据依赖
- 需要最快执行速度
- 每个任务产出独立、可独立审查的输出

**不要用的情况:**
- 任务有顺序依赖 (用 subagent-driven-development)
- 只有 1-2 个任务 (开销 > 收益)
- 多个任务改同一个文件 (有冲突风险)

<HARD-GATE>
派发前必须完成依赖分析并构建依赖矩阵。跳过分析直接派发 = 不可接受。
并行派发修改同一文件的任务 = 不可接受。
</HARD-GATE>

## 依赖分析

在并行派发之前，分析方案的依赖关系:

1. **对每个任务，识别:**
   - 创建的文件
   - 修改的文件
   - 对其他任务输出的依赖

2. **构建依赖图:**
   - 两个任务修改同一文件 → 顺序执行
   - 任务 B 依赖任务 A 的输出 → 顺序执行
   - 无重叠 → 可并行

3. **分组为并行批次:**
   - 批次 1: 所有零依赖任务 → 并行运行
   - 批次 2: 依赖批次 1 的任务 → 下一波

## 执行过程

### 第 1 步: 加载方案并分析
1. 从任务目录读取方案文件
2. 构建依赖矩阵
3. 识别并行批次
4. 将批次计划呈现给人类伙伴审批

### 第 2 步: 并行派发批次

对当前批次的每个任务，同时派发:

```
Agent(
  subagent_type="general-purpose",
  description="任务 N: [组件名]",
  prompt="[完整的自包含任务上下文]",
  run_in_background=true
)
```

**并行上限:** 最多 4 个并发子代理。批次超出的排队等待。

### 第 3 步: 收集结果

每个后台子代理完成后:
1. 从通知/任务输出中读取结果
2. 验证所有步骤已执行
3. 运行测试（如适用）
4. 标记任务状态

### 第 4 步: 审查批次

批次中所有任务完成后:
1. 验证并行输出之间无文件冲突
2. 运行完整测试套件
3. 全部通过 → 提交批次 → 下一批
4. 有失败 → 顺序修复 → 重新测试

### 第 5 步: 收尾

所有批次完成后:
- 完整测试套件验证
- 输出每个任务状态的摘要
- **必须调用的下一个 Skill:** 0004_superpowers:012_finishing-a-development-branch

## 集成

**输入:** writing-plans 输出，含独立任务分析
**前提:** subagent-driven-development 的审查流程
**替代方案:** 0004_superpowers:007_subagent-driven-development (用于顺序任务)

## 反模式: "这几个任务看起来不冲突，直接跑"

不分析依赖直接并行是赌博。两个任务可能修改同一文件的相邻行、同一个函数的签名、同一份配置 — 这会产生静默冲突，不是 git 能自动检测的。

## 红灯 — 立即停止

- 派发修改同一文件的任务
- 并发超过 4 个子代理
- 跳过依赖分析
- 并行批次后不验证文件冲突

## 牢记

- 最多 4 个并发子代理
- 每个提示词自包含，无交叉引用
- 派发前必须做依赖分析
- 批次审查发现并行冲突
