# Requesting Code Review

**核心原则：** 尽早审查，频繁审查。

## 何时请求审查

**必须请求：**
- subagent 驱动开发中每完成一个 task 之后
- 完成主要功能之后
- 合并到 main 之前

**建议但非强制：**
- 遇到障碍时（需要新视角）
- 重构前（基线检查）
- 修复复杂 bug 之后

## 如何请求

**1. 获取 git SHA：**
```bash
BASE_SHA=$(git rev-parse HEAD~1)  # 或 origin/main
HEAD_SHA=$(git rev-parse HEAD)
```

**2. 先自行审查 diff：**
- 完整阅读 `git diff BASE_SHA..HEAD_SHA`
- 检查：每个变更是否服务于计划？有无无关变更？是否缺少测试？

**3. 对照计划/需求进行审查：**
- 加载原始计划或规格说明
- 逐项检查每个需求是否已实现
- 确认无遗漏

**4. 编写审查摘要：**
- 已实现内容 vs 计划内容
- 不确定的区域
- 所做的约束条件或权衡取舍

## 审查检查清单

- [ ] 每个计划中的 task 已实现
- [ ] 所有测试通过
- [ ] 无无关或偏离的变更
- [ ] 存在错误处理
- [ ] 已覆盖边界情况
- [ ] 代码遵循既有模式

## 红线

- 因为"很简单"而跳过审查
- 忽略 Critical 级别的问题
- 在问题未修复的情况下继续推进
- 对有效的技术反馈进行争辩

**如果审查者判断有误：**
- 用技术理由进行反驳
- 展示能证明正确的代码/测试
- 请求澄清
