# Subagent-Driven Development

## 概述

为每个任务派发全新的子代理来执行实施方案。每个子代理只拿到计划和当前任务 — 零上下文污染，零累积偏差。任务间的两阶段审查在问题累积之前就将其拦截。

**何时选择:**
- **用这个** — 有 3+ 个任务的计划、复杂依赖、或上下文质量至关重要的场景
- **用 executing-plans** — 短计划 (1-2 个任务) 或简单顺序工作

## 何时使用

- writing-plans 产出的多任务实施方案
- 上下文偏差会降低质量的复杂功能
- 所有支持子代理的平台 (Claude Code, Codex)

<HARD-GATE>
在完成当前任务的两阶段审查并确认通过之前，不得派发下一个任务。审查未通过 = 不得继续。
</HARD-GATE>

## 执行过程

### 第 1 步: 加载并审查方案
1. 从任务目录读取方案文件 (如 `00_local_task/tasks/{任务ID}/plans/方案文件名.md`)
2. 批判性审查 — 派发前先发现疑问
3. 如有疑虑：先与人类伙伴沟通
4. 分析任务依赖 — 哪些必须顺序执行，哪些相互独立

### 第 2 步: 按顺序派发任务

对每个任务（按依赖顺序）：

1. **构造子代理提示词** — 包含:
   - 方案中该任务的确切要求
   - 确切的文件路径 (新建/修改/测试)
   - 预期的测试命令和输出
   - 前面已完成任务的上下文

2. **派发子代理:**
   ```
   Agent(
     subagent_type="general-purpose",
     description="任务 N: [组件名]",
     prompt="[方案中该任务的完整上下文]"
   )
   ```

3. **等待完成** — 子代理返回结果

4. **第一阶段审查 — 自检:**
   - [ ] 任务中的所有步骤都完成了？
   - [ ] 测试通过（从子代理输出中验证）？
   - [ ] 输出与方案预期一致？
   - [ ] 有没有不相关的改动？

5. **第二阶段审查 — 人工门 (对重要任务):**
   - 呈现：做了什么、测试结果、做出的决策
   - 等待审批后再进入下一个任务

6. **通过** → 按方案提交 → 下一个任务
7. **失败** → 同一子代理修复问题 → 重新审查

### 第 3 步: 最终审查

所有任务完成后:
- 验证全部测试通过 (完整套件)
- 检查没有与方案不符的需求缺口
- 输出完成摘要

### 第 4 步: 交接

- 宣告完成
- **必须调用的下一个 Skill:** 0004_superpowers:012_finishing-a-development-branch
- 按该 skill 验证、呈现选项、执行

## 集成

**输入:** writing-plans 输出格式 (带文件路径、命令、预期输出的编号任务)
**输出:** 每个任务已提交，所有测试通过，准备好进入 finishing-a-development-branch
**替代方案:** 0004_superpowers:008_dispatching-parallel-agents (用于独立任务)

## 反模式: "不需要审查，这个任务太简单了"

每个任务都要审查。简单任务也可能引入回归或遗漏边界情况。审查可以是快速的 (30 秒扫一眼自检清单)，但绝不能跳过。

## 红灯 — 立即停止

- 跳过任务间的审查
- 当前任务未验证就派发下一个
- 子代理提示词中没有确切文件路径
- 信任子代理的成功汇报而不独立验证

## 常见借口

| 借口 | 现实 |
|------|------|
| "这个任务太简单，不用审查" | 简单任务也会引入回归，审查只需 30 秒 |
| "上一个任务通过了，直接继续" | 通过 ≠ 与方案一致，必须对照方案逐项确认 |
| "子代理说成功了" | 独立验证，不能信任二手汇报 |

## 牢记

- 每个任务全新子代理 = 干净上下文 = 更高质量
- 两阶段审查尽早发现问题
- 每个子代理提示词必须自包含 (零依赖会话历史)
- **末尾必须调用的下一个 Skill:** 0004_superpowers:012_finishing-a-development-branch
