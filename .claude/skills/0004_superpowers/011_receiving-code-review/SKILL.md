# Receiving Code Review

**核心原则：** 审查是技术求真，不是自我辩护。

## 回应模式

对每一条反馈：
1. **理解** — 用自己的话复述反馈内容
2. **评估** — 反馈在技术上是否正确？是否指出了真正的问题？
3. **回应** — 确认正确的反馈。如果不正确，用理由进行反驳。
4. **执行** — 修复真正的问题

## 禁止的回应

- "它能跑" — 能运行不代表正确
- "我手动测过了" — 手动测试不代表全面
- "我一直这么做的" — 习惯不代表正确
- 防御心态 — 审查针对的是代码，不是个人
- 默默顺从 — 如果你不同意，用技术理由说明

## 何时反驳

在以下情况反驳：
- 反馈在技术上是错误的（解释原因）
- 建议的修改会破坏其他功能
- 审查者误解了上下文或约束条件

## 常见错误

- 不加评估地采纳每一条建议
- 忽略看不懂的反馈
- 把反馈当作人身攻击
- 用快速变通方案代替正确修复

## 底线

不搞表面附和。始终追求技术严谨。
