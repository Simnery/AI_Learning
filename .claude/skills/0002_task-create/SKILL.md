# 任务创建 Skill

**格式规则**：Skill 是模板，禁止出现具体任务 ID、文件路径、数量。统一用占位符：`{任务ID}` `{已完成}` `{总计}` `{X}%`。

---

## 调用时机

### 主动触发（必须提示用户）

发现以下情况时，**主动询问用户**是否创建任务记录：

| 信号 | 示例 |
|------|------|
| 3 个及以上独立步骤 | "先做A，再B，最后C" |
| 涉及多个文件/目录操作 | "重构 XX 模块"、"给所有文件加 XX" |
| 预计跨会话才能完成 | "帮我整理这些资料"（几十个文件） |
| 用户明确说要做"项目/任务/实验" | "开始XX项目"、"做XX实验" |

**提示模板**：
> 这个操作涉及多个步骤（预估 N 步），当前没有任务追踪文档。是否先创建一个任务记录？防止窗口中断丢失进度。

### 被动触发（用户直接要求）

用户明确要求"创建任务"、"新建任务"、"记录任务"时直接执行创建流程。

## 执行流程

### 0. 环境初始化（首次执行时）
检查 `.claude/settings.local.json` 是否存在：
- 若**不存在** → 创建文件，写入空权限配置：
  ```json
  {
    "permissions": {
      "allow": []
    }
  }
  ```
- 确认 `.gitignore` 包含 `.claude/settings.local.json`（已默认配置）

### 1. 确定任务信息
- **任务标题**：简要描述任务目标
- **类别**：理论学习 / 动手实验 / 项目实战 / 论文阅读 / 其他
- **创建时间**：当前日期

### 2. 创建任务文件

**文件结构**（.md 文件必须放在同名文件夹**内部**）：

```
00_local_task/tasks/{日期}_{类别}_{简要描述}/
  └── {日期}_{类别}_{简要描述}.md       ← 任务主文件（Claude / Cursor 共用，唯一入口）
  └── (实验数据、笔记等附件存放在此文件夹)
```

- 步骤1：在 `00_local_task/tasks/` 下创建文件夹（项目根目录，不在 `.claude/` 内，无权限弹窗）
- 步骤2：在文件夹内部创建 .md 文件
- 命名规则：文件夹名与 .md 文件名相同（即 `{任务ID}`）
- 格式：使用 `.claude/memory/tasks/_template.md` 模板
- 此目录已配置 .gitignore 排除；禁止在其它路径维护第二份进度

### 2.1 初始化任务索引（若缺失）
- 目标文件：`00_local_task/task_index.md`
- 模板来源：`.claude/memory/tasks/task_index.template.md`

### 3. 更新任务索引
- 新建 → `active` / 暂停 → `paused` / 完成 → `done`
- 字段：`id | status | owner | last_update | task_file`

### 4. 任务进行中持续更新

**强制规则**：每完成一个子步骤后，**先更新文档，再继续下一步**。不更新不往下走。

固定更新位置（按顺序修改以下 6 处）：

| # | 位置 | 操作 |
|---|------|------|
| 1 | `> **更新时间**` (文件头部) | 刷新为当前时间 |
| 2 | `## TL;DR` | 一句话概括当前状态 |
| 3 | `## 进度表` | 对应步骤行 → `✅`，下一行 → `⏳` |
| 4 | `## 当前操作` | 更新为下一步的 checklist |
| 5 | `## 已完成操作` | 追加刚才完成的事项 |
| 6 | `## 进度日志` | **追加一行**：时间 \| 步骤 \| 操作 \| 结果 \| 下一步 |

**自检**：任务进行中，若发现进度日志最新行与 TL;DR/进度表 不一致 → 立即同步后再继续。

**批量操作**：每完成一个子步骤就完整走一遍 6 处更新，不允许攒到最后一次更新。

#### 4.1 审计机制（机械验证，不靠 AI 自述）

**每次更新进度日志后，立即执行以下机械检查**。审计不通过 → 回退修正，不得继续。

| # | 审计项 | 方法 | 通过标准 |
|---|--------|------|---------|
| A | 文件存在 | `ls -la <文件路径>` | 文件存在，非空 |
| B | 行数有效 | `wc -l <文件路径>` | 200-500 行 |
| C | 6 块齐全 | `grep -c "## " <文件路径>` | 6 个 `## ` 标题（核心概念/原理讲解/代码实战/进阶方向/常见问题/延伸阅读） |
| D | 代码块存在 | `grep -c "\`\`\`" <文件路径>` | ≥ 2 对代码块（至少一个 Python） |
| E | 进度表一致 | 对比进度表状态 vs 实际文件 | 进度表 `✅` 的文件实际存在 |
| F | 日志链完整 | 进度日志最新行 vs 上一步预期 | 日志行中"下一步"与本次实际操作一致 |

**审计时机**：
- 每完成 1 个文件：执行 A-D
- 每完成 5 个文件：执行 A-F 全套
- 每章完成时（检查点）：执行 A-F 全套 + 进度百分比验算

**审计命令执行规则（重要）**：

> **禁止用 `&&` / `;` / `|` 链式执行审计命令**。变量赋值 + `&&` 链会触发安全拦截器弹窗。
>
> 正确做法：**每条审计命令独立一个 `Bash` 调用**，并行发送。

**审计失败处理**：
- 任一条不通过 → 记录到「遇到的问题」表，暂停，先修正再继续
- 同一条连续 2 次不通过 → 暂停当前任务，等用户介入

### 5. 多实验流程

#### 数据收集规范
```
任务目录/
  ├── {任务ID}.md
  ├── exp1/                 ← 每次实验一个文件夹
  │   ├── config.yaml
  │   ├── train_log.txt
  │   └── metrics.json
  ├── exp2/
  └── ...
```

### 6. 任务收尾沉淀（done 前强制）
- **流程优化检查**：主动询问用户是否有值得沉淀的改进
- **知识沉淀**：是否值得提取为独立笔记
- **索引更新**：状态改为 done，迁移到 Done 分区

### 7. 长任务防精度衰减（检查点 + 预算监控 + 状态驱动）

**问题**：上下文快满时，模型精度下降，容易遗漏步骤或偏离方案。

#### 7.1 检查点机制

每完成一章（或每 N 个子步骤），在 `## 进度日志` 中写一条 `[检查点]` 条目：

```
| 时间 | 步骤 | 操作 | 结果 | 下一步 |
|------|------|------|------|--------|
| 23:00 | [检查点] 第1章完成 | {已完成}/{总计} 文件，进度 {X}% | 文件范围 全部自检通过 | 下一章入口 |
```

检查点内容：
- 已完成多少
- 当前进度百分比
- 关键决策/变更（如有）
- 下一步精确入口

#### 7.2 上下文预算监控

当出现以下信号时，**立即写检查点，提示用户执行 `/clear`**：
- 对话轮次明显增多，响应变慢
- 需要频繁回顾前文才能记得当前状态
- 已连续处理超过 10 个子步骤

**提醒模板**：
> 检查点已写入。建议 `/clear` 清理上下文后，发送以下命令恢复：
>
> ```
> 继续 {任务ID}
> ```

**优先级**：`/clear` > 新开会话。`/clear` 保持工作目录和配置，恢复更快。

**禁止行为**：必须用上述模板做**陈述句提示**。禁止出现以下问句形式：
- ❌ "是否现在启动检查，还是先 /clear？"
- ❌ "要继续还是暂停？"
- ❌ "需要 /clear 吗？"

**Why:** 任务编写原则规定"不询问'继续还是暂停'"，上下文不足时直接给指令，不让用户做选择。陈述句 = 告诉用户现在怎么做最优。问句 = 把决策负担推回给用户。

> // 更新锚点: 7.2-禁止问句-2026-05-31

#### 7.3 `/clear` 后恢复协议

执行 `/clear` 后，**只需一句话恢复**：

```
继续 {任务ID}
```

AI 收到此命令后的标准动作：
1. 定位任务文件 `00_local_task/tasks/{任务ID}/{任务ID}.md`（注意：任务数据在 `00_local_task/tasks/`，不在 `.claude/memory/tasks/`）
2. 读任务文档 `## TL;DR` → 了解当前状态
3. 读 `## 当前操作` → 确定下一步
4. 读 `## 进度日志` 最新 3 行 → 确认最后进度
5. 立即继续执行，不追问、不翻旧对话

> Cursor 用户可用 `@00_local_task/tasks/{任务ID}/{任务ID}.md` 附带恢复，仍只读这一份主文档。
